Симетрична vs Асиметрична криптографія

Симетрична

  • Один спільний ключ K
  • Дуже висока швидкість
  • Ключів: 1 на пару
  • Довжина: 128–256 біт
  • Проблема: як безпечно передати K?

Асиметрична

  • Пара ключів: public + private
  • Повільніша, але усуває проблему розподілу
  • Ключів: 2 (public + private)
  • Довжина: 2048–4096 біт (RSA)
  • Рішення: public key поширюється відкрито

RSA (Rivest–Shamir–Adleman, 1977)

Принцип безпеки: складність факторизації великих цілих чисел

Генерація ключів (спрощено):
 ① Обираємо два великих простих p та q
 ② n = p × q  (відкрита частина: модуль)
 ③ Обчислюємо показники e та d
 ④ Public Key  = (n, e)
 ⑤ Private Key = (n, d)

 Шифрування:   C = M^e mod n
 Дешифрування: M = C^d mod n
Розмір ключаСтійкістьРекомендація
1024 біт~80 біт❌ Застарілий
2048 біт~112 біт✅ Мінімум (~2030)
3072 біт~128 біт✅ Рекомендований
4096 біт~140 біт✅ Довгострокове

Застосування

  • TLS/HTTPS: шифрування або підпис у Key Exchange
  • Цифровий підпис документів та коду
  • S/MIME, PGP — захист e-mail

Принцип безпеки: задача дискретного логарифму на еліптичній кривій (ECDLP)

Популярні криві

  • P-256 (secp256r1) — NIST, TLS, HTTPS
  • P-384 — підвищена безпека (NSA Suite B)
  • Curve25519 — Tor, Signal, OpenSSH

Переваги ECC

  • Менший ключ → менше CPU і пам'яті
  • Менший трафік
  • Ідеальний для IoT та мобільних пристроїв
  • Швидше, ніж RSA при однаковому рівні безпеки

Геометрична інтуїція

Знаючи точку P і результат Q = k×P, знайти k — обчислювально неможливо (ECDLP)

Diffie-Hellman (DH)

Мета: виробити спільний секрет через незахищений канал, не передаючи його безпосередньо

  Alice                    Bob
  ──────                   ──────
  Генерує a (private)      Генерує b (private)
  A = g^a mod p            B = g^b mod p
       ─────────A──────────▶
       ◀────────B────────────
  S = B^a mod p            S = A^b mod p
  └──────────── S = g^(ab) mod p ────────┘

  Підслуховувач бачить: g, p, A, B
  → знайти S неможливо

PFS (Perfect Forward Secrecy): нові ключі для кожної сесії → компрометація довгострокового ключа не розкриває минулий трафік

⚠️ DH не забезпечує автентифікацію! → використовується разом із цифровим підписом або сертифікатами

Цифровий підпис

Цифровий підпис — криптографічний аналог власноручного підпису, що прив'язує дані до ідентичності підписанта

1. АВТЕНТИЧНІСТЬ

"Саме Аліса підписала цей документ"

2. ЦІЛІСНІСТЬ

"Документ не змінено після підписання"

3. НЕВІДРЕЧЕНІСТЬ

"Аліса не може заявити 'я не підписувала'"

Що підписується насправді?

Підписується не саме повідомлення, а його криптографічний хеш (дайджест) — фіксований розмір, будь-яка зміна → повністю інший хеш

Надсилається: Документ M + Підпис S + Сертифікат Alice

Ключовий момент: підпис S створюється закритим ключем Alice → лише Alice могла його створити

Фаза перевірки

Перевіряльник (Bob) отримує: Документ M + Підпис S

①

Обчислює хеш отриманого документа: M → SHA-256 → h₁

②

Дешифрує підпис відкритим ключем Alice: S + pub(A) → RSA/ECDSA → h₂

③

Порівнює: h₁ == h₂?

✅ ТАК → підпис дійсний, дані не змінено

❌ НІ → підпис недійсний або дані змінено

Звідки Bob знає pub(A)? → З цифрового сертифіката Alice, виданого довіреним центром сертифікації (CA)

Рекомендовані сьогодні

  • RSA-PSS ≥ 2048 біт — сумісність з legacy-системами
  • ECDSA P-256 — широке застосування (TLS, HTTPS, підпис коду)
  • EdDSA (Ed25519) — найновіший, найшвидший, стійкий до помилок реалізації
TLS 1.3 підтримує лише RSA-PSS, ECDSA та EdDSA (старіші схеми виключені)

Невідреченість (Non-repudiation)

Технічний аспект

  • Підпис перевіряється public key Аліси
  • Public key прив'язаний до Аліси сертифікатом CA
  • CA перевірив особу Аліси при видачі
  • → Аліса не може відмовитися від підпису

Умови забезпечення

  • Захист private key — тільки власник (HSM, smart card)
  • Довіра до CA — надійна перевірка ідентичності
  • Timestamp (TSA) — фіксує момент підписання
  • Цілісність — хеш зафіксований у підписі

Правовий аспект (EU eIDAS)

  • Simple — базовий рівень
  • Advanced — підвищений рівень
  • Qualified — юридично еквівалентний власноручному

Qualified вимагає:

  • Кваліфікований CA (Trusted List)
  • Захищений пристрій підпису (QSCD)
  • Ідентифікація особи

PKI — Public Key Infrastructure

PKI — система, що вирішує фундаментальну проблему: як довіряти відкритому ключу?

Проблема

Bob отримує public_key_alice з інтернету. Як знати, що це справжній ключ Аліси, а не підробка зловмисника?

Рішення — PKI

  • Alice надає CSR → CA перевіряє особу
  • CA підписує public key Alice своїм private key
  • Bob перевіряє підпис CA (якому він довіряє)
  • → Bob довіряє ключу Alice

Прив'язка ключа до ідентичності

Сертифікат = ключ + ідентичність + підпис CA

Розподіл довіри

Ієрархія CA: один Root CA → багато Intermediate CA → кінцеві суб'єкти

Управління lifecycle

Видача, поновлення, відкликання сертифікатів

Стандарт X.509 v3

X.509 — стандарт ITU-T (RFC 5280), що визначає формат сертифіката відкритого ключа

Важливо

Підпис CA охоплює весь блок TBSCertificate → будь-яка зміна будь-якого поля робить підпис недійсним

  • v2 — додано UniqueID (рідко)

Життєвий цикл сертифіката

①

Генерація: пара ключів RSA/ECC

②

CSR: власник формує запит та надсилає до CA

③

Перевірка: CA/RA верифікує особу заявника

④

Видача: CA підписує та повертає сертифікат

⑤

Встановлення: сервер/клієнт встановлює сертифікат

⑥

Використання: TLS, підпис, шифрування e-mail

⑦

Поновлення: за 30–60 днів до закінчення терміну

⑧

Причини відкликання

Компрометація ключа · Зміна даних власника · Видача за помилкових обставин · Ліквідація організації

CSR — Certificate Signing Request

CSR — запит на підпис сертифіката, підписаний власним закритим ключем заявника

1

Генерація пари ключів:

openssl genrsa -out server.key 2048
2

Формування CSR:

openssl req -new -key server.key -out server.csr
3

Надсилання CSR → CA (веб-інтерфейс або ACME)

4

CA перевіряє: DV (DNS/HTTP) · OV (документи) · EV (юридична особа)

5

CA підписує та повертає .crt файл + ланцюжок Intermediate CA

Що містить CSR?

  • Public Key заявника
  • Subject (CN, O, C, ST, L)
  • Підпис self-signed (доказ володіння private key)

⚠️ Важливо

CSR не містить закритого ключа. Private key ніколи не залишає систему власника.

CRL — Certificate Revocation List

CRL — підписаний CA список серійних номерів відкликаних сертифікатів

CRL (підписаний CA):
  Issuer:      CN=DigiCert CA
  This Update: 2024-03-27 10:00 UTC
  Next Update: 2024-03-28 10:00 UTC
  ─────────────────────────────
  Serial: 0x1A2B3C  Revoked: 2024-03-20
  Reason: keyCompromise
  ─────────────────────────────
  Serial: 0x4D5E6F  Revoked: 2024-03-25
  Reason: cessationOfOperation
  ─────────────────────────────
  Signature: <підпис CA>

Причини відкликання

  • keyCompromise
  • cACompromise
  • affiliationChanged
  • superseded
  • cessationOfOperation
  • privilegeWithdrawn

Недоліки CRL

  • Клієнт завантажує весь список
  • Актуальність обмежена інтервалом публікації
  • CRL Caching → застарілі дані

OCSP — Online Certificate Status Protocol

OCSP (RFC 6960) — протокол запиту статусу одного конкретного сертифіката

Client → OCSP Responder:
  OCSP Request:
    SerialNumber:    0x1A2B3C
    IssuerNameHash:  SHA1(IssuerName)
    IssuerKeyHash:   SHA1(IssuerKey)

OCSP Responder → Client:
  OCSP Response:
    Status:       "good"

OCSP Stapling (RFC 6066)

  • Сервер заздалегідь отримує OCSP-відповідь від VA
  • "Прикріплює" її до TLS Handshake
  • Клієнт не звертається до OCSP Responder окремо
  • Менша затримка + збереження приватності

Soft-fail vs Hard-fail

  • Soft-fail: OCSP недоступний → прийнятий (ризик: OCSP-атака)
  • Hard-fail: OCSP недоступний → відхилено (вища безпека)

Контрольні питання

  • У чому принципова відмінність симетричної та асиметричної криптографії? Чому їх комбінують?
  • Поясніть концепцію PFS та чому для неї використовують DHE/ECDHE замість статичного DH.
  • Які три властивості забезпечує цифровий підпис? Що таке невідреченість і як вона досягається технічно?
  • Опишіть процес перевірки цифрового підпису: які ключі використовуються на кожному кроці?
  • Навіщо PKI використовує ієрархію Root CA → Intermediate CA? Яка перевага зберігання Root CA офлайн?
  • Перерахуйте основні поля сертифіката X.509 v3 та поясніть призначення розширень Key Usage та SAN.
  • Що таке CSR і чому закритий ключ не повинен залишати систему власника?
  • Порівняйте CRL та OCSP: у яких сценаріях краще кожен? Що таке OCSP Stapling?
  • Назвіть три типи сертифікатів за рівнем валідації та поясніть різницю між DV, OV та EV.
  • Як PKI використовується в IPsec/IKEv2 для автентифікації вузлів VPN?

Підсумок лекції

Криптографічні алгоритми

  • RSA — факторизація, підпис та шифрування
  • ECC / ECDH — менші ключі, вища ефективність
  • DHE / ECDHE — Perfect Forward Secrecy

Цифровий підпис

  • Автентичність + Цілісність + Невідреченість
  • Private key підписує → Public key верифікує
  • Підписується хеш, а не самі дані

PKI та сертифікати

  • CA прив'язує ключ до ідентичності (X.509)
  • Ієрархія: Root → Intermediate → End-Entity
  • Lifecycle: CSR → видача → використання → відкликання

Відкликання сертифікатів

  • CRL — список від CA (офлайн, але може бути великим)
  • OCSP — онлайн запит статусу
  • OCSP Stapling — відповідь у TLS Handshake
