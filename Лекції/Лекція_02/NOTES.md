# Конспект — Лекція 02 — Асиметричне шифрування, цифрові підписи, PKI та центри сертифікації

### 1. Симетрична та асиметрична криптографія

Симетрична криптографія ґрунтується на використанні **одного спільного секретного ключа** для шифрування і дешифрування. Алгоритми AES, ChaCha20 забезпечують дуже високу швидкість і є основою для захисту масового трафіку. Проте фундаментальна проблема симетричних схем — **проблема розподілу ключа**: як безпечно передати спільний ключ між двома сторонами, якщо вони спілкуються через незахищений канал?

Асиметрична криптографія вирішує цю проблему, використовуючи **пару математично пов'язаних ключів** — відкритий (public key) і закритий (private key). Відкритий ключ можна вільно публікувати; закритий зберігається виключно у власника і ніколи не передається. Математичний зв'язок між ключами дозволяє: те, що зашифровано відкритим ключем, може розшифрувати лише відповідний закритий ключ, і навпаки.

Асиметричні алгоритми є обчислювально значно дорожчими, тому у реальних системах (наприклад, TLS) їх поєднують із симетричними: асиметрична криптографія використовується **тільки для узгодження ключів та підпису**, а масове шифрування даних виконує симетричний алгоритм (AES-GCM тощо).

### 2. RSA

**RSA (Rivest–Shamir–Adleman, 1977)** — найпоширеніший асиметричний алгоритм, безпека якого базується на обчислювальній складності **факторизації** великих цілих чисел: маючи добуток `n = p × q` двох великих простих чисел p та q, практично неможливо знайти ці множники.

Пара ключів RSA генерується наступним чином: обираються два великих простих числа p і q, обчислюється модуль `n = p × q`, після чого визначаються відкрита (`e`) та закрита (`d`) показникові частини. Відкритий ключ — пара `(n, e)`, закритий — `(n, d)`. Шифрування: `C = M^e mod n`; дешифрування: `M = C^d mod n`.

RSA використовується у TLS для підпису під час рукостискання, у протоколах захисту електронної пошти (S/MIME, PGP) та підписі коду. Рекомендований мінімальний розмір ключа — **2048 біт** (придатний до приблизно 2030 р.); для довгострокової безпеки — 3072 або 4096 біт. Головне обмеження RSA порівняно з ECC — значно більший розмір ключа при однаковому рівні криптографічної стійкості.

### 3. ECC (Elliptic Curve Cryptography)

**ECC** спирається на математику еліптичних кривих. Безпека ECC базується на складності задачі **дискретного логарифму на еліптичній кривій (ECDLP)**: знаючи точку P на кривій і результат Q = kP (де k — скалярне множення), знайти k практично неможливо.

Головна перевага ECC — **набагато коротший ключ при еквівалентній криптографічній стійкості**. Наприклад, ключ ECC довжиною 256 біт забезпечує стійкість, еквівалентну 3072-бітному ключу RSA. Це означає менші сертифікати, швидше встановлення з'єднань і нижче навантаження на процесор — що особливо важливо для IoT-пристроїв і мобільних пристроїв.

Найпоширеніші стандартизовані криві: **P-256 (secp256r1)** та **P-384** (NIST, широко застосовуються у TLS та HTTPS) та **Curve25519** (використовується у Tor, Signal, OpenSSH — відрізняється дуже високою швидкістю обчислень та стійкістю до помилок реалізації).

### 4. Diffie-Hellman (DH та ECDH)

**Протокол Diffie-Hellman (1976)** вирішує задачу виробки **спільного секрету через незахищений канал** без безпосередньої його передачі. Обидві сторони обмінюються публічними значеннями (A та B), обчисленими на основі своїх приватних значень (a та b) та спільних публічних параметрів (g, p). Після обміну кожна сторона обчислює однаковий спільний секрет `S = g^(ab) mod p`, але підслуховувач, бачачи лише A, B, g та p, не може відновити S.

Важлива характеристика: DH сам по собі **не забезпечує автентифікації сторін** — потрібне поєднання з цифровим підписом або сертифікатами.

Варіант **ECDH** використовує еліптичні криві, забезпечуючи ті самі переваги (коротші ключі, вища продуктивність). Ще важливіша властивість забезпечується режимами **DHE та ECDHE** — Ephemeral (тимчасові ключі): для кожної сесії генерується нова пара ключів DH, що не зберігається. Це дає **Perfect Forward Secrecy (PFS)**: навіть якщо довгостроковий приватний ключ сервера буде скомпрометовано, розшифрувати трафік попередніх сесій буде неможливо. TLS 1.3 використовує виключно ECDHE для обміну ключами.

### 5. Цифровий підпис

**Цифровий підпис** — криптографічний механізм, що прив'язує повідомлення або документ до ідентичності його підписанта. Він забезпечує три ключові властивості: **автентичність** (підтвердження особи підписанта), **цілісність** (виявлення будь-яких змін після підписання) та **невідреченість** (підписант не може заперечити факт підписання).

На відміну від шифрування, де захист забезпечується через таємницю, підпис ґрунтується на тому, що лише власник закритого ключа може **сформувати** підпис, тоді як перевірити його здатен будь-хто, маючи відкритий ключ.

**Механізм підписання:** спочатку обчислюється **криптографічний хеш** (дайджест) повідомлення — наприклад, SHA-256. Хеш має фіксований розмір незалежно від розміру документа, і будь-яка зміна у вихідних даних повністю змінює хеш. Далі підписант застосовує операцію підпису зі своїм **закритим ключем** до цього хешу (у RSA-PSS це математично пов'язана операція, в ECDSA та EdDSA — окремий алгоритм підпису, не ідентичний шифруванню). Результат — підпис S — надсилається разом із повідомленням.

**Механізм перевірки:** одержувач обчислює хеш отриманого повідомлення (h₁), а також виконує операцію верифікації підпису з **відкритим ключем** підписанта, отримуючи h₂. Якщо h₁ == h₂, підпис дійсний і повідомлення не було змінено.

**Невідреченість** технічно ґрунтується на тому, що закритий ключ зберігається виключно у власника (в ідеалі — на захищеному пристрої, наприклад HSM або smart-card), а сертифікат CA підтверджує зв'язок між відкритим ключем та ідентичністю особи. З юридичного боку в Євросоюзі питання регулює Регламент **eIDAS**, що визначає три рівні підпису: Simple, Advanced та Qualified. Кваліфікований електронний підпис (QES) юридично прирівнюється до власноручного.

Серед сучасних рекомендованих алгоритмів — **ECDSA** (P-256 або P-384, широко застосовується у TLS), **EdDSA / Ed25519** (найновіший, найшвидший, стійкий до помилок реалізації) та **RSA-PSS** (для сумісності з legacy-системами, ключ ≥ 2048 біт). TLS 1.3 підтримує лише RSA-PSS, ECDSA та EdDSA (застарілі схеми RSA PKCS#1 v1.5 та DSA виключено з TLS 1.3, але RSA PKCS#1 sha256/384/512 ще допускаються для підпису сертифікатів).

### 6. PKI — Інфраструктура відкритих ключів

**PKI (Public Key Infrastructure)** — це система, що вирішує фундаментальну проблему: як довіряти відкритому ключу, отриманому через незахищений канал? PKI забезпечує прив'язку відкритого ключа до конкретної ідентичності (особи, організації, сервера) за допомогою **цифрового сертифіката**, підписаного довіреною третьою стороною.

Основні компоненти PKI:
- **CA (Certificate Authority)** — видає, підписує та відкликає сертифікати. CA несе повну відповідальність за достовірність виданих сертифікатів і зобов'язаний дотримуватися задокументованих практик (Certificate Policy та Certification Practice Statement).
- **RA (Registration Authority)** — перевіряє особу заявника перед видачею сертифіката. RA зменшує навантаження на CA, виконуючи адміністративну частину перевірки.
- **VA (Validation Authority)** — відповідає на запити про поточний статус сертифікатів (OCSP).
- **Сховище (Repository)** — зберігає видані сертифікати та актуальні CRL.
- **TSA (Time Stamp Authority)** — проставляє довірені позначки часу на підписи.

**Ієрархія довіри** будується за принципом ланцюжка: Root CA стоїть на вершині, Intermediate CA знаходяться посередині ієрархії, а End-Entity Certificates видаються кінцевим суб'єктам — серверам, користувачам, пристроям. Root CA зберігається **офлайн** задля максимального захисту від компрометації. Його сертифікат вбудовується безпосередньо в операційну систему або браузер (Trust Store). Intermediate CA — операційний рівень ієрархії, він доступний онлайн і видає кінцеві сертифікати. Така схема означає, що при компрометації Intermediate CA Root CA може відкликати лише його сертифікат і видати новий, не змінюючи Trust Store на всіх клієнтах.

### 7. X.509 — Стандарт цифрового сертифіката

**X.509 v3** — стандарт ITU-T (описаний у RFC 5280), що визначає формат сертифіката відкритого ключа. Сертифікат є структурованим набором полів, підписаних CA, і містить:

- **Version** — версія стандарту (v3 є поточним стандартом).
- **Serial Number** — унікальний номер у межах CA (використовується у CRL для ідентифікації відкликаного сертифіката).
- **Signature Algorithm** — алгоритм, яким CA підписав сертифікат (наприклад, `SHA256withRSA` або `ecdsa-with-SHA256`).
- **Issuer** — Distinguished Name (DN) центру сертифікації.
- **Validity (Not Before / Not After)** — терміни дії сертифіката.
- **Subject** — DN власника (для сервера: `CN=example.com`).
- **Subject Public Key Info** — відкритий ключ та алгоритм (RSA, EC тощо).
- **Extensions (v3)** — набір розширень, що значно розширюють функціональність.
- **Signature** — цифровий підпис CA, що охоплює весь блок TBSCertificate. Будь-яка зміна будь-якого поля сертифіката робить підпис недійсним.

Розширення X.509 v3 поділяються на **критичні** (клієнт зобов'язаний їх обробити, інакше — відхилити сертифікат) та **некритичні** (можна ігнорувати). Найважливіші: **Basic Constraints** (чи є суб'єкт CA), **Key Usage** (дозволені операції: digitalSignature, keyEncipherment, keyCertSign), **Subject Alternative Name — SAN** (альтернативні ідентифікатори: DNS-імена, IP-адреси, e-mail), **Extended Key Usage (EKU)** (TLS Server Auth, Code Signing тощо), **CRL Distribution Points (CDP)** та **Authority Information Access (AIA)** (URL OCSP-респондера та URL сертифіката видавця).

### 8. Перевірка ланцюжка та життєвий цикл сертифіката

Коли клієнт підключається до сервера через TLS, він отримує від сервера його сертифікат разом із сертифікатом Intermediate CA. Клієнт виконує **перевірку ланцюжка (chain validation)** поетапно: спочатку перевіряє підпис сертифіката сервера ключем Intermediate CA, потім — підпис Intermediate CA ключем Root CA, після чого переконується, що Root CA присутній у локальному Trust Store. Крім підписів перевіряються також терміни дії кожного сертифіката, статус відкликання (через CRL або OCSP), допустимі операції (Key Usage та EKU) та відповідність Subject/SAN до адреси сервера.

Сертифікат видається за процедурою **CSR (Certificate Signing Request)**: власник генерує пару ключів і формує запит, що містить відкритий ключ і ідентифікаційні дані, підписаний власним закритим ключем. CA перевіряє особу заявника (для DV — контроль DNS або HTTP, для OV — документи організації, для EV — розширена юридична перевірка) і підписує сертифікат. Закритий ключ ніколи не покидає систему власника.

**Відкликання** сертифіката здійснюється через:
- **CRL (Certificate Revocation List)** — підписаний CA список серійних номерів відкликаних сертифікатів, що публікується за розкладом. Недолік: клієнт завантажує весь файл, актуальність обмежена інтервалом публікації.
- **OCSP (Online Certificate Status Protocol, RFC 6960)** — клієнт надсилає запит із серійним номером, VA повертає відповідь `good`, `revoked` або `unknown`. Актуальніший за CRL, але вимагає звернення до OCSP-сервера.
- **OCSP Stapling (RFC 6066)** — сервер заздалегідь кешує підписану OCSP-відповідь і передає її клієнту безпосередньо під час TLS-рукостискання, знижуючи затримку і захищаючи приватність.

### 9. Застосування PKI

PKI є фундаментом безпеки сучасних мереж. У **TLS/HTTPS** сертифікат X.509 дозволяє клієнту переконатися в автентичності сервера; після перевірки ECDHE забезпечує узгодження сеансового ключа без передачі його в незашифрованому вигляді. У **VPN (IKEv2/IPsec)** сертифікати замінюють PSK-автентифікацію, яка важко масштабується: кожен вузол отримує сертифікат від корпоративного CA, і автентифікація в IKE_AUTH виконується через цифровий підпис. У **mTLS** (взаємний TLS) обидві сторони — і сервер, і клієнт — пред'являють свої сертифікати, що широко застосовується у мікросервісних архітектурах та Zero Trust. У **підписі коду** сертифікати підтверджують автентичність програмного забезпечення: ОС перевіряє підпис перед запуском.

| Сценарій | Протокол/Стандарт | Роль PKI |
|----------|------------------|---------|
| **TLS/HTTPS** | TLS 1.2 / 1.3 | Автентифікація сервера, узгодження ключів |
| **mTLS** | TLS з клієнтськими сертифікатами | Взаємна автентифікація |
| **VPN (IPsec/IKEv2)** | IKEv2 (RFC 7296) | Автентифікація вузлів замість PSK |
| **Підпис коду** | Authenticode, APK, JAR | Верифікація ПЗ та оновлень |
| **E-mail (S/MIME)** | RFC 5751 | Підпис та шифрування повідомлень |
| **IoT / Device Auth** | EST (RFC 7030) | Автоматична видача сертифікатів пристроям |

### 11. Certificate Pinning та HSTS

Certificate Pinning — механізм захисту від атак «Man-in-the-Middle» навіть якщо зловмисник має сертифікат, підписаний довіреним CA. Клієнт (браузер або застосунок) заздалегідь «знає» очікуваний сертифікат або public key і відхиляє з'єднання якщо отриманий сертифікат не збігається.

Два варіанти pinning: Certificate pinning — клієнт порівнює весь сертифікат (обмежений термін дії — при оновленні сертифіката застосунок зламається); Public Key pinning — порівнюється лише public key (стабільніший при ротації сертифіката).

HTTP Public Key Pinning (HPKP, RFC 7469): HTTP заголовок Public-Key-Pins, що вказує браузеру «pinned» ключі. Був видалений з Chrome та Firefox через ризик «брикування» сайтів (якщо ключ скомпрометований — сайт стає недоступним). HSTS (HTTP Strict Transport Security, RFC 6797): примушує браузер використовувати HTTPS для домену протягом заданого часу (max-age). Преloaded HSTS список включений у всі major браузери. У VPN-контексті: SSL VPN web-інтерфейси повинні мати HSTS для захисту від SSL-stripping атак.

### 12. OCSP Stapling та Certificate Transparency

OCSP Stapling (RFC 6961): сервер сам запитує OCSP відповідь від CA та включає її у TLS handshake. Переваги: клієнт не звертається до CA (приватність, швидкість), OCSP відповідь кешується сервером. Реалізація: nginx ssl_stapling on, Apache SSLUseStapling On.

Certificate Transparency (CT, RFC 9162): всі видані сертифікати мають публікуватись у відкритих CT log серверах. Браузери вимагають наявності SCT (Signed Certificate Timestamp) від мінімум двох CT log. Це дозволяє: виявляти незаконно видані сертифікати, аудит CA поведінки, спільноті моніторити видачу сертифікатів для їх доменів. Інструмент crt.sh дозволяє шукати всі сертифікати для домену.

### 13. PKI у хмарних середовищах

AWS Certificate Manager (ACM): безкоштовна видача та автооновлення TLS сертифікатів для AWS сервісів (ALB, CloudFront, API Gateway). Не дозволяє експортувати приватний ключ — лише для використання всередині AWS. AWS Private CA: платний сервіс для внутрішніх PKI потреб (видача клієнтських сертифікатів для VPN, mTLS між мікросервісами).

Azure Key Vault: зберігання та управління сертифікатами у Azure. Інтеграція з Azure Application Gateway, Azure Front Door. Підтримка HSM (Hardware Security Module) для захисту приватних ключів. Автооновлення сертифікатів від Let's Encrypt або DigiCert.

Let's Encrypt: безкоштовний, автоматичний CA для публічних доменів. Використовує ACME протокол (Automatic Certificate Management Environment, RFC 8555) для автоматичної видачи та оновлення. Обмеження: лише domain validation (DV), немає організаційної валідації, не підходить для внутрішніх PKI.

### 14. Постквантова криптографія та PKI

NIST Post-Quantum Cryptography стандарти 2024 безпосередньо впливають на PKI та VPN. Алгоритм Шора може зламати RSA та ECDSA при достатньо потужному квантовому комп'ютері — тобто компрометує всю поточну PKI інфраструктуру.

Перехід на постквантові алгоритми: ML-DSA (CRYSTALS-Dilithium, FIPS 204) замінить ECDSA для підпису сертифікатів. ML-KEM (CRYSTALS-Kyber, FIPS 203) замінить ECDH для обміну ключами. Гібридний підхід: поки квантові комп'ютери не стали реальною загрозою, використовуємо ECDSA + ML-DSA разом.

X.509 v3 сертифікати вже зараз можуть містити постквантові публічні ключі (як додаткові розширення). OpenSSL 3.x підтримує PQC алгоритми через liboqs провайдера. Cloudflare та Google вже тестують гібридний TLS з Kyber у production.
