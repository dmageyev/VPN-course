# Конспект — Лекція 08 — Огляд хмарних VPN-рішень. Моніторинг, діагностика та аналіз інцидентів безпеки

### 1. Хмарні VPN: огляд та класифікація

Хмарні VPN-рішення набули широкого поширення разом з міграцією корпоративних інфраструктур до публічних хмарних провайдерів (AWS, Azure, Google Cloud). Вони вирішують завдання безпечного підключення корпоративних офісів та окремих співробітників до хмарних ресурсів, а також забезпечують зашифровану комунікацію між різними хмарними регіонами та між хмарою і on-premises інфраструктурою.

Хмарні VPN можна класифікувати за кількома критеріями. За типом підключення: Site-to-Site VPN (між корпоративною мережею та хмарним VPC/VNet), Client VPN (Remote Access для окремих користувачів до хмарних ресурсів), VPC-to-VPC Peering (між різними хмарними регіонами або аккаунтами). За технологією: IPsec VPN (найпоширеніший стандарт, підтримується всіма major провайдерами), SSL/TLS VPN (для client VPN), MPLS або Dedicated connections (AWS Direct Connect, Azure ExpressRoute, GCP Cloud Interconnect — не VPN, але альтернатива для гарантованої пропускної здатності та нижчої latency).

Ключові виклики хмарних VPN: мультирегіональне розгортання, автоматичне масштабування (auto-scaling VPN Gateways), висока доступність (HA — High Availability), централізоване управління та моніторинг, інтеграція з хмарними IAM системами.

### 2. AWS VPN

AWS пропонує два типи VPN: AWS Site-to-Site VPN та AWS Client VPN.

AWS Site-to-Site VPN: підключення між on-premises мережею та AWS VPC. Компоненти: Virtual Private Gateway (VGW) — термінує VPN на стороні AWS; Customer Gateway (CGW) — конфігурація вашого on-premises пристрою або шлюзу; VPN Connection — конфігурація VPN між VGW та CGW. AWS автоматично надає два тунелі для redundancy (active/active або active/standby). Підтримувані протоколи: IKEv1 та IKEv2, AES-128/256-GCM, AES-128/256-CBC, шифрування phase 1 та phase 2 налаштовуються окремо. Ціноутворення: $0.05/год за VPN Connection плюс трафік.

AWS Transit Gateway: централізований хаб для підключення множини VPC та on-premises мереж. Спрощує управління routing замість mesh двоточкових пірингів. AWS Transit Gateway VPN дозволяє підключити on-premises до Transit Gateway та автоматично розповсюдити маршрути до всіх підключених VPC.

AWS Client VPN: повністю керований сервіс Remote Access VPN на основі OpenVPN-протоколу. Клієнти підключаються через стандартний OpenVPN-клієнт. Аутентифікація: Active Directory (AWS Managed Microsoft AD або on-premises AD через AD Connector), сертифікати (взаємна TLS аутентифікація), Single Sign-On (SAML через AWS IAM Identity Center). Ціноутворення: $0.10/год за ендпойнт плюс $0.05/год за кожне підключення клієнта.

### 3. Azure VPN Gateway

Azure VPN Gateway надає два режими: Route-Based (IKEv2, підтримує Point-to-Site та Site-to-Site) та Policy-Based (тільки IKEv1, тільки Site-to-Site, застарілий).

SKU (цінові плани) Azure VPN Gateway: Basic (100 Мбіт/с, 10 тунелів S2S, застарілий), VpnGw1 (650 Мбіт/с, 30 тунелів), VpnGw2 (1 Гбіт/с, 30 тунелів), VpnGw3 (1.25 Гбіт/с, 30 тунелів), VpnGw4 (5 Гбіт/с), VpnGw5 (10 Гбіт/с). Ціни відрізняються суттєво — від $26/міс для Basic до $2000+/міс для VpnGw5.

Azure Virtual WAN: аналог AWS Transit Gateway. Хаб-and-spoke топологія для глобального з'єднання між офісами та Azure. Два рівні: Basic (лише Site-to-Site VPN) та Standard (S2S + P2S + ExpressRoute + VNet peering).

Point-to-Site (P2S) Azure VPN: Remote Access VPN для окремих клієнтів. Протоколи: OpenVPN (рекомендований), IKEv2, SSTP. Аутентифікація: сертифікати, Azure AD (OIDC/OAuth2), RADIUS.

### 4. Google Cloud VPN

Google Cloud VPN надає захищене підключення між on-premises або іншими хмарними мережами та Google Cloud VPC через IPsec VPN тунелі.

Classic VPN (застарілий): один gateway, один зовнішній IP, підтримує лише статичний routing та policy-based тунелі. Не рекомендований для нових розгортань.

HA VPN (High Availability VPN): рекомендований варіант. Два інтерфейси (external IPs), SLA 99.99% (при двох тунелях до різних peer інтерфейсів), підтримує динамічний routing через BGP (Cloud Router), IKEv2 та IKEv1, AES-256-GCM, ECDH P-256/P-384.

Cloud Router: Google Cloud сервіс, що виконує BGP routing між Cloud VPN та on-premises маршрутизаторами. Автоматично поширює маршрути при підключенні нових підмереж. Є ключовим компонентом для динамічного routing.

### 5. SD-WAN та VPN

SD-WAN (Software-Defined Wide Area Network) — сучасна альтернатива традиційним VPN-рішенням для зв'язку між офісами. SD-WAN управляє трафіком між офісами через централізований controller, оптимізуючи маршрутизацію по кількох каналах (MPLS, Інтернет, 4G/5G) на основі якості з'єднання у реальному часі.

Ключові відмінності SD-WAN від VPN: VPN просто шифрує трафік через один канал; SD-WAN розподіляє трафік між кількома каналами на основі метрик (latency, jitter, packet loss). SD-WAN також надає Application-Aware Routing: відеоконференції — через кращий канал, фонові резервні копії — через дешевший.

SD-WAN провайдери: Cisco Meraki, Fortinet (FortiGate SD-WAN), VMware (VeloCloud), Citrix SD-WAN, Silver Peak (HPE Aruba EdgeConnect). Більшість використовують IPsec або власний шифрований протокол для тунелів між вузлами.

### 6. Zero Trust Network Access (ZTNA)

ZTNA (Zero Trust Network Access) представляє фундаментальний зсув від VPN-моделі: замість надання доступу до мережі, ZTNA надає доступ до конкретних додатків на основі ідентичності та контексту.

Ключові принципи ZTNA: «ніколи не довіряй, завжди перевіряй» (never trust, always verify), мінімальні привілеї (least privilege access), мікросегментація (micro-segmentation), безперервна верифікація (continuous validation), припущення що периметр порушено (assume breach).

Архітектура ZTNA: ZTNA Connector (або Application Connector) — встановлюється поруч з захищеним додатком, ініціює вихідне з'єднання до ZTNA Cloud. ZTNA Gateway/Cloud — посередник між клієнтом та додатком. ZTNA Client — на пристрої користувача, перехоплює трафік до захищених додатків. Policy Engine — оцінює запити доступу на основі identity, device posture, location, behavior.

Переваги ZTNA над VPN: додаток невидимий для Інтернету (знижена attack surface), доступ до конкретного додатку а не до всієї мережі, перевірка стану пристрою перед наданням доступу, краща видимість та аудит доступу.

Провайдери ZTNA: Cloudflare Access, Zscaler Private Access, Palo Alto Prisma Access, Cisco Umbrella ZTNA, Google BeyondCorp Enterprise.

### 7. Метрики якості VPN та інструменти моніторингу

Ефективна експлуатація VPN вимагає постійного моніторингу ключових показників продуктивності. Основні метрики:

Latency (Затримка): час проходження пакета (RTT). У VPN latency завжди вища через шифрування та маршрутизацію через сервер. Нормальні значення: до 20 мс в локальній мережі, 20-100 мс для VPN в тому ж регіоні, 100-300 мс для міжконтинентального VPN. Надмірна latency погіршує досвід інтерактивних застосунків (RDP, VoIP, відеоконференції).

Jitter (Джитер): коливання затримки між пакетами. Критично для голосового та відео трафіку. VoIP потребує jitter < 30 мс. Причини jitter у VPN: нерівномірне навантаження на CPU шифрування, нерівномірна черга пакетів, коливання якості каналу.

Packet Loss (Втрата пакетів): відсоток пакетів, що не досягли призначення. У TCP тунелях навіть 0.1% packet loss може суттєво знизити throughput через TCP congestion control. Норма для VPN: < 0.1%. Причини: перевантаження каналу, неправильний MTU (великі пакети відкидаються), фізичні проблеми з каналом.

Throughput (Пропускна здатність): реальна швидкість передачі даних. Може бути значно нижче номінальної через overhead шифрування, TCP congestion, MTU проблеми.

### 8. Інструменти діагностики та вимірювання

iPerf3: стандартний інструмент для вимірювання пропускної здатності мережі. Запуск сервера: iperf3 -s. Запуск клієнта: iperf3 -c server_ip -t 30 -P 4. Параметр -P визначає кількість паралельних потоків. Результат показує bandwidth у Mbit/s. Для VPN тестування запускають iPerf3 між двома вузлами по обидва боки тунелю — це дає реальний throughput через шифрований тунель. Порівняння результатів з прямим з'єднанням показує overhead від VPN.

MTR (Matt's Traceroute): комбінує функції ping та traceroute. Показує кожен hop по маршруту з живою статистикою: RTT середнє, мінімум, максимум, стандартне відхилення, packet loss на кожному hop. Команда: mtr -n -r destination_ip. Корисна для виявлення: де саме виникає packet loss або висока latency, чи є проблеми на конкретному hop, асиметрії маршрутів.

Wireshark: потужний аналізатор пакетів. Для VPN-діагностики: захоплення на зовнішньому інтерфейсі (шифровані ESP пакети), захоплення на внутрішньому tun/wg інтерфейсі (розшифрований трафік). Дозволяє аналізувати: правильність IKE хендшейку, наявність ESP пакетів, ретрансмісії TCP, DNS-запити всередині тунелю.

Команди для діагностики IPsec: ip xfrm state (активні SA), ip xfrm policy (SPD правила), ip xfrm monitor (живий моніторинг подій xfrm).

### 9. Типові проблеми та їх діагностика

MTU проблеми: симптом — малі пакети (ping, DNS) проходять, великі пакети (HTTP, HTTPS, SSH) не проходять або вкрай повільні. Діагностика: ping -M do -s 1400 destination_ip. Якщо 1400-байтний ping проходить а 1450 — ні, проблема в MTU. Рішення: зменшити MTU тунельного інтерфейсу, налаштувати TCP MSS Clamping.

DNS-витоки: симптом — VPN активний, але DNS-запити йдуть через ISP. Тест: відвідати dnsleaktest.com або викликати dig myip.opendns.com @resolver1.opendns.com. Якщо показує ISP DNS — є витік. Рішення: примусово встановити DNS-сервер через VPN, заблокувати firewall правилами DNS через не-VPN інтерфейси.

Помилки автентифікації: симптом — VPN-з'єднання не встановлюється, у логах «AUTH_FAILED» або «Certificate verification failed». Причини: прострочений сертифікат, неправильний PSK, невірна дата/час на пристрої (сертифікати валідні за часом), відкликаний сертифікат. Перевірка: openssl verify -CAfile ca.crt client.crt.

### 10. Аналіз інцидентів безпеки VPN

Розглянемо реальні інциденти безпеки VPN 2023-2024 років, які стали показовими для галузі.

Ivanti Connect Secure 2024 (CVE-2023-46805 + CVE-2024-21887): CVE-2023-46805 — обхід автентифікації в web component (CVSS 8.2). CVE-2024-21887 — ін'єкція команд (CVSS 9.1). Ланцюжок: обхід auth → ін'єкція → RCE (Remote Code Execution). Група UNC5221 (пов'язана з Китаєм) використовувала цю комбінацію для встановлення веб-шелів WIREFIRE та LIGHTWIRE, крадіжки облікових даних прямо з пам'яті пристрою, повного контролю над VPN-шлюзом.

Уроки з Ivanti-інциденту: навіть спеціалізовані засоби перевірки цілісності (ICT — Integrity Checker Tool) Ivanti були обмануті — зловмисники замінили ICT-бінарник. Будь-яке VPN-обладнання є потенційним вектором атаки. Необхідні термінові патчі при виявленні вразливостей. Моніторинг мережевого трафіку VPN-шлюза.

Cisco ASA SSL VPN (CVE-2023-20269): помилка у web management інтерфейсі, brute force атаки на облікові записи без обмеження спроб. Використана угрупованнями ransomware для первинного доступу.

Загальні уроки: VPN-шлюзи є критичними точками периметру і активно атакуються. Своєчасне оновлення прошивок є обов'язковим. MFA suттєво знижує ризик компрометації навіть при наявності вразливостей. Principle of least privilege для VPN-доступу.

### 11. Hardening VPN-шлюзів

Hardening (посилення захисту) VPN-шлюзів включає кілька ключових заходів.

Управління патчами: регулярне оновлення прошивок та ПО. Підписка на бюлетені безпеки вендора. Тестування патчів на staging перед production. Automatization патчування де можливо.

Вимкнення невикористаних сервісів: закрити всі порти та протоколи, що не потрібні. Вимкнути HTTP (лише HTTPS). Обмежити доступ до management інтерфейсу: лише з довірених IP або окремого management VLAN. Вимкнути SNMP v1/v2 (використовувати SNMPv3 або syslog).

MFA: обов'язкове для всіх VPN-підключень. TOTP (Time-based OTP через Google Authenticator, Authy) або апаратні токени (YubiKey). Інтеграція з RADIUS для централізованого управління. Certificate-based auth як альтернатива або доповнення.

Обмеження доступу: geo-blocking — дозволяти підключення лише з очікуваних країн. IP allowlisting для адміністративного доступу. Rate limiting для неуспішних аутентифікацій. Account lockout після N невдалих спроб.

Моніторинг та SIEM: всі VPN-логи мають відправлятись у централізовану SIEM систему. Alerting на підозрілу активність: одночасні підключення з різних географічних точок, незвичайний час підключення, великий обсяг переданих даних, помилки аутентифікації. Log retention відповідно до вимог compliance (мінімум 90 днів, рекомендовано 1 рік).

### 12. Комплексна стратегія безпеки VPN

Ефективний захист VPN-інфраструктури вимагає комплексного підходу, що поєднує технічні заходи та організаційні процеси.

На рівні мережевої архітектури: VPN-шлюзи розташовуються у DMZ (demilitarized zone), ізольовано від внутрішньої мережі. Трафік від VPN-клієнтів проходить через Next-Gen Firewall з IPS перед доступом до ресурсів. Мікросегментація всередині VPN: різні групи користувачів мають доступ лише до своїх ресурсів через ACL.

На рівні аутентифікації та авторизації: завжди MFA. Сертифікатна автентифікація для корпоративних пристроїв. Перевірка стану пристрою (device compliance) перед наданням доступу: актуальні оновлення безпеки, активний антивірус, заблоковані диски. Регулярний аудит прав доступу та відкликання для звільнених співробітників.

На рівні моніторингу та реагування: SIEM з правилами кореляції для виявлення атак. SOC (Security Operations Center) для аналізу алертів. Incident Response Plan для реагування на компрометацію VPN. Регулярні penetration tests VPN-інфраструктури.

На рівні управління змінами: все VPN-обладнання та ПО має бути в реєстрі активів. Change management процес для будь-яких змін конфігурації. Backup та restore конфігурацій. Тестування failover та відновлення.

### 13. VPN у DevOps та мікросервісах

Сучасні DevOps практики та мікросервісна архітектура змінюють підходи до використання VPN.

GitOps та Infrastructure as Code: конфігурації VPN зберігаються у Git-репозиторії, автоматично розгортаються через CI/CD pipeline (Terraform, Ansible, Helm). Це дозволяє версіонування конфігурацій, аудит змін, відтворюваність інфраструктури.

Kubernetes та VPN: у Kubernetes кластері мережева взаємодія між подами забезпечується CNI (Container Network Interface) плагінами. Деякі CNI (Cilium, Calico) підтримують шифрування міжподового трафіку через WireGuard або IPsec — по суті, «внутрішньокластерний VPN». Для зовнішньої комунікації між кластерами у різних хмарах або регіонах використовуються Overlay мережі поверх існуючих хмарних VPN.

Service Mesh (Istio, Linkerd): реалізує mTLS (mutual TLS) для шифрування трафіку між мікросервісами. Кожен pod отримує власний сертифікат від mesh CA. Всі комунікації між сервісами автоматично шифруються та аутентифікуються. Це альтернатива VPN для внутрішньокластерного трафіку.

### 14. Перспективи розвитку VPN-технологій

VPN-технології продовжують еволюціонувати у відповідь на нові виклики та вимоги.

Постквантова криптографія у VPN: NIST у 2024 році стандартизував перші постквантові алгоритми (ML-KEM, ML-DSA). Провайдери VPN та хмарних рішень вже починають додавати гібридний режим: класичний ECDH + постквантовий Kyber. AWS та Cloudflare оголосили про підтримку постквантових алгоритмів у своїх TLS та VPN реалізаціях.

QUIC-based VPN (MASQUE, RFC 9298): новий стандарт для тунелювання через QUIC-протокол. Переваги: QUIC нативно підтримує multiplexing, 0-RTT відновлення з'єднання, швидку міграцію між мережами (QUIC Connection Migration). Потенційно замінить OpenVPN over TCP для сценаріїв де потрібно обходити рестриктивні firewall.

VPN та AI/ML: машинне навчання використовується для: виявлення аномальної поведінки VPN-користувачів (DLP — Data Loss Prevention), class identification зашифрованого трафіку (для QoS без дешифрування), автоматичного вибору VPN-сервера на основі поточної мережевої якості.

Mesh VPN нового покоління: Tailscale (засноване на WireGuard) та Netbird реалізують peer-to-peer mesh VPN з централізованою координацією. Кожен пристрій отримує постійну IPv6 адресу та може прямо спілкуватись з іншими пристроями через NAT traversal, мінімально залежачи від центрального сервера.


### 15. Аудит VPN-інфраструктури: методологія та інструменти

**Аудит VPN-безпеки** — систематична перевірка стану захисту VPN-інфраструктури, спрямована на виявлення неправильних конфігурацій, застарілих компонентів та вразливостей до їх використання зловмисниками. Регулярний аудит є вимогою більшості стандартів інформаційної безпеки (ISO 27001, SOC 2, PCI DSS).

**Аудит конфігурації** охоплює перевірку криптографічних параметрів: виключення застарілих алгоритмів (3DES, RC4, MD5, SHA-1, DH Group 1/2/5), перевірку використання Perfect Forward Secrecy, наявність MFA, правильну конфігурацію TLS (мінімум 1.2, бажано 1.3). Для IPsec перевіряється наявність IKEv2 (не IKEv1), правильні алгоритми в proposals (AES-256-GCM, ECDHE, SHA-256+). Для OpenVPN: версія OpenVPN ≥ 2.5, tls-version-min 1.2, tls-crypt (не лише tls-auth), verify-x509-name.

**Технічні інструменти аудиту.** `ike-scan` — перевірка IKEv1/IKEv2 конфігурацій: визначення підтримуваних алгоритмів та mode (main/aggressive). `testssl.sh` — комплексна перевірка TLS конфігурації SSL VPN: cipher suites, протоколи, сертифікат, HSTS, заголовки безпеки. `sslyze` — Python-бібліотека для програматичного аналізу TLS. `nmap` зі скриптами `ike-version`, `ssl-enum-ciphers` для виявлення та fingerprinting VPN-шлюзів. `openssl s_client -connect vpn.company.com:443` для ручної перевірки SSL VPN.

**Аудит сертифікатів**: перевірка терміну дії (попередження за 90/30/7 днів), налаштованість CRL/OCSP, правильна ланцюжок до довіреного Root CA, відсутність self-signed у production, наявність SANs замість CN у сучасних сертифікатах. Автоматизовані системи моніторингу сертифікатів (Prometheus SSL exporter, Grafana dashboards) дозволяють відстежувати стан тисяч сертифікатів у реальному часі.

### 16. Реагування на інциденти безпеки VPN

**Incident Response для VPN** вимагає чіткого плану дій, оскільки VPN-інциденти часто мають широкий радіус ураження: компрометація VPN-шлюзу або облікових даних може відкрити зловмиснику доступ до всієї корпоративної мережі.

**Тригери для IR**: масові невдалі спроби автентифікації (brute force), нетипові географічні підключення (неочікувана країна/місто), великий обсяг переданих даних за одну сесію, підключення у нетипові години, спрацювання IDS/SIEM правил на аномалії у VPN-трафіку.

**Фаза Containment (ізоляція).** При підозрі на компрометацію облікових даних: негайно заблокувати підозрілий обліковий запис через Identity Provider (Active Directory, Okta). Якщо підозрюється компрометація самого VPN-шлюзу: ізолювати шлюз на рівні мережі (firewall ACL), перемістити трафік на резервний. Для IPsec: `ip xfrm state flush` та `ip xfrm policy flush` для примусового розриву всіх активних SA на Linux. Документувати всі дії з timestamp та суб'єктами для подальшого розслідування.

**Фаза Investigation (розслідування)** включає аналіз VPN-логів (auth log, syslog, SIEM), мережевих захоплень (pcap), системних логів VPN-шлюзу. Для IPsec ключові артефакти: `ip xfrm state` snapshot, IKE логи (charon.log, /var/log/strongswan.log), netflow для аномалій у трафіку. Forensics VPN-шлюзу: чи були внесені несанкціоновані зміни у конфігурацію? Чи є сліди persistence механізмів (нові SSH-ключі, cron jobs, systemd unit files)?

**Recovery**: після ізоляції та розслідування — патч або відновлення з відомого-хорошого snapshot, ротація всіх ключів та сертифікатів, примусовий скидання паролів, збільшення моніторингу на 30 днів після інциденту.

### 17. Хмарні VPN: AWS Transit Gateway та розширені можливості

**AWS Transit Gateway (TGW)** — регіональний мережевий hub для підключення VPC, VPN та Direct Connect. До появи TGW підключення між VPC будувалося через VPC Peering (без транзитної маршрутизації) або через transit VPC (окремий VPC зі VPN-шлюзами, managed вручну). TGW вирішує проблему масштабування: один TGW може обслуговувати до 5000 VPC attachments та тисячі VPN-підключень.

**TGW VPN** підключає on-premises через IPsec тунелі до Transit Gateway, а не до окремих VGW (Virtual Private Gateway) в кожному VPC. Це суттєво спрощує топологію: замість N×VPN-тунелів (по одному для кожного VPC) — M×VPN-тунелів від TGW, що підключає їх до всіх потрібних VPC через route tables. BGP через TGW VPN дозволяє динамічну маршрутизацію без ручного управління статичними маршрутами.

**TGW Connect** — новіша функція, що використовує GRE тунелі (замість IPsec) між TGW та transit appliances всередині VPC. Це дозволяє досягти throughput до 10 Гбіт/с на GRE attachment (порівняно з 1.25 Гбіт/с для класичного VPN attachment), що критично для bandwidth-intensive сценаріїв.

**AWS Client VPN** базується на OpenVPN та підтримує дві моделі автентифікації: взаємна автентифікація (клієнтські X.509 сертифікати) та Active Directory (через AWS Directory Service або Microsoft AD). Split-tunnel vs Full-tunnel налаштовується у конфігурації ендпоінта. Authorization rules контролюють, які групи AD мають доступ до яких підмереж. VPN endpoint автоматично масштабується горизонтально при зростанні кількості клієнтів без ручного втручання.

### 18. Google Cloud HA VPN та Azure VPN Gateway

**GCP HA VPN** (High Availability VPN) — рекомендоване рішення Google для Site-to-Site підключень з SLA 99.99% (менше 52 хвилин простою на рік). Ключова архітектурна особливість: HA VPN gateway має **два зовнішніх інтерфейси** (два незалежних IP), і рекомендована конфігурація передбачає мінімум 2 тунелі до peer — по одному до кожного інтерфейсу. При відмові одного тунелю трафік автоматично переключається на другий за доли секунди.

**Cloud Router** є невід'ємним компонентом HA VPN GCP. Cloud Router динамічно управляє BGP-сесіями через VPN-тунелі та автоматично розповсюджує маршрути GCP підмереж до on-premises. Налаштування HA VPN без Cloud Router (статичні маршрути) не рекомендується для production, оскільки не підтримує автоматичний failover. Cloud Router підтримує route policy для фільтрації та маніпуляції BGP-маршрутами, що є аналогом route-map у Cisco.

**Azure VPN Gateway** пропонує кілька варіантів по швидкості (SKU): Basic (100 Мбіт/с), VpnGw1 (650 Мбіт/с) до VpnGw5 (10 Гбіт/с). Для HA Azure VPN Gateway підтримує **Active-Active** конфігурацію: два gateway instances з різними публічними IP, кожен підтримує BGP-сесію до on-premises. Azure рекомендує дві пари тунелів (чотири тунелі загалом) для максимальної відмовостійкості. Інтеграція Azure VPN Gateway з **Azure Virtual WAN** дозволяє побудувати SD-WAN топологію у хмарі з автоматичною оптимізацією маршрутів між регіонами, on-premises та VPN клієнтами.

### 19. Постквантова криптографія та майбутнє VPN

**«Harvest now, decrypt later»** — стратегія, активно застосована державними спецслужбами: записати великий обсяг зашифрованого VPN-трафіку сьогодні, щоб розшифрувати його після появи достатньо потужних квантових комп'ютерів. Оцінки часу появи криптографічно релевантних квантових комп'ютерів варіюються від 10 до 20 років, але вже зараз організації, що обробляють дані, які залишаться конфіденційними більше 10 років (урядові, медичні, юридичні), повинні починати міграцію.

**NIST PQC стандарти (2024)**: після 7-річного конкурсу NIST стандартизував перший набір постквантових алгоритмів. **ML-KEM** (CRYSTALS-Kyber, FIPS 203) — для обміну ключами, замінює ECDH. Kyber базується на задачі Module Learning With Errors (MLWE) — вважається стійкою проти квантових комп'ютерів, включаючи алгоритм Шора. **ML-DSA** (CRYSTALS-Dilithium, FIPS 204) та **SLH-DSA** (SPHINCS+, FIPS 205) — для цифрових підписів. AES-256 та SHA-256/384/512 залишаються безпечними проти алгоритму Гровера (лише вдвічі знижує ефективну довжину ключа).

**Гібридний підхід** — рекомендована стратегія міграції. Замість повної заміни класичних алгоритмів, системи використовують обидва: `Classic ECDH + Kyber`. Ключ сесії дорівнює хешу від обох секретів: `session_key = KDF(ECDH_secret || Kyber_secret)`. Це забезпечує безпеку проти як класичних (ECDH), так і квантових (Kyber) атак. Навіть якщо Kyber виявиться вразливим до невідомих атак, система залишається захищеною завдяки ECDH. OpenSSH 9.0 вже реалізував hybrid ECDH + Kyber. AWS TLS 1.3 підтримує ECDH + ML-KEM для окремих кінцевих точок. strongSwan розробляє підтримку PQC для IKEv2.

### 20. VPN телеметрія та поведінковий аналіз

Сучасний підхід до безпеки VPN виходить за межі периметрового захисту і включає постійний **поведінковий аналіз** підключень для виявлення аномалій та потенційних компрометацій.

Ключові поведінкові сигнали для моніторингу: **географічні аномалії** — підключення з незвичайної країни або швидка зміна географії (impossible travel: підключення з Києва та Нью-Йорку з різницею у 30 хвилин фізично неможливе і свідчить про крадіжку credentials); **часові аномалії** — підключення в нетипові години (3 годині ночі для офісного працівника); **обсяг трафіку** — раптовий великий upload може свідчити про data exfiltration; **кількість одночасних сесій** — credential sharing або компрометація.

SIEM-правила для VPN зазвичай реалізуються через **кореляцію подій**: лише один окремий подія аутентифікація з незвичайного IP — не критично, але поєднання «new geo + off-hours + large transfer + many failed logins before success» — це вже серйозний індикатор компрометації. Splunk ES, Microsoft Sentinel та Elastic SIEM надають готові detection rules для VPN-аномалій, які можна кастомізувати під специфіку організації.

Довгострокове зберігання VPN-логів є вимогою для compliance (PCI DSS вимагає 1 рік, частина HIPAA-вимог — до 6 років). Логи повинні включати: timestamp, username, source IP, assigned VPN IP, bytes sent/received, session duration, authentication result, disconnect reason. При централізованому зборі логів у SIEM важливо забезпечити tamper-proof зберігання (незмінні logи), що є критичним для судових розслідувань та підтвердження compliance.
