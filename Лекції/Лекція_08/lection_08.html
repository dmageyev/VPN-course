SD-WAN та його зв'язок з VPN

SD-WAN (Software-Defined WAN) — технологія централізованого програмного управління WAN-з'єднаннями. Використовує IPsec VPN як транспорт, але додає інтелектуальну оркестрацію.

Zero Trust Network Access (ZTNA)

ZTNA — модель безпеки «ніколи не довіряй, завжди перевіряй». Доступ надається на основі ідентичності та контексту, а не мережевого розташування.

Модель VPN (застаріла)

Клієнт → Тунель до мережі → Всі ресурси доступні
Компроміс одного пристрою = доступ до всього.

Модель ZTNA

Клієнт → Перевірка ID + пристрою → Тільки дозволені ресурси
Мікросегментація на рівні кожного додатку.

Ключові принципи ZTNA

  • Мінімальні привілеї (Least Privilege)
  • Безперервна автентифікація та авторизація
  • Мікросегментація доступу
  • Ізоляція додатків від мережевого рівня

Рішення: Zscaler ZPA · Cloudflare Access · BeyondCorp (Google)

Перспективи розвитку VPN-технологій

🔐 Post-Quantum Cryptography

Перехід до алгоритмів, стійких до квантових комп'ютерів: CRYSTALS-Kyber, CRYSTALS-Dilithium. Стандарти NIST PQC затверджені у 2024 р.

🌐 Meshnet / P2P VPN

Децентралізовані моделі без центрального сервера: Tailscale, NetBird на базі WireGuard.

⚡ QUIC-based VPN

Тунелювання поверх QUIC зменшує затримку при перемиканні мереж (0-RTT відновлення сесії).

☁️ SASE

Метрики якості VPN-з'єднань (KPI)

Ефективна експлуатація VPN вимагає постійного моніторингу ключових показників якості (KPI):

МетрикаВизначенняНорма для VPNКритичний поріг
Latency (Затримка) Час проходження пакета (RTT) < 50 мс (LAN), < 150 мс (WAN) > 300 мс
Jitter (Джитер) Коливання затримки < 10 мс > 30 мс (VoIP)
Packet Loss Частка загублених пакетів < 0,1 % > 1 %

Важливо: Затримка у VPN завжди вища за native-з'єднання через шифрування та маршрутизацію через сервер. Нормальна надбавка: +5–30 мс залежно від протоколу.

Latency та Jitter: деталі

Latency — компоненти затримки

RTT = 2 × (Propagation delay
        + Transmission delay
        + Encryption overhead)
  • Фізична відстань (≈ 200 км/мс)
  • Шифрування AES-256 (< 1 мс на CPU)
  • Навантаження на сервер (черги)

Jitter — наслідки

  • Критичний для VoIP — погіршення якості голосу
  • Критичний для відеоконференцій — артефакти
  • Вирішується jitter-buffer на рівні додатку

Вимірювання

ping -c 100 <VPN_server>
# min/avg/max/mdev — показник jitter

Packet Loss: вплив на продуктивність

Формула Mathis (TCP throughput)

Throughput ≈ MSS / (RTT × √PacketLoss)

При RTT=100мс та Loss=1%:
Throughput ≈ 1460 / (0.1 × √0.01)
           ≈ 146 кБ/с
(замість 10+ МБ/с!)

Навіть 1% втрат знижує пропускну здатність у 68 разів!

Причини втрат у VPN

  • Перевантаження каналу між клієнтом і сервером
  • Проблеми з MTU (фрагменти відкидаються)
  • Нестабільний Wi-Fi (остання миля)
  • Перевантаження апаратного шлюза

UDP vs TCP тунелі

OpenVPN over UDP стійкіший до втрат, ніж over TCP (проблема «TCP over TCP» — подвійна ретрансмісія).

iPerf3: вимірювання пропускної здатності

# Запуск сервера (на VPN-хості):
iperf3 -s -p 5201

# TCP-тест (30 секунд):
iperf3 -c 10.0.0.1 -p 5201 -t 30

# UDP-тест (100 Мбіт/с):
iperf3 -c 10.0.0.1 -u -b 100M -t 30

# Паралельні потоки (реальніший сценарій):
iperf3 -c 10.0.0.1 -P 4 -t 30

Інтерпретація результатів

  • Bitrate — фактична пропускна здатність (МБіт/с)
  • Retr — ретрансмісії TCP (> 0 → є congestion)
  • Jitter / Lost — тільки в UDP-режимі

Практика

Порівнюйте результати з VPN і без VPN — різниця = overhead шифрування. Типово WireGuard: -3–8%, OpenVPN: -10–20%.

MTR: динамічний трасувальник

MTR (My Traceroute) — комбінація traceroute та ping, показує затримку та втрати на кожному хопі в реальному часі.

$ mtr --report --report-cycles 100 203.0.113.1

Host                    Loss%   Snt   Last   Avg  Best  Wrst StDev
1. 192.168.1.1           0.0%   100    1.2   1.3   1.1   2.1   0.2
2. 10.20.0.1             0.0%   100    8.5   9.1   8.2  12.4   0.8
3. 203.0.113.1           2.0%   100   45.3  46.7  44.1  89.2   4.1

Діагностика

  • Loss% > 0 на проміжному хопі → проблема у провайдера
  • Loss% тільки на останньому → ICMP-обмеження
  • Різкий стрибок Avg → bottleneck

Корисні опції

mtr -u 203.0.113.1   # UDP-режим
mtr -T 203.0.113.1   # TCP-режим
mtr -n 203.0.113.1   # без DNS lookup

Wireshark: глибокий аналіз пакетів

Фільтри для VPN-діагностики

# IKE (узгодження IPsec):
isakmp

# ESP (зашифрований IPsec):
esp

# WireGuard UDP (порт 51820):
udp.port == 51820

# TCP ретрансмісії (ознака втрат):
tcp.analysis.retransmission

# DF bit (діагностика MTU):
ip.flags.df == 1 && icmp

Важливо для VPN

Зашифровані тунелі (ESP/WireGuard) не дозволяють аналізувати внутрішній трафік. Аналізуйте на TUN-інтерфейсі клієнта до шифрування.

Корисні функції

  • Statistics → I/O Graph (графік трафіку)
  • Statistics → TCP Stream Graphs (RTT)
  • Follow → TCP Stream (потік з'єднання)

Типові несправності: MTU-проблеми

Симптоми MTU-проблем

  • Малі пакети (ping) — проходять нормально
  • HTTP/HTTPS-сторінки — зависають або дуже повільні
  • SSH — підключається, але виведення «зависає»
  • Файлові передачі — починаються, потім зупиняються

Діагностика та виправлення

# Пошук правильного MTU:
ping -M do -s 1472 <VPN_server>
ping -M do -s 1400 <VPN_server>
ping -M do -s 1350 <VPN_server>

# Поточний MTU тунелю:
ip link show tun0 | grep mtu

# Встановлення MTU:
ip link set tun0 mtu 1420

# MSS Clamping (iptables):
iptables -t mangle -A FORWARD \
  -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu

DNS-витоки: виявлення та усунення

Що таке DNS-витік?

DNS-запити обходять VPN-тунель і передаються через ISP, розкриваючи активність користувача — навіть якщо сам трафік зашифрований.

  1. Хост підключається до VPN (Split-Tunnel)
  2. DNS налаштований на ISP-сервер
  3. DNS-запити йдуть напряму через ISP
  4. ISP бачить всі запити → витік!

Перевірка та виправлення

# Поточний DNS:
cat /etc/resolv.conf

# Онлайн-перевірка:
# dnsleaktest.com
# ipleak.net
  • Вказати DNS = 10.0.0.1 у конфіг WireGuard/OpenVPN
  • Увімкнути DNS leak protection у клієнті
  • Використовувати DoH/DoT поверх тунелю

Кейс-стаді: масові зломи VPN (2024)

У 2024–2025 роках виявлено критичні вразливості в корпоративних VPN-шлюзах Ivanti Connect Secure (раніше — Pulse Secure).

Хронологія подій

  • Грудень 2023 — виявлено CVE-2023-46805 та CVE-2024-21887
  • Січень 2024 — CISA видала екстрене попередження для федеральних агентств США
  • Лютий 2024 — понад 2100 організацій скомпрометовано
  • 2025 — CVE-2025-0282 продовжує активно використовуватись

Масштаб атаки

  • Урядові, оборонні та фінансові організації
  • 20+ країн по всьому світу
  • Хакерська група: UNC5221 (пов'язана з китайськими APT)
  • Мета: шпигунство та lateral movement

Технічний аналіз вразливостей Ivanti

CVE-2023-46805 — Обхід автентифікації

CVSS: 8.2 High

Маніпуляція URL-шляхами у веб-інтерфейсі для обходу перевірки автентифікації без будь-яких облікових даних.

CVE-2024-21887 — Ін'єкція команд

CVSS: 9.1 Critical

Виконання довільних команд ОС через вразливий endpoint. У комбінації з першою — без автентифікації!

Ланцюжок атаки (UNC5221)

1. HTTP-запит з маніпульованим URL
   → CVE-2023-46805 (обхід auth)
      ↓
2. Ін'єкція команди у endpoint
   → CVE-2024-21887 (RCE)
      ↓
3. Встановлення бекдору
   (GLASSTOKEN web shell)
      ↓
4. Lateral movement у мережу

Уроки з аналізу інциденту Ivanti

Що пішло не так

  • VPN-шлюзи з прямим доступом з Інтернету (немає DMZ)
  • Відсутній моніторинг аномалій на шлюзі
  • Затримка патчів — тижні замість годин
  • Немає сегментації: шлюз → доступ до всієї мережі

Індикатори компрометації (IoC)

- Аномальні HTTP-запити
  з маніпульованими URL
- Нові файли у
  /data/runtime/mtmp/
- Незвичайні вихідні з'єднання
  з VPN-шлюза
- Зміни конфігурації без
  адмін-сесій

Реакція CISA: рекомендувала відключити всі Ivanti-шлюзи та виконати повне відновлення з перевіреного образу (factory reset + оновлення прошивки).

Hardening VPN-шлюзів

Мінімізація поверхні атаки

  • Відключити невикористовувані сервіси та протоколи
  • Вимкнути застарілі: SSLv3, TLS 1.0/1.1, IKEv1
  • Відключити веб-консоль з публічного Інтернету
  • Обмежити SSH до IP-адрес управління (allowlist)
  • Вимкнути SNMP v1/v2 → тільки SNMPv3

Автентифікація та доступ

  • Обов'язкове MFA для всіх VPN-користувачів
  • Сертифікати замість PSK (де можливо)
  • Обмеження кількості сеансів на користувача
  • Тайм-аут сесії — максимум 8 годин
  • RBAC: мінімально необхідні привілеї

Комплексна стратегія безпеки VPN

Defense-in-Depth — багаторівневий захист, де кожен рівень стримує атаки, якщо попередній подолано.

Периметр: Anti-DDoS → WAF → Firewall
↓
Шлюз: VPN Gateway ← Патчі та Hardening
↓
Доступ: MFA + PKI → RBAC → Least Privilege
↓
Мережа: Сегментація → IDS/IPS → Zero Trust
↓
Моніторинг: SIEM → SOC → Incident Response

Ключовий принцип: Жоден окремий захід не є достатнім. Тільки багаторівневий захист ефективно протистоїть APT-атакам класу UNC5221.
