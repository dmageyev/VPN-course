## Слайд 3 — Хмарні VPN: огляд

# Хмарні VPN-рішення

**Хмарні VPN** — керовані постачальниками хмарних послуг рішення для безпечного підключення корпоративних мереж до хмарної інфраструктури.

**Три основні платформи:**

| Постачальник | Сервіс | Протоколи |
|---|---|---|
| **Amazon AWS** | AWS Site-to-Site VPN / Client VPN | IPsec, OpenVPN |
| **Microsoft Azure** | Azure VPN Gateway | IKEv2, SSTP, OpenVPN |
| **Google Cloud** | Cloud VPN (HA VPN) | IKEv2/IPsec |

**Спільні риси:**
- Висока доступність (SLA 99.9 %+)
- Масштабування без додаткового обладнання
- Інтеграція з IAM та хмарним моніторингом

## Слайд 4 — AWS VPN

# AWS Site-to-Site VPN та Client VPN

**AWS Site-to-Site VPN** — з'єднує корпоративну мережу з AWS VPC через IPsec-тунелі.

```
Корпоративна мережа ──IPsec──▶ Virtual Private Gateway ──▶ AWS VPC
```

**Ключові характеристики:**
- Два резервні тунелі (Active/Standby) для HA
- Пропускна здатність: до 1,25 Гбіт/с на тунель
- Статична або динамічна маршрутизація (BGP)

**AWS Client VPN** — доступ окремих користувачів до AWS-ресурсів:
- Протокол: OpenVPN (порт 443 TCP/UDP)
- Інтеграція з AWS IAM та Active Directory
- Підтримка Split-Tunnel для оптимізації трафіку

## Слайд 5 — Azure VPN Gateway

# Azure VPN Gateway

**Azure VPN Gateway** — забезпечує зашифровані з'єднання між Azure VNet і корпоративними мережами.

**Типи шлюзів:**

| SKU | Пропускна здатність | Призначення |
|---|---|---|
| Basic | 100 Мбіт/с | Розробка/тестування |
| VpnGw1 | 650 Мбіт/с | Малий бізнес |
| VpnGw3 | 1,25 Гбіт/с | Корпоративний рівень |
| VpnGw5 | 10 Гбіт/с | Великі підприємства |

**Протоколи клієнтського VPN (Point-to-Site):**
- IKEv2 — для Windows, macOS, Linux
- SSTP — тільки Windows (порт 443)
- OpenVPN — кросплатформений

## Слайд 6 — Google Cloud VPN

# Google Cloud VPN

**Cloud VPN** — з'єднує мережу Google Cloud з on-premises мережами або іншими хмарами через IPsec.

**HA VPN (High Availability VPN):**
- Два зовнішніх IP та два тунелі (Active/Active)
- SLA 99,99 % (на відміну від Classic VPN — 99,9 %)
- Автоматичне перемикання при збої тунелю

```
On-Premises ──IKEv2──▶ Cloud VPN Gateway ──▶ GCP VPC
             ══IKEv2══▶  (резервний)     ══▶
```

**Особливості:**
- Підтримка тільки IKEv2 (Classic VPN підтримує IKEv1)
- Динамічна маршрутизація через Cloud Router (BGP)
- Максимальна пропускна здатність: 3 Гбіт/с на тунель

## Слайд 7 — SD-WAN та VPN

# SD-WAN та його зв'язок з VPN

**SD-WAN (Software-Defined Wide Area Network)** — технологія централізованого програмного управління WAN-з'єднаннями.

**Відмінність від традиційного VPN:**

| Критерій | VPN | SD-WAN |
|---|---|---|
| Управління | Ручне / децентралізоване | Централізований контролер |
| Вибір шляху | Статичний | Динамічний (QoS-aware) |
| Видимість | Обмежена | Повна аналітика |
| Резервування | Вручну | Автоматичне |

**SD-WAN використовує VPN як транспорт:**
- IPsec-тунелі між vEdge/vSmart пристроями
- Динамічний вибір каналу (MPLS, LTE, broadband)
- Приклади: Cisco Viptela, VMware SD-WAN, Fortinet

## Слайд 8 — Zero Trust Network Access (ZTNA)

# Zero Trust Network Access (ZTNA)

**ZTNA** — модель безпеки «ніколи не довіряй, завжди перевіряй», де доступ до ресурсів надається на основі ідентичності та контексту, а не мережевого розташування.

**Принципові відмінності від VPN:**

```
VPN:   Клієнт ──▶ Тунель до мережі ──▶ Всі ресурси доступні

ZTNA:  Клієнт ──▶ Перевірка ID+пристрою ──▶ Тільки дозволені ресурси
```

**Ключові принципи ZTNA:**
- Мінімальні привілеї (Least Privilege)
- Безперервна автентифікація та авторизація
- Мікросегментація доступу
- Ізоляція додатків від мережевого рівня

**Приклади рішень:** Zscaler Private Access, Cloudflare Access, BeyondCorp (Google)

- **Terraform/Ansible** — управління VPN-шлюзами як кодом (IaC)
- **Hashicorp Vault** — розподіл секретів через VPN-захищений канал

## Слайд 11 — Метрики якості VPN

# Метрики якості VPN-з'єднань (KPI)

Ефективна експлуатація VPN вимагає постійного моніторингу ключових показників якості (KPI):

**Три основні метрики:**

| Метрика | Визначення | Норма для VPN | Критичний поріг |
|---|---|---|---|
| **Latency (Затримка)** | Час проходження пакета (RTT) | < 50 мс (LAN), < 150 мс (WAN) | > 300 мс |
| **Jitter (Джитер)** | Коливання затримки | < 10 мс | > 30 мс (для VoIP) |
| **Packet Loss (Втрата пакетів)** | Частка загублених пакетів | < 0,1 % | > 1 % |

**Важливо:** Затримка у VPN завжди вища за native з'єднання через шифрування та маршрутизацію через сервер. Нормальна надбавка: +5–30 мс залежно від протоколу.

## Слайд 12 — Latency та Jitter

# Latency та Jitter: деталі

**Latency (Затримка)** складається з кількох компонентів:

```
RTT = 2 × (Propagation delay + Transmission delay + Encryption overhead)
```

**Джерела затримки у VPN:**
- Фізична відстань до VPN-сервера (швидкість світла ≈ 200 км/мс)
- Час шифрування/дешифрування (AES-256: < 1 мс на сучасному CPU)
- Навантаження на сервер (черги пакетів)

**Jitter (Джитер)** — дисперсія затримки:
- Критичний для VoIP (погіршує якість голосу)
- Критичний для відеоконференцій (артефакти зображення)
- Вирішується буферизацією (jitter buffer) на рівні додатку

**Вимірювання:** `ping -c 100 <VPN_server>` → аналіз min/avg/max/mdev

## Слайд 13 — Packet Loss

# Packet Loss: вплив на продуктивність

**Packet Loss (Втрата пакетів)** прямо впливає на пропускну здатність, особливо у TCP-тунелях.

**Вплив втрат на TCP-пропускну здатність (формула Mathis):**

```
Throughput ≈ MSS / (RTT × √PacketLoss)

При RTT = 100 мс та Loss = 1%:
Throughput ≈ 1460 / (0.1 × √0.01) ≈ 146 кБ/с  (замість 10+ МБ/с!)
```

**Причини втрат пакетів у VPN:**
- Перевантаження каналу між клієнтом і VPN-сервером
- Проблеми з MTU (фрагментовані пакети відкидаються)
- Нестабільний Wi-Fi (на останній милі)
- Проблеми з апаратним забезпеченням VPN-шлюза

**UDP vs TCP тунелі:** OpenVPN over UDP стійкіший до втрат, ніж over TCP (проблема «TCP over TCP»).

## Слайд 14 — iPerf3

# iPerf3: вимірювання пропускної здатності

**iPerf3** — утиліта для вимірювання максимальної пропускної здатності мережі.

**Встановлення сервера та клієнта:**

```bash
# На VPN-сервері (або цільовому хості):
iperf3 -s -p 5201

# На клієнті — TCP тест:
iperf3 -c 10.0.0.1 -p 5201 -t 30

# UDP тест з цільовою пропускною здатністю 100 Мбіт/с:
iperf3 -c 10.0.0.1 -u -b 100M -t 30
```

**Інтерпретація результатів:**
- `Bitrate` — фактична пропускна здатність (МБіт/с)
- `Retr` — кількість TCP-ретрансмісій (> 0 → є втрати або congestion)
- `Jitter` / `Lost` — тільки в UDP-режимі

**Важливо:** Порівнюйте результати з VPN і без VPN для визначення overhead шифрування.

## Слайд 15 — MTR

# MTR: динамічний трасувальник

**MTR (My Traceroute)** — комбінація `traceroute` та `ping`, що показує затримку та втрати на кожному хопі в реальному часі.

```bash
# Запуск MTR до VPN-сервера:
mtr --report --report-cycles 100 203.0.113.1

# Приклад виводу:
Host              Loss%   Snt   Last   Avg  Best  Wrst StDev
1. 192.168.1.1    0.0%    100   1.2    1.3   1.1   2.1   0.2
2. 10.20.0.1      0.0%    100   8.5    9.1   8.2  12.4   0.8
3. 203.0.113.1    2.0%    100  45.3   46.7  44.1  89.2   4.1
```

**Діагностика за допомогою MTR:**
- `Loss%` > 0 на проміжному хопі → проблема у провайдера
- `Loss%` тільки на останньому хопі → можливо ICMP-обмеження
- Різкий стрибок `Avg` → bottleneck на конкретному маршрутизаторі

## Слайд 16 — Wireshark

# Wireshark: глибокий аналіз пакетів

**Wireshark** — аналізатор мережевого трафіку для детальної діагностики VPN-з'єднань.

**Корисні фільтри для аналізу VPN:**

```
# Показати тільки IKE (фаза узгодження IPsec):
isakmp

# Аналіз ESP-пакетів (зашифрований IPsec трафік):
esp

# UDP трафік WireGuard (порт 51820):
udp.port == 51820

# Виявлення ретрансмісій TCP (ознака втрат):
tcp.analysis.retransmission

# Пакети з DF bit = 1 (для діагностики MTU):
ip.flags.df == 1 && icmp
```

**Важливо для VPN-трафіку:** Зашифровані тунелі (ESP/WireGuard) не дають змоги аналізувати внутрішній трафік — аналізуйте на TUN-інтерфейсі клієнта до шифрування.

## Слайд 17 — Діагностика: MTU-проблеми

# Типові несправності: MTU-проблеми

**Симптоми MTU-проблем у VPN:**
- Малі пакети (ping < 1400 байт) проходять нормально
- Великі сторінки (HTTP/HTTPS) або файли — зависають або завантажуються дуже повільно
- SSH-підключення працює, але виведення команд «зависає»

**Діагностика MTU:**

```bash
# Знаходження правильного MTU методом перебору:
ping -M do -s 1472 <VPN_server>   # 1472 + 28 = 1500 байт
ping -M do -s 1400 <VPN_server>   # якщо попередній не пройшов
ping -M do -s 1350 <VPN_server>   # зменшуємо до успіху

# Перевірка поточного MTU тунелю:
ip link show tun0 | grep mtu

# Встановлення правильного MTU:
ip link set tun0 mtu 1420
```

**Рішення:** Встановити MTU тунелю на 1420 байт (WireGuard) або налаштувати TCP MSS Clamping.

## Слайд 18 — Діагностика: DNS-витоки

# DNS-витоки: виявлення та усунення

**DNS-витік (DNS Leak)** — ситуація, коли DNS-запити обходять VPN-тунель і передаються через ISP, розкриваючи активність користувача.

**Симптоми та перевірка:**

```bash
# Перевірка поточного DNS-сервера:
cat /etc/resolv.conf
nslookup myip.opendns.com resolver1.opendns.com

# Онлайн-перевірка:
# https://dnsleaktest.com
# https://ipleak.net
```

**Типові причини DNS-витоків у VPN:**
- DNS налаштований на ISP-сервер, а не на VPN-сервер
- Split-tunnel не включає DNS-трафік у тунель
- ОС кешує старий DNS-сервер після підключення VPN

**Виправлення:**
- Вказати `DNS = 10.0.0.1` у конфігурації WireGuard/OpenVPN
- Увімкнути `DNS leak protection` у клієнтському ПЗ
- Використати DoH/DoT поверх VPN-тунелю

## Слайд 19 — Помилки автентифікації

# Помилки автентифікації у VPN

**Найпоширеніші помилки та їх причини:**

| Помилка | Причина | Рішення |
|---|---|---|
| `AUTH_FAILED` | Неправильний PSK або сертифікат | Перевірити ключі, оновити сертифікати |
| `TLS handshake failed` | Невідповідність cipher suite | Узгодити алгоритми між клієнтом і сервером |
| `Certificate expired` | Прострочений TLS-сертифікат | Оновити сертифікат, налаштувати auto-renew |
| `Clock skew` | Розбіжність часу > 5 хв (IKE/Kerberos) | Синхронізувати NTP на всіх вузлах |
| `No response from server` | Фаєрвол блокує UDP 500/4500 | Відкрити порти для IKE/NAT-T |

**Діагностика IKEv2:**

```bash
# Перегляд логів strongSwan:
journalctl -u strongswan -f

# Детальний вивід OpenVPN:
openvpn --config client.ovpn --verb 4
```

## Слайд 21 — Технічний аналіз: CVE Ivanti

# Технічний аналіз вразливостей Ivanti

**CVE-2023-46805** — обхід автентифікації (Authentication Bypass):
- CVSS Score: **8.2 (High)**
- Атака: маніпуляція URL-шляхами в web-інтерфейсі для обходу перевірки автентифікації

**CVE-2024-21887** — ін'єкція команд (Command Injection):
- CVSS Score: **9.1 (Critical)**
- Атака: виконання довільних команд через автентифікований запит (в комбінації з CVE-2023-46805 — без автентифікації!)

**Ланцюжок атаки (UNC5221 / APT):**

```
1. HTTP-запит з маніпульованим URL (обхід auth) 
   → CVE-2023-46805
2. Ін'єкція команди у вразливий endpoint
   → CVE-2024-21887
3. Встановлення бекдору (GLASSTOKEN web shell)
4. Lateral movement у корпоративну мережу
```

## Слайд 22 — Уроки з Ivanti-інциденту

# Уроки з аналізу інциденту Ivanti

**Що пішло не так:**
- VPN-шлюзи мали прямий доступ з Інтернету (немає DMZ)
- Відсутній моніторинг аномальної активності на шлюзі
- Затримка з випуском та застосуванням патчів (тижні замість годин)
- Відсутність сегментації: скомпрометований шлюз = доступ до всієї мережі

**Індикатори компрометації (IoC):**
```
- Аномальні HTTP-запити з маніпульованими URL-шляхами
- Нові файли у /data/runtime/mtmp/ на пристрої
- Незвичайні вихідні з'єднання з VPN-шлюза
- Модифікації конфігурації без адмін-сесій
```

**Реакція:** CISA рекомендувала відключити всі Ivanti-шлюзи та виконати повне відновлення з перевіреного образу.

## Слайд 23 — Hardening VPN-шлюзів

# Hardening VPN-шлюзів: найкращі практики

**Мінімізація поверхні атаки:**

```
✅ Відключити всі невикористовувані сервіси та протоколи
✅ Вимкнути старі протоколи: SSLv3, TLS 1.0/1.1, IKEv1
✅ Відключити веб-консоль управління з публічного Інтернету
✅ Обмежити SSH-доступ до адмін-інтерфейсу тільки з IP управління
✅ Вимкнути SNMP v1/v2 (використовувати тільки SNMPv3)
```

**Аутентифікація та доступ:**

```
✅ Обов'язкове MFA (багатофакторна автентифікація) для всіх користувачів
✅ Сертифікати замість PSK (Pre-Shared Key) де можливо
✅ Обмеження кількості сеансів на користувача
✅ Налаштувати тайм-аут сесії (максимум 8 годин)
```

## Слайд 24 — Управління патчами

# Управління патчами для VPN-інфраструктури

**Процес Patch Management:**

```
1. Підписка на CVE-повідомлення постачальника
   (Ivanti, Palo Alto, Fortinet, Cisco newsletters)
   ↓
2. Оцінка CVSS Score та наявності PoC-експлойту
   ↓
3. Тестування патча у staging-середовищі
   ↓
4. Розгортання у Production (maintenance window)
   ↓
5. Верифікація та документування
```

**Критерії пріоритизації:**
- CVSS ≥ 9.0 + активна експлуатація → **патч протягом 24–48 годин**
- CVSS 7.0–8.9 → протягом 7 днів
- CVSS < 7.0 → плановий патч-цикл (30 днів)

**Інструменти:** Tenable Nessus, Qualys, OpenVAS для сканування вразливостей.

## Слайд 25 — Моніторинг та SIEM

# Моніторинг VPN-інфраструктури та SIEM

**Що логувати на VPN-шлюзі:**

```
✅ Всі успішні та невдалі автентифікації (з IP, user-agent)
✅ Зміни конфігурації (хто, коли, що змінив)
✅ Аномальна кількість з'єднань від одного IP
✅ Підключення з нестандартних географічних локацій
✅ Об'єм переданого трафіку (виявлення exfiltration)
```

**Інтеграція з SIEM (Security Information and Event Management):**

| SIEM-платформа | Інтеграція |
|---|---|
| Splunk | Forwarder + VPN App |
| Microsoft Sentinel | Connector для Cisco/Palo Alto |
| Elastic SIEM | Filebeat + Syslog |
| IBM QRadar | DSM для основних вендорів |

**Алерти:** Налаштувати сповіщення для Brute Force, географічних аномалій та масових відключень.

## Слайд 26 — Комплексна стратегія безпеки

# Комплексна стратегія безпеки VPN

**Defense-in-Depth для VPN-інфраструктури:**

```
Периметр:   [ Інтернет ] → [ Anti-DDoS ] → [ WAF/Firewall ]
                                                    ↓
Шлюз:       [ VPN Gateway ] ← [ Патчі та Hardening ]
                                                    ↓
Доступ:     [ MFA + PKI ] → [ RBAC ] → [ Least Privilege ]
                                                    ↓
Мережа:     [ Сегментація ] → [ IDS/IPS ] → [ Zero Trust ]
                                                    ↓
Моніторинг: [ SIEM ] → [ SOC ] → [ Incident Response ]
```

**Ключовий принцип:** Жоден окремий захід не є достатнім. Тільки багаторівневий захист ефективно протистоїть APT-атакам.
