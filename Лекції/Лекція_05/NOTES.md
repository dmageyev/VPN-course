# Конспект — Лекція 05 — Протоколи IPsec

IPsec (Internet Protocol Security) — набір відкритих стандартів для захисту IP-трафіку на мережевому рівні (L3). Забезпечує конфіденційність, цілісність та автентифікацію без зміни застосунків — прозоро для рівня застосунків. Розроблений IETF, описаний у серії RFC (2401, 4301, 7296 та ін.). IPsec підтримується нативно в ядрі Linux (xfrm), Windows, macOS та iOS.

### 1. Архітектура IPsec

IPsec — не один протокол, а ціла архітектура, що включає кілька компонентів: протоколи захисту даних (AH та ESP), бази даних управління SA та SP, та протокол обміну ключами IKE.

Ключові концепції архітектури:
- Security Association (SA) — однонаправлений захищений канал між двома вузлами
- Security Association Database (SAD) — база даних активних SA
- Security Policy Database (SPD) — база даних правил: які пакети і як захищати
- Security Parameter Index (SPI) — 32-бітний ідентифікатор SA, що міститься у кожному IPsec пакеті

Коли IP-пакет надходить на IPsec-шлюз, механізм IPsec спочатку перевіряє SPD: що робити з цим пакетом? Є три дії: PROTECT (захистити за допомогою IPsec), BYPASS (пропустити без захисту), DISCARD (відкинути). Якщо дія PROTECT, шлюз шукає відповідну SA у SAD. Якщо SA немає — запускає IKE для встановлення нової SA.

### 2. Security Association (SA)

Security Association (SA) — це однонаправлений логічний «договір» між двома IPsec-вузлами, що визначає всі параметри захисту конкретного потоку трафіку. SA є однонаправленою: для захисту двостороннього трафіку між вузлами A і B потрібні дві SA: одна для трафіку A до B, інша — для B до A.

SA визначає такі параметри: SPI (Security Parameter Index) — унікальний ідентифікатор, протокол захисту (AH або ESP), алгоритм шифрування та ключ (наприклад, AES-256-GCM), алгоритм автентифікації та ключ (HMAC-SHA256), режим (Transport або Tunnel), Sequence Number Counter для захисту від Replay атак, термін дії SA (за часом або обсягом переданих даних).

SAD (Security Association Database) — таблиця всіх активних SA на вузлі. Індексується за трійкою: (SPI, dst IP, протокол). При отриманні ESP або AH пакета вузол читає SPI з заголовка та шукає відповідну SA у SAD для дешифрування.

SPD (Security Policy Database) — таблиця правил, що визначає, що робити з кожним пакетом. Правила задаються за селекторами: src IP/порт, dst IP/порт, протокол. Пріоритет правил визначається порядком у SPD. Приклад: весь трафік між 10.0.1.0/24 та 10.0.2.0/24 — PROTECT ESP Tunnel Mode; трафік до 8.8.8.8 — BYPASS.

### 3. Протокол AH (Authentication Header)

AH (Authentication Header, RFC 4302) — протокол IP номер 51. Забезпечує: цілісність IP-заголовка та payload, автентифікацію джерела, захист від Replay-атак (через Sequence Number). AH не шифрує дані — він лише автентифікує.

Формат заголовка AH:
- Next Header (8 біт): номер протоколу наступного заголовка (TCP=6, UDP=17, ESP=50)
- Payload Length (8 біт): довжина AH заголовка у 32-бітних словах мінус 2
- Reserved (16 біт): зарезервовано (нулі)
- SPI (32 біти): індекс Security Association
- Sequence Number (32 біти): монотонно зростаючий лічильник, захищає від Replay
- ICV (Integrity Check Value): змінна довжина (96-256 біт); HMAC-SHA256 дає 96 біт; обчислюється над незмінними полями IP-заголовка + AH заголовок + payload

Ключова проблема AH: включає незмінні поля IP-заголовка (src IP, dst IP) у свій MAC. NAT змінює IP-адреси — тому AH повністю несумісний з NAT. Сучасні VPN-розгортання здебільшого використовують лише ESP, оскільки AH несумісний з NAT-T.

### 4. Протокол ESP (Encapsulating Security Payload)

ESP (Encapsulating Security Payload, RFC 4303) — протокол IP номер 50. Забезпечує: конфіденційність (шифрування payload), цілісність та автентифікацію payload, захист від Replay-атак. На відміну від AH, ESP не включає зовнішній IP-заголовок у MAC — тому сумісний з NAT.

Формат ESP пакета:
1. ESP Header: SPI (32 біти) + Sequence Number (32 біти) — відкрито, не шифровано
2. IV/Nonce (довжина залежить від алгоритму): для AES-CBC 16 байтів, для AES-GCM 8 байтів implicit nonce
3. Payload (зашифровано): inner IP-пакет (Tunnel Mode) або вищі шари (Transport Mode)
4. Padding (0-255 байтів): вирівнювання для блочних шифрів та маскування довжини пакета
5. Pad Length (8 біт) + Next Header (8 біт): в зашифрованій частині
6. ICV (Integrity Check Value): HMAC або AEAD тег, обчислюється над ESP Header + зашифрованим payload

AEAD режими (AES-GCM, ChaCha20-Poly1305): шифрування та автентифікація виконуються одночасно. SPI та Sequence Number є «додатковими автентифікованими даними» (AAD) — вони не шифруються, але включаються у тег автентифікації, тому не можуть бути змінені без виявлення.

ESP є de facto стандартним протоколом для сучасного IPsec. AH використовується рідко і лише у середовищах без NAT, де потрібна автентифікація IP-заголовка.

### 5. Transport Mode vs Tunnel Mode

IPsec підтримує два режими застосування: Transport Mode та Tunnel Mode. Вибір режиму залежить від топології та вимог.

Transport Mode: зберігається оригінальний IP-заголовок пакета. IPsec захищає лише payload (TCP/UDP сегмент або вищий рівень). ESP у Transport Mode: оригінальний IP заголовок + ESP заголовок + зашифрований TCP/UDP + ESP трейлер + ICV. Джерела та призначення IP-адрес видно. Використовується для Host-to-Host захисту між конкретними вузлами. Менший overhead (немає другого IP-заголовка).

Tunnel Mode: весь оригінальний IP-пакет (заголовок + payload) інкапсулюється у новий зовнішній пакет. Структура: новий зовнішній IP заголовок + ESP заголовок + зашифровано (оригінальний IP заголовок + payload) + ESP трейлер + ICV. Внутрішні IP-адреси та будь-яка інформація вище рівня IP повністю приховані. Використовується для Site-to-Site VPN між шлюзами та для Remote Access VPN.

Порівняльна таблиця Transport Mode vs Tunnel Mode:
| Параметр | Transport Mode | Tunnel Mode |
|---|---|---|
| IP-заголовок | Оригінальний | Новий зовнішній |
| Внутрішні IP | Видно | Приховані |
| Overhead | Менший (~30-40 байт) | Більший (~50-70 байт) |
| Використання | Host-to-Host | Site-to-Site, Remote Access |
| NAT сумісність (ESP) | Так (з NAT-T) | Так (з NAT-T) |

### 6. IKEv1 — Internet Key Exchange версія 1

IKEv1 (RFC 2409) — протокол автоматичного встановлення SA та обміну ключами. Двофазна модель.

Фаза 1 IKEv1 встановлює захищений IKE SA (ISAKMP SA), яка використовується для захисту подальшого обміну Phase 2. Є два варіанти:

Main Mode (6 повідомлень):
- Повідомлення 1-2 (SA Propose/Accept): ініціатор пропонує набір алгоритмів (шифрування, хешування, автентифікація, DH group); responder обирає один.
- Повідомлення 3-4 (KE + Nonce): обидві сторони обмінюються Diffie-Hellman public values та nonce; обчислюється SKEYID.
- Повідомлення 5-6 (ID + AUTH, зашифровано): ідентифікація та автентифікація сторін всередині зашифрованого каналу. PSK або сертифікати.
- Перевага: identity захищена шифруванням. Повна конфіденційність ідентичності.

Aggressive Mode (3 повідомлення):
- Повідомлення 1: ініціатор надсилає SA proposals + KE + Nonce + ID — все відкрито
- Повідомлення 2: responder надсилає вибір алгоритмів + KE + Nonce + ID + AUTH — identity та AUTH відкриті
- Повідомлення 3: ініціатор підтверджує AUTH
- Недолік: identity передається відкрито, AUTH (HMAC з PSK) передається відкрито → вразливий до offline словникових атак на PSK. Використовується коли ініціатор має динамічну IP-адресу (responder не знає заздалегідь ким є ініціатор, тому потребує identity у першому пакеті).

Фаза 2 IKEv1 — Quick Mode (3 повідомлення, всередині Phase 1 SA):
- Узгоджує IPsec SA (шифрування, автентифікація, SPI, термін дії, Traffic Selectors)
- Опційний DH-обмін для PFS (Perfect Forward Secrecy)
- 3 повідомлення: Propose → Accept + Nonce (+ KE) → Confirm

### 7. IKEv2 — Internet Key Exchange версія 2

IKEv2 (RFC 7296) — значно спрощена версія IKEv1. Базове з'єднання потребує лише 4 повідомлення замість мінімум 9 у IKEv1. Підтримує MOBIKE (RFC 4555) для мобільності.

IKE_SA_INIT exchange (2 повідомлення, відкрито):
- Ініціатор надсилає: SA (список алгоритмів), KE (DH public value), Ni (nonce ініціатора), NAT_DETECTION_SOURCE_IP, NAT_DETECTION_DESTINATION_IP
- Responder відповідає: SA (вибраний алгоритм), KE (DH public value), Nr (nonce responder'а), CERTREQ (необов'язково)
- Обидві сторони незалежно обчислюють: SKEYSEED = prf(Ni | Nr, g^ir), з якого деривуються 7 ключів через PRF+ функцію: SK_d (для дитячих SA), SK_ai, SK_ar (для IKE SA integrity), SK_ei, SK_er (для IKE SA encryption), SK_pi, SK_pr (для AUTH)

IKE_AUTH exchange (2 повідомлення, зашифровано SK_ei/SK_er):
- Ініціатор надсилає: IDi (ідентифікатор), CERT (сертифікат, необов'язково), AUTH (підпис або MAC для аутентифікації), SAi2 (пропозиція IPsec SA), TSi, TSr (Traffic Selectors)
- Responder відповідає: IDr, CERT (необов'язково), AUTH, SAr2 (вибрана IPsec SA), TSi, TSr
- Результат: IKE SA встановлена, перша Child SA (IPsec SA) встановлена

Додаткові обміни IKEv2:
- CREATE_CHILD_SA: встановлення нових або оновлення існуючих IPsec SA з PFS (новий DH-обмін)
- INFORMATIONAL: повідомлення DELETE (видалення SA), NOTIFY (інформація про помилки), конфігурації MOBIKE

### 8. NAT-Traversal (NAT-T)

NAT-Traversal (RFC 3948) вирішує проблему несумісності IPsec з NAT. При виявленні NAT між вузлами, IKEv1/IKEv2 автоматично переходять на режим UDP encapsulation.

Виявлення NAT: у IKE_SA_INIT обидві сторони включають NAT_DETECTION payloads — HMAC-хеш власної IP-адреси та порту. Якщо отриманий хеш не збігається з обчисленим з src IP пакета — NAT виявлено.

При виявленні NAT: IKE мігрує на порт UDP 4500 (замість 500). ESP пакети обгортаються у UDP: UDP заголовок (src/dst порт 4500) + Non-ESP Marker (4 нульових байти) + ESP пакет. NAT gateway може відстежувати стан UDP 4500 і коректно виконувати трансляцію. Keepalive UDP пакети (1 байт 0xFF) надсилаються кожні 20-30 секунд для збереження NAT mapping.

### 9. Криптографічні алгоритми IPsec

Сучасний IPsec підтримує широкий набір криптографічних алгоритмів. Рекомендовані алгоритми за RFC 8221 та NIST:

Шифрування ESP:
- AES-256-GCM (AEAD): рекомендований для TLS 1.3 та IPsec. Забезпечує шифрування та автентифікацію одночасно. Підтримує AES-NI.
- AES-128-GCM (AEAD): прийнятний для більшості застосувань.
- ChaCha20-Poly1305 (AEAD): рекомендований для пристроїв без AES-NI. Стандарт у WireGuard та TLS 1.3.
- AES-256-CBC + HMAC-SHA256: legacy режим, не рекомендований для нових розгортань.

DH групи для обміну ключами:
- Group 19 (P-256, secp256r1): 256-бітна еліптична крива NIST, мінімальний рекомендований
- Group 20 (P-384): 384-бітна, вища безпека
- Group 21 (P-521): 521-бітна, максимальна безпека серед NIST кривих
- Group 31 (Curve25519): Edwards крива, розроблена Bernstein, широко використовується у WireGuard та новому IPsec
- Group 14 (MODP-2048): класичний DH, прийнятний але менш ефективний ніж EC

Аутентифікація IKE:
- RSA-2048 або ECDSA P-256 підпис сертифіката
- PSK з HMAC-SHA256 (лише для малих, статичних конфігурацій)
- EAP методи: EAP-TLS (сертифікати), EAP-MSCHAPv2 (пароль у захищеному EAP-PEAP тунелі)

### 10. IKEv1 vs IKEv2 — порівняння

| Параметр | IKEv1 | IKEv2 |
|---|---|---|
| Повідомлень (базова SA) | 9+ (6 Phase1 + 3 Phase2) | 4 (IKE_SA_INIT + IKE_AUTH) |
| MOBIKE підтримка | Ні | Так (RFC 4555) |
| EAP підтримка | Обмежена (XAUTH) | Нативна |
| NAT-T | Extension (RFC 3947) | Вбудований |
| PFS | Опційний (Phase 2 DH) | Вбудований (CREATE_CHILD_SA) |
| Dead Peer Detection | Extension (RFC 3706) | Вбудований |
| Стандартизація | RFC 2409 (1998) | RFC 7296 (2014) |

### 11. Perfect Forward Secrecy у IPsec

PFS (Perfect Forward Secrecy) — властивість, при якій компрометація довгострокового ключа (PSK або приватного ключа сертифіката) не дозволяє розшифрувати записаний раніше трафік.

В IKEv1 PFS досягається через опційний DH-обмін у Phase 2 (Quick Mode). Якщо PFS не увімкнено, Child SA деривуються зі SKEYID встановленого у Phase 1 — що означає, що вони пов'язані з довгостроковим матеріалом.

В IKEv2 PFS досягається через новий KE payload у CREATE_CHILD_SA exchange. Кожне оновлення Child SA (rekeying) генерує новий ephemeral DH ключ. Рекомендується завжди увімкнувати PFS для IPsec.

Вибір DH групи для PFS: переважно використовувати ті самі групи, що й для IKE SA. Мінімальний рекомендований рівень: Group 14 (MODP-2048) або Group 19 (P-256).

### 12. Сценарії застосування IPsec

Site-to-Site VPN: два шлюзи автоматично встановлюють IPsec Tunnel Mode SA. Весь трафік між корпоративними мережами шифрується. Управляється через IKEv2 з сертифікатами. Типові продукти: Cisco ASA, Fortinet, strongSwan, libreswan.

Remote Access VPN: IPsec IKEv2 з EAP-автентифікацією для окремих користувачів. Клієнт отримує внутрішній IP та DNS через IKEv2 Configuration Payload (CP). Підтримується нативно у Windows (IKEv2), macOS, iOS, Android.

Host-to-Host шифрування: IPsec Transport Mode між конкретними серверами. Прозора для застосунків шифрування трафіку бази даних або мікросервісів. Конфігурується через Linux xfrm або Windows IPsec Policies.

Захист BGP сесій: важливі BGP сесії між AS можуть бути захищені IPsec Transport Mode для запобігання BGP hijacking атакам.

### 13. Типові помилки налаштування IPsec та їх усунення

Налаштування IPsec вимагає узгодженості конфігурації на обох кінцях тунелю. Найпоширеніші помилки:

Невідповідність Phase 1 параметрів: обидві сторони повинні пропонувати та погоджувати однакові параметри шифрування, хешування, DH групи. Якщо один бік пропонує AES-256-GCM а інший — лише AES-128-CBC, IKE не зможе встановити SA. Симптом: IPsec тунель не піднімається, у логах «no proposal chosen» або «no matching proposal».

Невідповідність Traffic Selectors: Traffic Selectors (TSi, TSr в IKEv2 або Proxy ID у IKEv1) визначають, які IP-адреси та порти захищаються даною SA. Якщо один бік вказує 10.0.0.0/8, а інший — 10.0.1.0/24, переговори можуть провалитися або встановитися неправильно. Симптом: частина трафіку не тунелюється.

Проблема «Aggressive Mode PSK»: використання Aggressive Mode з PSK автентифікацією є серйозною вразливістю. Зловмисник може перехопити першу відповідь responder'а та провести офлайн атаку на PSK. Рекомендація: використовувати Main Mode або IKEv2 з сильними PSK або сертифікатами.

MTU та фрагментація: ESP overhead збільшує розмір пакета, що може перевищити MTU. Симптом: малі пакети (ping) проходять, але HTTP/HTTPS не працює. Рішення: зменшити MTU тунельного інтерфейсу або налаштувати TCP MSS Clamping.

Термін дії SA: IPsec SA мають налаштований термін дії (lifetime — за часом або обсягом даних). Після закінчення терміну SA реконструюється через rekeying. Якщо rekeying не вдається (наприклад, через firewall), трафік зупиняється. Симптом: тунель регулярно «падає» раз на X хвилин. Рішення: перевірити firewall, збільшити lifetime або налагодити rekeying.

### 14. Безпека IPsec — відомі вразливості

IPsec є зрілим та добре дослідженим стандартом. Більшість практичних атак пов'язані з неправильною конфігурацією, а не фундаментальними вразливостями протоколу.

IKEv1 Aggressive Mode + PSK: як зазначено вище, це серйозна вразливість. Hashcat та oclHashcat можуть атакувати PSK хеш на GPU зі швидкістю мільярдів спроб за секунду. Багато корпоративних VPN досі використовують цю комбінацію.

SWEET32 (CVE-2016-2183): атака на блокові шифри з 64-бітним блоком (3DES). При достатньо великих обсягах трафіку через один SA з 3DES можлива атака на конфіденційність через birthday paradox. Рішення: перейти на AES-GCM.

Слабкі DH групи: DH Group 1 (MODP-768), Group 2 (MODP-1024), Group 5 (MODP-1536) вважаються зламаними або слабкими. NSA PRISM і Logjam атака показали, що 1024-бітний MODP може бути зламаний державними акторами. Рішення: використовувати мінімум MODP-2048 або ECDH P-256.

NULL шифрування ESP: ESP підтримує режим без шифрування (NULL encryption) для автентифікації без конфіденційності. Це небезпечно у відкритих мережах і повинно використовуватись лише в закритих сегментах де потрібна лише цілісність.

Replay атаки: IPsec захищається від replay через Sequence Number та Replay Window. Якщо Replay Window занадто мала або Sequence Number не перевіряється, можлива повторна ін'єкція пакетів. Усі сучасні реалізації перевіряють replay window за замовчуванням.

### 15. Порівняння IPsec з альтернативами

IPsec займає особливе місце серед VPN-технологій завдяки нативній підтримці у всіх основних ОС та апаратних рішеннях. Порівняємо його з основними альтернативами.

IPsec vs OpenVPN: IPsec нативно підтримується в ОС (без окремого клієнта), швидший (kernel-space обробка), стандартизований та інтероперабельний між різними виробниками. OpenVPN більш гнучкий, краще проходить через firewall (TCP 443), легше налагоджувати завдяки відкритому логуванню.

IPsec vs WireGuard: WireGuard значно простіший у налаштуванні, має менший код (4000 vs 400000 рядків), використовує сучасніший криптографічний стек (Curve25519, ChaCha20-Poly1305). IPsec більш зрілий, інтероперабельний, підтримує більше сценаріїв та алгоритмів, має широку апаратну підтримку.

IPsec vs SSL VPN (OpenVPN, SSTP): SSL VPN краще проходить через обмежуючі firewall та проксі (TCP 443), не потребує ядерної підтримки. IPsec більш ефективний, нативно підтримується без окремого ПО клієнта.
