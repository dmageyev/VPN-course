## Слайд 3

## Що таке IPsec?

IPsec (Internet Protocol Security) — це набір відкритих стандартів для захисту IP-трафіку на мережевому рівні (L3) моделі OSI. Забезпечує конфіденційність, цілісність та автентифікацію без зміни застосунків.

### Три гарантії IPsec

- Конфіденційність — шифрування даних (ESP)

- Цілісність — захист від підміни (AH/ESP)

- Автентифікація джерела — підтвердження відправника

### Ключові стандарти (RFC)

- RFC 4301 — Архітектура IPsec

- RFC 4302 — Authentication Header (AH)

- RFC 4303 — ESP (Encapsulating Security Payload)

- RFC 7296 — Internet Key Exchange v2 (IKEv2)

- RFC 2409 — Internet Key Exchange v1 (IKEv1)

## Слайд 5

## Security Association (SA)

SA (Security Association) — однонаправлений «логічний канал» між двома вузлами, що визначає всі параметри захисту для певного потоку трафіку.

### Параметри SA (SAD-запис)

- SPI — Security Parameter Index (32 біти)

- IP-адреса призначення

- Протокол: AH або ESP

- Алгоритм шифрування + ключ

- Алгоритм автентифікації + ключ

- Lifetime (за часом або обсягом)

- Режим: Transport або Tunnel

### Двонаправленість

SA — однонаправлена. Для двостороннього зв'язку потрібні дві SA: одна від A→B, інша від B→A.

              Хост A
              → SA₁ (SPI=1001) →
              Хост B

              Хост A
              ← SA₂ (SPI=2002) ←
              Хост B

## Слайд 6

## Бази даних SPD та SAD

### SPD — Security Policy Database

Визначає правила: що робити з кожним пакетом. Для кожного потоку трафіку (за 5-tuple: src/dst IP, протокол, порти) задається одна з дій:

- PROTECT — застосувати IPsec (AH або ESP)

- BYPASS — пропустити без захисту

- DISCARD — відкинути пакет

### SAD — Security Association Database

Містить активні SA з усіма параметрами захисту. Кожен вхідний пакет ідентифікується за трійкою:

{ IP_dst, SPI, Протокол }

IKE автоматично заповнює SAD після узгодження параметрів.

## Слайд 7

## Протокол AH (Authentication Header)

AH — протокол аутентифікації та цілісності. Захищає незмінні поля IP-заголовка, весь IP-пакет цілісно, але не шифрує дані.

### Що захищає AH

- Автентифікація джерела пакета

- Цілісність IP-заголовка і даних

- Захист від replay-атак (Sequence Number)

- IP-протокол номер: 51

### Обмеження AH

- Не забезпечує конфіденційність (немає шифрування)

- Несумісний із NAT (змінює IP-заголовок, що ламає ICV)

- Рідко використовується окремо від ESP

Алгоритми HMAC: HMAC-MD5-96, HMAC-SHA1-96, HMAC-SHA256-128, AES-XCBC-MAC-96

## Формат заголовка AH

      Один рядок = 32 біти (4 байти)

        Next Header8 біт
        Payload Len8 біт
        Reserved16 біт

        Security Parameters Index (SPI)32 біти

        Sequence Number32 біти

        Integrity Check Value (ICV / HMAC)96 або 128 біт (змінна довжина, кратна 32 бітам)

### Next Header

Ідентифікує протокол наступного заголовка (TCP=6, UDP=17, ESP=50)

### SPI

Вказує SA для обробки пакета на стороні одержувача

### Sequence Number

Монотонно зростаючий лічильник — захист від replay-атак

## Слайд 9

## Протокол ESP (Encapsulating Security Payload)

ESP — основний протокол IPsec для захисту даних. Забезпечує шифрування, аутентифікацію, захист від replay. Протокол IP номер: 50.

### Можливості ESP

- Конфіденційність — шифрування Payload

- Автентифікація — HMAC над ESP-пакетом

- Цілісність — Integrity Check Value (ICV)

- Захист від replay-атак

- Необов'язковість шифрування (NULL-шифр)

### Алгоритми шифрування ESP

- AES-CBC (128/256 біт) — найпоширеніший

- AES-GCM — AEAD (шифрування + автентифікація)

- AES-CTR — потоковий режим

- ChaCha20-Poly1305 — сучасна альтернатива

- 3DES-CBC — застарілий, уникати

ESP аутентифікує від SPI до кінця Trailer (включно з Padding), але не зовнішній IP-заголовок у Tunnel Mode.

## Слайд 11

## Transport Mode (Транспортний режим)

У транспортному режимі оригінальний IP-заголовок зберігається. Захищається лише корисне навантаження (TCP/UDP + дані). Використовується для захисту зв'язку між двома кінцевими хостами.

      До застосування IPsec (звичайний IP-пакет):

        Orig IP Hdrsrc→dst
        TCP/UDP
        Дані (Payload)

      ESP Transport Mode:

        Orig IP Hdrнезмінений
        ESP HdrSPI+Seq
        TCP/UDP🔒 зашифр.
        Дані🔒 зашифр.
        ESP Trlr🔒
        ICV

      AH Transport Mode:

        Orig IP Hdrз мітками
        AH HdrICV/HMAC
        TCP/UDP
        Дані (відкриті)

Обмеження: Transport Mode не приховує реальні IP-адреси відправника й одержувача — вони видно в оригінальному IP-заголовку.

## Слайд 12

## Tunnel Mode (Тунельний режим)

У тунельному режимі весь оригінальний IP-пакет інкапсулюється в новий пакет із зовнішнім IP-заголовком. IP-адреси шлюзів приховують топологію внутрішньої мережі. Найпоширеніший у VPN-шлюзах.

      ESP Tunnel Mode:

        New IP HdrGW→GW
        ESP HdrSPI+Seq
        Orig IP Hdr🔒 зашифр.
        TCP/UDP🔒
        Дані🔒 зашифр.
        ESP Trlr🔒
        ICV

      AH Tunnel Mode:

        New IP HdrGW→GW
        AH HdrICV/HMAC
        Orig IP Hdrвідкритий
        TCP/UDP
        Дані (відкриті)

Перевага: Tunnel Mode приховує оригінальні IP-адреси. Саме цей режим використовується у Site-to-Site VPN (IPsec між двома шлюзами).

## Слайд 14

## AH vs ESP: Порівняння протоколів

        | Можливість | AH (протокол 51) | ESP (протокол 50) | AH + ESP |

        | Конфіденційність (шифрування) | ✗ Ні | ✓ Так | ✓ Так |
        | Автентифікація даних | ✓ Так | ✓ Так | ✓ Так |
        | Захист IP-заголовка | ✓ Повний | Частковий | ✓ Повний |
        | Захист від replay | ✓ Так | ✓ Так | ✓ Так |
        | Сумісність з NAT | ✗ Ні | ✓ (NAT-T) | ✗ Ні |
        | Практичне використання | Рідко | Найпоширеніший | Специфічні випадки |

На практиці ESP з автентифікацією (ESP+HMAC або ESP-GCM) замінює комбінацію AH+ESP, оскільки надає всі необхідні гарантії безпеки.

## Слайд 15

## IKE — Internet Key Exchange

IKE — протокол автоматичного встановлення і керування Security Associations. Виконується поверх UDP порт 500 (або 4500 для NAT-T). Існують дві версії: IKEv1 та IKEv2.

### IKEv1 (RFC 2409)

- Розроблений у 1998 р.

- Дві фази: Фаза 1 (IKE SA) та Фаза 2 (IPsec SA)

- Фаза 1: Main Mode (6 пов.) або Aggressive Mode (3 пов.)

- Фаза 2: Quick Mode (3 повідомлення)

- Складний, надлишковий

### IKEv2 (RFC 7296)

- Розроблений у 2005 р., оновлений у 2014 р.

- Спрощений: лише 4 повідомлення для базового з'єднання

- Вбудована підтримка EAP, MOBIKE, NAT-T

- Стандартний для сучасних реалізацій

- Надійніший та ефективніший

---

### Методи автентифікації (Фаза 1)

## Слайд 19

## IKEv1 Фаза 2 — Quick Mode (3 повідомлення)

Quick Mode виконується всередині захищеного IKE SA, встановленого у Фазі 1. Його мета — узгодити параметри IPsec SA (AH або ESP) для захисту реального трафіку.

        Ініціатор

          1→

Hash + SA + Nonce [+ KE] [+ ID] — пропозиція IPsec SA, нові nonce для PFS🔒 зашифровано ключами IKE SA

          ←2

Hash + SA + Nonce [+ KE] [+ ID] — відповідь з вибраними параметрами🔒 зашифровано

          3→

Hash — підтвердження (commit confirmation)🔒 зашифровано · IPsec SA встановлено!

        Відповідач

PFS (Perfect Forward Secrecy): опційний обмін DH у Quick Mode забезпечує незалежність ключів сесії від довгострокового ключа.

## Слайд 20

## IKEv2 — Сучасна заміна IKEv1

IKEv2 (RFC 7296) значно спрощує процес встановлення з'єднання та усуває недоліки IKEv1. Базове з'єднання потребує лише 4 повідомлення (2 обміни).

### Переваги IKEv2

- Менше повідомлень (4 vs 9+ у IKEv1)

- Вбудована підтримка EAP (мобільні клієнти)

- NAT-T інтегровано в стандарт

- MOBIKE — переміщення між мережами

- Надійніша повторна автентифікація

- Чітке розмежування IKE SA та Child SA

### Обміни IKEv2

- IKE_SA_INIT — DH-обмін та параметри IKE SA

- IKE_AUTH — автентифікація та перша Child SA

- CREATE_CHILD_SA — нові або оновлені Child SA

- INFORMATIONAL — DELETE, NOTIFY, рекеінг

## Слайд 22

## IKEv2 — CREATE_CHILD_SA та рекеінг

### CREATE_CHILD_SA

Після встановлення IKE SA та першої Child SA через IKE_AUTH, нові IPsec SA (Child SA) або поновлення виконуються обміном CREATE_CHILD_SA:

→ HDR, SK{ SA, Ni [,KEi] [,TSi,TSr] }
← HDR, SK{ SA, Nr [,KEr] [,TSi,TSr] }
          Два повідомлення. З опційним DH (KEi/KEr) забезпечується PFS.

### Рекеінг (Rekeying)

Автоматичне оновлення ключів по закінченні lifetime SA (за часом або обсягом):

- IKE SA рекеінг — новий IKE_SA_INIT

- Child SA рекеінг — CREATE_CHILD_SA

- Новий SA встановлюється до видалення старого

### INFORMATIONAL обмін

Видалення SA (DELETE), повідомлення про помилки (NOTIFY), оновлення адрес (MOBIKE).

## Слайд 24

## NAT-Traversal (NAT-T)

AH несумісний з NAT, оскільки NAT змінює IP-заголовок (SRC IP), що ламає ICV. ESP без NAT-T теж проблемний: NAT не вміє обробляти ESP-пакети (відсутній порт).

### Проблема: ESP + NAT

            New IP
NAT-addr
            ESP Hdr
            🔒 Дані
            ICV

NAT-пристрій не може відстежувати стан ESP-пакетів (немає порту). Неможливе перенаправлення між кількома клієнтами за одним NAT-IP.

### Рішення: NAT-T (RFC 3948)

            New IP
NAT-addr
            UDP
4500
            ESP Hdr
            🔒 Дані
            ICV

ESP-пакет обгортається в UDP (порт 4500). NAT коректно обробляє UDP. IKE автоматично виявляє наявність NAT під час IKE_SA_INIT.

## Слайд 25

## Криптографічні алгоритми IPsec

        | Категорія | Рекомендовані (Suite B) | Застарілі (уникати) |

        | Шифрування | AES-GCM-128/256, AES-CBC-256, ChaCha20-Poly1305 | 3DES, DES, RC4, AES-CBC-128 (deprecated) |
        | Автентифікація / HMAC | HMAC-SHA2-256, HMAC-SHA2-384, AES-XCBC-96 | HMAC-MD5, HMAC-SHA1 |
        | Обмін ключами (DH) | ECDH P-256 / P-384 (Group 19/20), DH Group 14 (2048 біт) | DH Group 1 (768 біт), Group 2 (1024 біт) |
        | Автентифікація IKE | RSA-2048+, ECDSA P-256/384, EdDSA | PSK з коротким паролем |
        | PRF (для ключів) | PRF-HMAC-SHA2-256, PRF-AES-128-XCBC | PRF-HMAC-MD5, PRF-HMAC-SHA1 |

AEAD-алгоритми (AES-GCM, ChaCha20-Poly1305) надають одночасно шифрування та автентифікацію — окремий HMAC не потрібен.

## Слайд 26

## Сценарії застосування IPsec

### Site-to-Site VPN

IPsec між двома VPN-шлюзами (Tunnel Mode, ESP). З'єднує корпоративні офіси через Інтернет. Трафік між офісами автоматично захищений. Типові рішення: Cisco ASA, FortiGate, pfSense/strongSwan.

### Remote Access VPN

IKEv2+EAP або IKEv2+CERT для мобільних користувачів. Клієнти: Windows (вбудований), iOS/macOS (IKEv2), Android (strongSwan).

### Host-to-Host

Transport Mode ESP між двома серверами в дата-центрі. Захист трафіку мікросервісів без зміни застосунків. Використовується в Kubernetes (мережеві плагіни).

### IPv6 + IPsec

У IPv6 IPsec є обов'язковою частиною стека (RFC 6434). Всі IPv6-пристрої мають підтримувати IPsec, хоча його використання опціональне.

## Слайд 28

## Контрольні питання

- Які три властивості безпеки забезпечує IPsec? Який протокол (AH чи ESP) надає кожну з них?

- Що таке Security Association? З яких параметрів складається запис у SAD?

- Чим відрізняється SPD від SAD? Які три дії може задати SPD для трафіку?

- Намалюйте формат заголовка AH і поясніть призначення кожного поля.

- Чому AH несумісний з NAT? Як ESP вирішує цю проблему через NAT-T?

- Порівняйте Transport Mode та Tunnel Mode: що захищається, коли застосовується кожен?

- Опишіть 6 повідомлень IKEv1 Main Mode. На якому кроці встановлюється шифрування?

- Чому Aggressive Mode вразливий? Яка конкретна загроза виникає при використанні PSK?

- Скільки повідомлень потрібно IKEv2 для встановлення IPsec SA? Назвіть ці обміни.

- Що таке PFS (Perfect Forward Secrecy) і як він реалізується в IKEv1 та IKEv2?
