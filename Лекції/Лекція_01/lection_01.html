Тріада CIA — Фундамент інформаційної безпеки

Фундаментальною моделлю інформаційної безпеки є тріада CIA — три взаємодоповнюючих принципи захисту будь-якої системи.

🔒 C — Confidentiality

Конфіденційність — захист інформації від несанкціонованого читання або розкриття стороннім особам.

🛡️ I — Integrity

Цілісність — гарантія того, що дані не були змінені несанкціоновано в процесі передачі або зберігання.

⚡ A — Availability

Доступність — гарантія того, що авторизовані користувачі мають доступ до ресурсів у потрібний момент.

VPN реалізує тріаду через:

  • Шифрування → Confidentiality
  • HMAC / Poly1305 → Integrity
  • Захищений канал → Availability

C — Конфіденційність (Confidentiality)

Загрози конфіденційності

  • Перехоплення трафіку (Man-in-the-Middle)
  • Прослуховування публічних Wi-Fi мереж
  • Аналіз трафіку провайдером (ISP)
  • Несанкціонований доступ до сховищ

Засоби захисту

  • Симетричне шифрування: AES-256, ChaCha20
  • Асиметричний обмін ключами: RSA, ECDH
  • Захищені канали: TLS, IPsec, WireGuard

Без VPN vs З VPN

❌ Без VPN: ISP та зловмисник бачать увесь відкритий трафік у публічній мережі.

✅ З VPN: трафік зашифровано — зловмисник бачить лише зашифровані байти без можливості читання.

I — Цілісність та A — Доступність

🛡️ Integrity (Цілісність)

Гарантія відсутності несанкціонованих змін у даних.

  • Хеш-функції: SHA-256, SHA-3
  • HMAC (прив'язка до секретного ключа)
  • Цифрові підписи (RSA, ECDSA)
  • У VPN: MAC на кожен пакет

⚡ Availability (Доступність)

Гарантія доступу до ресурсів для авторизованих користувачів.

  • Захист від DDoS-атак
  • Резервування каналів зв'язку
  • Балансування навантаження
  • У VPN: обхід блокувань та цензури

Криптографічні хеш-функції

Хеш-функція перетворює дані довільного розміру у фіксований дайджест фіксованої довжини. У VPN використовується для контролю цілісності пакетів.

Властивості хеш-функції

  • Детермінованість: однакові дані → однаковий хеш
  • Лавинний ефект: 1 біт різниці → кардинально інший хеш
  • Незворотність: неможливо відновити вхід із хешу
  • Стійкість до колізій: не знайти два входи з однаковим хешем

HMAC — Hash-based Message Authentication Code

Цілісність у VPN забезпечується HMAC — поєднанням хеш-функції із секретним ключем. Це підтверджує не лише цілісність, а й автентичність джерела пакета.

Формула HMAC

HMAC(K, M) = H(
  (K ⊕ opad) ||
  H( (K ⊕ ipad) || M )
)

K  — секретний ключ
M  — повідомлення
H  — хеш-функція (SHA-256)

HMAC vs простий хеш

  • Простий хеш: підтверджує цілісність
  • HMAC: підтверджує і цілісність, і автентичність
  • Захист від атак підміни пакетів
  • Потребує знання секретного ключа

Використання у VPN

  • IPsec: HMAC-SHA256, HMAC-SHA512
  • OpenVPN: HMAC-SHA256 (tls-auth)
  • WireGuard: Poly1305 (AEAD)

Симетрична криптографія

Симетричне шифрування використовує один і той самий ключ для шифрування та дешифрування. Конфіденційність у VPN базується саме на ньому.

Переваги

  • Надвисока швидкість (AES-NI апаратне прискорення)
  • Ефективне для великих обсягів даних
  • Малий розмір ключа: 128/256 біт

Виклик: обмін ключем

Головна проблема — Key Distribution Problem: як безпечно передати ключ через публічну мережу до початку шифрування?

У VPN вирішується через асиметричний handshake (Diffie-Hellman), після якого обидві сторони мають спільний симетричний ключ.

AES — Advanced Encryption Standard

Параметри AES

  • Розмір блоку: 128 біт (16 байт) — фіксовано
  • Розмір ключа: 128, 192 або 256 біт
  • Раундів: 10 (AES-128) / 14 (AES-256)
  • Прийнятий NIST у 2001 р., стандарт FIPS 197

Операції одного раунду

  • SubBytes — підстановка через S-box
  • ShiftRows — зсув рядків матриці
  • MixColumns — лінійне перемішування
  • AddRoundKey — XOR з ключем раунду

Режими роботи у VPN

Рекомендований вибір: AES-256-GCM — забезпечує шифрування та автентифікацію одночасно (AEAD).

ChaCha20-Poly1305

ChaCha20 — потоковий шифр Деніела Бернштейна (djb). Ключова альтернатива AES для пристроїв без апаратного прискорення (мобільні, IoT).

ChaCha20-Poly1305 = AEAD

ChaCha20 → шифрування потоку (256-біт ключ)
Poly1305 → автентифікація MAC (128 біт)
Разом → AEAD (Confidentiality + Integrity)

ARX-операції (Add, Rotate, XOR) — константний час виконання → стійкість до timing-атак.

Режим GCM та принцип AEAD

AEAD (Authenticated Encryption with Associated Data) — шифрування, що одночасно забезпечує конфіденційність та цілісність одним проходом.

Схема GCM (AES-GCM)

Вхід: Текст (P) + Nonce + AAD
    ↓ AES-CTR (шифрування)
Шифртекст (C)
    ↓ GHASH (автентифікація)
Auth Tag (128 біт)

Вихід: C || Tag

Associated Data (AAD)

Заголовки пакетів, які автентифікуються, але не шифруються. Маршрутизатори можуть читати IP-заголовки, але не можуть їх підмінити.

Переваги AEAD

  • Один прохід замість двох → вища швидкість
  • Захист від атак з відтворенням (Replay)
  • Детектує будь-яку модифікацію пакета

VPN — Virtual Private Network

VPN — технологія створення захищеного логічного з'єднання (тунелю) поверх публічної мережі. Використовує інкапсуляцію — вкладення одного пакета в інший із додаванням нових заголовків і шифрування.

Ключові концепції

  • Underlay — реальна публічна мережа (Інтернет)
  • Overlay — захищений логічний канал поверх Underlay
  • Інкапсуляція — вкладення оригінального пакета в зовнішній
  • Тунель — захищений шлях від клієнта до сервера

Функції VPN

  • Захист трафіку у публічних мережах
  • Анонімність (приховування IP-адреси)
  • Об'єднання розподілених мереж
  • Обхід географічних обмежень та цензури

Site-to-Site VPN

Site-to-Site VPN — постійний тунель між двома мережами (офісами, дата-центрами). Тунель встановлюється між VPN-шлюзами, а не окремими хостами.

Характеристики

  • Хости використовують тунель автоматично (без VPN-клієнта)
  • Тунель зазвичай постійний (always-on)
  • Прозорість для кінцевих пристроїв

Типові сценарії

  • Об'єднання корпоративних філій
  • Підключення офісу до хмари (AWS VPC)
  • Резервування між дата-центрами

Модель OSI — 7 рівнів

Рівні OSI

7 │ Прикладний    │ HTTP, FTP, DNS
6 │ Представлення │ TLS/SSL (шифрування)
5 │ Сеансовий     │ Управління сеансами
4 │ Транспортний  │ TCP, UDP
3 │ Мережевий     │ IP, ICMP (маршрутизація)
2 │ Канальний     │ Ethernet (MAC-адреси)
1 │ Фізичний      │ Біти, кабелі, сигнали

Де діє VPN-шифрування

  • L7 — TLS/SSL VPN (SSTP, OpenVPN)
  • L3 — IPsec, WireGuard (Мережевий рівень)
  • L2 — L2TP, PPTP (Канальний рівень)

Правило

Нижчий рівень тунелювання → більший overhead, але більша прозорість для застосунків (L2 VPN тунелює навіть Ethernet-трафік).

Класифікація VPN за рівнем OSI

Рівень OSI Протокол Що тунелюється Приклади

Інкапсуляція у моделі OSI

Рівні моделі OSI в контексті тунелювання: кожен рівень обгортає пакет попереднього рівня у власний заголовок.

WireGuard (L3 VPN через UDP)

Оригінальний пакет (VPN-мережа):
[ IP (10.0.0.x) │ TCP │ Дані ]

↓ Шифрування ChaCha20-Poly1305

Після інкапсуляції (реальна мережа):
[ IP (реальний) │ UDP │ WG Hdr │ Шифртекст │ MAC ]
  ↑ L3 Underlay  ↑ L4   ↑ VPN     ↑ Overlay

Overhead WireGuard

  • Зовнішній IP-заголовок: 20 байт
  • UDP-заголовок: 8 байт
  • WireGuard-заголовок: 32 байт
  • Poly1305 MAC: 16 байт
  • Загалом: ~76 байт на пакет

При MTU = 1500 байт ефективний розмір payload пакета: 1500 – 76 = 1424 байт

IPsec у моделі OSI

IPsec — найпоширеніший корпоративний VPN-стек. Працює на Рівні 3 (Мережевий), захищаючи IP-пакети.

Стек IPsec

L7 │ Прикладний    │ HTTP, SMTP…
L4 │ Транспортний  │ TCP / UDP
L3 │ Мережевий     │ ← IPsec тут (ESP/AH)
L2 │ Канальний     │ Ethernet
L1 │ Фізичний      │ Кабель / Wi-Fi

Компоненти IPsec

  • IKEv2 — обмін ключами (UDP 500/4500)
  • ESP — шифрування + автентифікація (IP 50)
  • AH — лише автентифікація (IP 51)

Два режими IPsec

  • Transport Mode: шифрує лише payload → Host-to-Host
  • Tunnel Mode: шифрує весь IP-пакет → Site-to-Site

Встановлення VPN-тунелю: IKEv2

Процес встановлення IPsec IKEv2 тунелю відбувається у двох фазах.

Клієнт              VPN-сервер
  │                       │
  │─ IKE_SA_INIT ────────►│ Фаза 1
  │  (алгоритми, DH pub)  │ (IKE SA)
  │◄─ IKE_SA_INIT ────────│
  │  (вибір, DH pub)      │
  │                       │
  │─ IKE_AUTH ───────────►│ Фаза 2
  │  (ідентифікація, cert) │ (Child SA)
  │◄─ IKE_AUTH ───────────│
  │  (підтвердження)      │
  │                       │
  │═════ Тунель ESP ══════│ Передача

Що відбувається

  • 1. Узгодження алгоритмів — DH-група, AES, SHA
  • 2. Diffie-Hellman — генерація спільного секрету
  • 3. Автентифікація — сертифікати або PSK
  • 4. SA (Security Association) — параметри тунелю
  • 5. Передача даних — зашифрований трафік ESP

Ключові концепції безпеки VPN

Perfect Forward Secrecy (PFS)

Кожна сесія використовує унікальні тимчасові DH-ключі. Компрометація поточного ключа не розкриває минулі сесії.

  • Реалізується через Ephemeral Diffie-Hellman (EDH)
  • Ключ сесії генерується на льоту та видаляється після використання

Практичні сценарії застосування VPN

🏢 Корпоративний Remote Access

Співробітники підключаються до офісних ресурсів з дому або у відрядженні. Cisco AnyConnect, GlobalProtect, WireGuard.

Тунелювання трафіку розумних пристроїв через захищений шлюз. Ізоляція IoT-підмереж.

Perfect Forward Secrecy (PFS)

PFS гарантує, що компрометація довгострокового ключа не розкриє минулих сесій.

❌ Без PFS

RSA ключ сервера скомпрометовано → зловмисник розшифровує весь записаний трафік

✅ З PFS (ECDHE)

Кожна сесія — новий ephemeral DH ключ. Минулий трафік не може бути розшифрований.

TLS 1.3 вимагає PFS обов'язково (лише ECDHE). IKEv2: PFS через CREATE_CHILD_SA.

Diffie-Hellman: обмін ключами

DH дозволяє двом сторонам узгодити спільний секрет через незахищений канал без попереднього обміну ключами.

🔢 Математика DH
  • Спільні параметри: g (генератор), p (просте число)
  • Alice: a (приватний) → g^a mod p (публічний)
  • Bob: b (приватний) → g^b mod p (публічний)
  • Shared secret: g^(ab) mod p
✅ Сучасні варіанти
  • ECDH на Curve25519 — WireGuard
  • ECDH на P-256/P-384 — TLS 1.3, IKEv2
  • Ephemeral DH → Perfect Forward Secrecy

Методи аутентифікації у VPN

⚠️ Слабкі методи
  • Pre-Shared Key (PSK) — вразливий до brute-force
  • Пароль без MFA — компрометується при фішингу
  • PPTP MS-CHAPv2 — повністю зламаний
✅ Рекомендовані методи
  • Сертифікати X.509 (PKI) + ECDSA
  • IKEv2 з EAP-TLS (клієнтський сертифікат)
  • MFA: TOTP + сертифікат
  • SAML/OIDC через IAM провайдера

Від периметра до Zero Trust

🏰 Класичний підхід (Castle-and-Moat)
  • Довіра базується на розташуванні в мережі
  • Всередині периметра = довірений
  • VPN надає доступ до всієї внутрішньої мережі
  • Lateral movement після компрометації
🔐 Zero Trust
  • «Ніколи не довіряй, завжди перевіряй»
  • Доступ до конкретного застосунку, не мережі
  • Постійна верифікація ідентичності
  • Перевірка стану пристрою (device posture)

Always-On VPN

Always-On VPN — режим, при якому VPN встановлюється автоматично при запуску пристрою та блокує мережевий доступ до підключення.

✅ Переваги
  • Жодного незахищеного мережевого трафіку
  • Прозоро для користувача
  • Централізоване управління через MDM
⚙️ Реалізація
  • Microsoft Intune + DirectAccess/AOVPN
  • Cisco AnyConnect з профілем Always-On
  • iOS/Android MDM з VPN On Demand
