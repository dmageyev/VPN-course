## Слайд 3 — Тріада CIA: Вступ

# Тріада CIA — Фундамент інформаційної безпеки

**Тріада CIA** — три основоположні принципи захисту інформації:

```
         ┌───────────────────────────────────┐
         │            БЕЗПЕКА                │
         │   ┌───────────────────────────┐   │
         │   │   C — Confidentiality     │   │
         │   │   I — Integrity           │   │
         │   │   A — Availability        │   │
         │   └───────────────────────────┘   │
         └───────────────────────────────────┘
```

VPN реалізує всі три принципи через:
- **Шифрування** → Confidentiality
- **Хешування (HMAC)** → Integrity
- **Захищений канал зв'язку** → Availability

## Слайд 4 — Конфіденційність (Confidentiality)

# C — Конфіденційність (Confidentiality)

**Визначення:** захист інформації від несанкціонованого читання або перехоплення.

**Загрози:**
- Перехоплення трафіку (Man-in-the-Middle)
- Прослуховування каналів зв'язку (Eavesdropping)
- Несанкціонований доступ до сховищ даних

**Засоби забезпечення:**
- Симетричне шифрування: AES-256, ChaCha20
- Асиметричне шифрування: RSA, ECDH (для обміну ключами)
- Шифрування каналу зв'язку: TLS, IPsec, WireGuard

> **У VPN:** весь трафік між клієнтом і сервером зашифровується, що унеможливлює читання даних провайдером або зловмисником.

## Слайд 5 — Цілісність та Доступність (Integrity & Availability)

# I — Цілісність і A — Доступність

## Integrity (Цілісність)

**Визначення:** гарантія того, що дані не були змінені несанкціонованим чином.

- Засоби: **хеш-функції** (SHA-256, SHA-3), **HMAC**, **цифрові підписи**
- У VPN: кожен пакет супроводжується MAC (Message Authentication Code)

## Availability (Доступність)

**Визначення:** гарантія доступу авторизованих користувачів до ресурсів у будь-який момент.

- Загрози: DDoS-атаки, збої обладнання, перевантаження мережі
- Засоби: резервування каналів, балансування навантаження, моніторинг
- У VPN: надійне та стійке з'єднання навіть через ненадійні мережі

## Слайд 6 — CIA у контексті VPN

# Тріада CIA у реалізації VPN

| Принцип CIA | Загроза | Механізм VPN |
|-------------|---------|--------------|
| **Confidentiality** | Перехоплення трафіку | Шифрування AES-256/ChaCha20 |
| **Integrity** | Підміна пакетів | HMAC-SHA256, Poly1305 |
| **Availability** | Блокування з'єднання | Резервні вузли, обхід DPI |

**Приклад сценарію атаки без VPN:**
```
Атакуючий (MITM)
     ↑
Клієнт ──── Wi-Fi ──── Маршрутизатор ──── Інтернет ──── Сервер
              ↑
         Прослуховування: дані читаються у відкритому вигляді
```

**З VPN:** увесь трафік зашифровано — атакуючий бачить лише зашифровані байти.

## Слайд 7 — Хеш-функції

# Криптографічні хеш-функції

**Хеш-функція** перетворює дані довільного розміру у фіксований дайджест.

**Властивості:**
- **Детермінованість:** однакові вхідні дані → завжди однаковий хеш
- **Лавинний ефект:** зміна 1 біта → кардинально інший хеш
- **Незворотність:** неможливо відновити вхідні дані з хешу
- **Стійкість до колізій:** неможливо знайти два різні входи з однаковим хешем

**Приклад (SHA-256):**
```
SHA-256("VPN")  = a7ef3c...2f14
SHA-256("vpn")  = 9b3d11...8e2a  ← зовсім інший хеш!
SHA-256("")     = e3b0c4...f854
```

| Алгоритм | Розмір дайджесту | Стан |
|----------|-----------------|------|
| MD5      | 128 біт          | Застарілий ⚠️ |
| SHA-1    | 160 біт          | Не рекомендується ⚠️ |
| SHA-256  | 256 біт          | Актуальний ✅ |
| SHA-3    | 256/512 біт      | Актуальний ✅ |

## Слайд 8 — HMAC та автентифікація пакетів

# HMAC — Hash-based Message Authentication Code

**HMAC** поєднує хеш-функцію із секретним ключем для автентифікації джерела повідомлення.

```
HMAC(K, M) = H( (K ⊕ opad) || H( (K ⊕ ipad) || M ) )
```

- **K** — секретний ключ
- **M** — повідомлення
- **H** — хеш-функція (наприклад, SHA-256)

**Переваги HMAC над простим хешем:**
- Прив'язує повідомлення до секретного ключа
- Підтверджує **автентичність джерела** (не лише цілісність)
- Захищає від атак підміни пакетів

**У VPN-протоколах:**
- **IPsec:** HMAC-SHA256, HMAC-SHA512
- **OpenVPN:** HMAC-SHA256 (tls-auth / tls-crypt)
- **WireGuard:** Poly1305 (AEAD — автентифіковане шифрування)

## Слайд 9 — Симетрична криптографія

# Симетрична криптографія

**Симетричне шифрування:** один і той самий ключ використовується для шифрування та дешифрування.

```
Відкритий текст ──[Ключ K]──► Шифртекст ──[Ключ K]──► Відкритий текст
                  Шифрування                Дешифрування
```

**Переваги:**
- Висока швидкість обробки (апаратне прискорення AES-NI)
- Ефективне для великих обсягів даних
- Невеликий розмір ключа (128/256 біт)

**Недолік:**
- Проблема безпечного обміну ключем (**Key Distribution Problem**)
- У VPN вирішується за допомогою асиметричної криптографії (handshake)

**Основні алгоритми у VPN:**
- **AES** (Advanced Encryption Standard) — галузевий стандарт
- **ChaCha20** — альтернатива для пристроїв без AES-NI

## Слайд 10 — AES: Advanced Encryption Standard

# AES — Advanced Encryption Standard

**AES** — блочний шифр, прийнятий NIST у 2001 р. як федеральний стандарт США.

**Параметри:**
- Розмір блоку: **128 біт** (16 байт) — фіксовано
- Розмір ключа: **128, 192 або 256 біт**
- Кількість раундів: 10 (AES-128), 12 (AES-192), 14 (AES-256)

**Операції одного раунду:**
1. **SubBytes** — підстановка байт через S-box
2. **ShiftRows** — зсув рядків матриці стану
3. **MixColumns** — перемішування стовпців (лінійне перетворення)
4. **AddRoundKey** — XOR зі стовпцем розкладеного ключа

**Режими роботи у VPN:**

| Режим | Особливість | Використання |
|-------|-------------|--------------|
| AES-CBC | Класичний, потребує окремий MAC | OpenVPN (старий) |
| AES-GCM | AEAD: шифрування + автентифікація | IPsec, OpenVPN 2.4+ |
| AES-CTR | Лічильниковий, паралельний | Потокові протоколи |

## Слайд 11 — ChaCha20-Poly1305

# ChaCha20-Poly1305

**ChaCha20** — потоковий шифр, розроблений Деніелом Бернштейном (djb).

**Особливості:**
- Розмір ключа: **256 біт**
- Базується на ARX-операціях (Add, Rotate, XOR) — ефективний без спеціального заліза
- Стійкий до timing-атак завдяки фіксованому часу виконання

**ChaCha20-Poly1305 = шифр + MAC:**
```
ChaCha20   → шифрування потоку даних
Poly1305   → автентифікація (MAC, 128 біт)
Разом      → AEAD (Authenticated Encryption with Associated Data)
```

**Порівняння AES-GCM vs ChaCha20-Poly1305:**

| Критерій | AES-256-GCM | ChaCha20-Poly1305 |
|----------|-------------|-------------------|
| Апаратне прискорення | AES-NI (Intel/AMD) | Не потрібне |
| Швидкість без AES-NI | Повільніший | **Швидший** |
| Швидкість з AES-NI | **Швидший** | Повільніший |
| Стійкість до timing-атак | Потрібна обережність | Константний час |
| Використання | IPsec, TLS 1.3 | WireGuard, TLS 1.3 |

## Слайд 12 — Режими шифрування GCM та AEAD

# Режими шифрування: GCM та AEAD

**GCM (Galois/Counter Mode)** — поєднує лічильниковий режим (CTR) з автентифікацією Галуа.

```
Вхід: Відкритий текст (P) + Асоційовані дані (AAD) + Nonce
         ↓ AES-CTR (шифрування)         ↓ GHASH (автентифікація)
Вихід: Шифртекст (C) + Authentication Tag (128 біт)
```

**AEAD (Authenticated Encryption with Associated Data):**
- Одночасно забезпечує **конфіденційність** і **цілісність/автентичність**
- **Associated Data (AAD)** — заголовки пакетів, які автентифікуються, але не шифруються
- Один прохід замість двох → вища продуктивність

**Практичне застосування:**

```
IPsec ESP (AES-256-GCM):
[ IP Hdr ][ ESP Hdr | Зашифрований payload | ESP Auth Tag ]
                                              ↑
                               Автентифікується через GCM
```

## Слайд 13 — Що таке VPN?

# VPN — Virtual Private Network

**VPN** — технологія створення захищеного логічного з'єднання поверх публічної мережі.

**Ключові поняття:**
- **Underlay** — реальна публічна мережа (Інтернет), що використовується для транспортування
- **Overlay** — захищений логічний канал поверх Underlay
- **Інкапсуляція** — вкладення оригінального пакета в зовнішній пакет із новими заголовками

**Схема роботи:**
```
Клієнт ──── [зашифрований тунель] ──── VPN-сервер ──── Ресурс
   │                                                       │
   │        ← Overlay (захищений VPN-канал) →             │
   └────────────── Underlay (Інтернет) ───────────────────┘
```

**Функції VPN:**
- Захист трафіку у публічних мережах (Wi-Fi, Інтернет)
- Забезпечення анонімності (приховування IP-адреси)
- Об'єднання розподілених мереж (корпоративні філіали)
- Обхід географічних обмежень та цензури

## Слайд 14 — Site-to-Site VPN

# Site-to-Site VPN

**Site-to-Site VPN** — постійний тунель між двома мережами (офісами, дата-центрами).

```
Офіс А (192.168.1.0/24)                    Офіс Б (192.168.2.0/24)
┌─────────────────────┐                    ┌─────────────────────┐
│ ПК  ─┐              │                    │              ┌─ ПК  │
│ ПК  ─┤─ VPN-шлюз A ─┼──── Інтернет ──────┼─ VPN-шлюз Б ┤─ ПК  │
│ ПК  ─┘              │    (тунель IPsec)   │              └─ ПК  │
└─────────────────────┘                    └─────────────────────┘
```

**Характеристики:**
- Тунель встановлюється між **VPN-шлюзами**, а не окремими хостами
- Хости всередині мереж використовують тунель **автоматично** (без VPN-клієнта)
- Тунель зазвичай **постійний** (always-on)

**Типові сценарії:**
- Об'єднання філій корпоративної мережі
- З'єднання офісу з хмарним дата-центром (AWS VPC, Azure VNet)
- Резервні канали між дата-центрами

**Протоколи:** IPsec/IKEv2, GRE over IPsec, WireGuard

## Слайд 15 — Remote Access VPN

# Remote Access VPN

**Remote Access VPN** — підключення окремого користувача до корпоративної мережі з будь-якого місця.

```
Ноутбук вдома          Інтернет           Корпоративна мережа
┌──────────┐           │                  ┌──────────────────────┐
│ VPN-     │──тунель───┼──────────────────┤ VPN-концентратор     │
│ клієнт   │           │                  │ (сервер доступу)     │
└──────────┘           │                  │                      │
                       │           Ресурси: файловий сервер,     │
Мобільний пристрій     │           корпоративні застосунки,      │
┌──────────┐           │           внутрішня БД                  │
│ VPN App  │──тунель───┘                  └──────────────────────┘
└──────────┘
```

**Порівняння із Site-to-Site:**

| Критерій | Site-to-Site | Remote Access |
|----------|--------------|---------------|
| Суб'єкти | Мережа ↔ Мережа | Користувач ↔ Мережа |
| Тунель | Постійний | На вимогу |
| VPN-клієнт | На шлюзі | На пристрої користувача |
| Автентифікація | Сертифікати/PSK | Логін/пароль + MFA |

## Слайд 16 — Класифікація VPN за рівнем OSI

# Класифікація VPN за рівнем моделі OSI

**VPN-рішення працюють на різних рівнях OSI:**

| Рівень OSI | Протокол | Що тунелюється | Приклад |
|------------|----------|----------------|---------|
| **Рівень 2 (L2)** | L2TP, PPTP | Ethernet-фрейми | Мостовий VPN |
| **Рівень 3 (L3)** | IPsec, WireGuard | IP-пакети | Маршрутований VPN |
| **Рівень 4 (L4)** | OpenVPN (TCP/UDP) | TCP/UDP сегменти | SSL VPN |
| **Рівень 7 (L7)** | SSTP, SSL/TLS VPN | HTTP/HTTPS | WebVPN |

**Ключова відмінність L2 vs L3 VPN:**
```
L2 VPN: [Ethernet Hdr][IP Hdr][TCP Hdr][Data]  ← тунелює з Ethernet
L3 VPN:              [IP Hdr][TCP Hdr][Data]   ← тунелює з IP
```

## Слайд 17 — Модель OSI: Огляд

# Модель OSI — 7 рівнів

```
7 │ Прикладний    │ HTTP, FTP, DNS, SSH
6 │ Представлення │ SSL/TLS (шифрування), кодування
5 │ Сеансовий     │ Управління сеансами
4 │ Транспортний  │ TCP, UDP — сегменти
3 │ Мережевий     │ IP, ICMP — пакети (маршрутизація)
2 │ Канальний     │ Ethernet, Wi-Fi — кадри (MAC-адреси)
1 │ Фізичний      │ Біти, сигнали, кабелі
```

**Де діє VPN-шифрування:**
- **TLS (OpenVPN):** рівень 6 (Представлення)
- **IPsec:** рівень 3 (Мережевий)
- **WireGuard:** рівень 3 (між Мережевим та Транспортним)
- **L2TP:** рівень 2 (Канальний)

**Правило:** чим нижчий рівень тунелювання — тим більший overhead, але більша прозорість для застосунків.

## Слайд 18 — VPN та рівні OSI: Інкапсуляція

# Інкапсуляція у моделі OSI

**Принцип інкапсуляції на прикладі WireGuard (UDP/IP):**

```
Оригінальний пакет (L3 VPN-мережі):
┌──────────────────────────────────────────────┐
│  IP Hdr (10.0.0.1→10.0.0.2) │ TCP │ Дані   │
└──────────────────────────────────────────────┘
                    ↓ Шифрування + AEAD
Після інкапсуляції WireGuard (реальна мережа):
┌───────────────────────────────────────────────────────────┐
│ IP Hdr (реальний) │ UDP Hdr │ WG Hdr │ [Зашифровано]     │
└───────────────────────────────────────────────────────────┘
                                         ↑
                         Оригінальний IP-пакет + MAC
```

**Заголовки WireGuard:**
- Зовнішній IP-заголовок: **20 байт**
- UDP-заголовок: **8 байт**
- WireGuard-заголовок: **32 байт**
- Poly1305 MAC: **16 байт**
- **Загальний overhead: ~76 байт**

## Слайд 19 — Рівні OSI та протоколи VPN

# Протоколи VPN у контексті OSI

**Стек IPsec (найпоширеніший корпоративний VPN):**

```
L7 │ Прикладний    │  HTTP, SMTP, будь-який трафік
L4 │ Транспортний  │  TCP / UDP
L3 │ Мережевий     │  ← IPsec шифрує тут (ESP/AH протоколи)
L2 │ Канальний     │  Ethernet
L1 │ Фізичний      │  Кабель / Wi-Fi
```

**Компоненти IPsec:**
- **IKE (Internet Key Exchange)** — обмін ключами (UDP 500/4500)
- **ESP (Encapsulating Security Payload)** — шифрування + автентифікація (протокол IP 50)
- **AH (Authentication Header)** — лише автентифікація (протокол IP 51)

**Два режими IPsec:**
- **Transport Mode:** шифрує лише payload, зберігає оригінальний IP-заголовок
- **Tunnel Mode:** шифрує весь IP-пакет, додає новий заголовок (Site-to-Site)

## Слайд 20 — VPN-тунель: Покроковий процес

# Як встановлюється VPN-тунель?

**Фази встановлення IPsec IKEv2:**

```
Клієнт                              VPN-сервер
  │                                     │
  │── IKE_SA_INIT (пропозиції алгоритмів, DH) ──►  │  Фаза 1
  │◄─ IKE_SA_INIT (вибір алгоритму, DH pub) ───    │  (IKE SA)
  │                                     │
  │── IKE_AUTH (ідентифікація, сертифікат) ─────►  │  Фаза 2
  │◄─ IKE_AUTH (підтвердження, CHILD SA) ──────    │  (Child SA)
  │                                     │
  │═══════════ Зашифрований тунель ═══════════════  │  Передача
```

**Що відбувається:**
1. **Узгодження алгоритмів** — DH, шифрування, хеш
2. **Обмін ключами** — Diffie-Hellman для генерації спільного секрету
3. **Автентифікація** — сертифікати або PSK
4. **Встановлення SA** — Security Association (параметри тунелю)
5. **Передача даних** — зашифрований трафік

## Слайд 21 — Порівняння VPN-протоколів

# Порівняння сучасних VPN-протоколів

| Протокол | Рівень OSI | Шифрування | Стан | Швидкість |
|----------|-----------|------------|------|-----------|
| **WireGuard** | L3 | ChaCha20-Poly1305 | ✅ Актуальний | ⚡ Найвища |
| **IPsec/IKEv2** | L3 | AES-256-GCM | ✅ Актуальний | ⚡ Висока |
| **OpenVPN** | L4 (TLS) | AES-256-GCM | ✅ Актуальний | 🔶 Середня |
| **L2TP/IPsec** | L2+L3 | AES-256 | 🔶 Застарілий | 🔶 Середня |
| **PPTP** | L2 | RC4 (40-128 біт) | ❌ Небезпечний | 🔶 Висока |
| **SSTP** | L7 (HTTPS) | TLS 1.2/1.3 | 🔶 Windows-only | 🔶 Середня |

**Рекомендований вибір:**
- **Корпоративне середовище:** IPsec/IKEv2 або WireGuard
- **Мобільні клієнти:** WireGuard або IKEv2 (MOBIKE)
- **Обхід Deep Packet Inspection:** OpenVPN (TCP 443) або SSTP

## Слайд 22 — Ключові концепції безпеки VPN

# Ключові концепції безпеки VPN

**Perfect Forward Secrecy (PFS):**
- Кожна сесія використовує унікальні ключі
- Компрометація одного ключа не розкриває минулі сесії
- Реалізується через тимчасові DH-параметри (Ephemeral DH)

**Defense in Depth — багаторівневий захист:**
```
Мережевий периметр (Firewall)
    ↓
VPN-шифрування (Confidentiality)
    ↓
Автентифікація (HMAC / Цифровий підпис)
    ↓
Авторизація (ACL, Firewall правила всередині тунелю)
    ↓
Аудит (Логування з'єднань та трафіку)
```

**Zero Trust і VPN:**
- Традиційний VPN надає повний доступ до мережі після автентифікації
- Zero Trust VPN (ZTNA) — мінімальний необхідний доступ для кожного користувача

## Слайд 23 — Практичне застосування VPN

# Сценарії практичного застосування VPN

**1. Корпоративний Remote Access:**
- Співробітники підключаються до офісних ресурсів з дому
- Протоколи: Cisco AnyConnect (IPsec/SSL), GlobalProtect, WireGuard

**2. Site-to-Site між офісами:**
- Постійний тунель між філіями
- Приклад: Kyiv HQ (192.168.1.0/24) ↔ Lviv Office (192.168.2.0/24)

**3. Захист у публічних Wi-Fi:**
- Шифрування трафіку у кафе, готелях, аеропортах
- Запобігання MITM-атакам та перехопленню сесій

**4. Обхід географічних обмежень:**
- Зміна видимої IP-адреси через VPN-сервер в іншій країні

**5. Безпека IoT-пристроїв:**
- Тунелювання трафіку розумних пристроїв через захищений шлюз

## Слайд 24 — Зведена таблиця: CIA + Криптографія + VPN

# Зведена таблиця курсу

| Компонент | Технологія | Принцип CIA | Алгоритм |
|-----------|-----------|------------|----------|
| Шифрування каналу | VPN-тунель | Confidentiality | AES-256-GCM / ChaCha20 |
| Контроль цілісності | HMAC / Poly1305 | Integrity | HMAC-SHA256 / Poly1305 |
| Обмін ключами | Diffie-Hellman | Confidentiality | ECDH P-256 / Curve25519 |
| Автентифікація | Сертифікати / PSK | Integrity | RSA / Ed25519 |
| Резервування | Multi-path VPN | Availability | — |
| Тунелювання | Інкапсуляція | All three | ESP / WireGuard |

**Ключове правило:**
> Сучасний VPN = AEAD-шифрування (Confidentiality + Integrity) + безпечний обмін ключами (DH) + стійка автентифікація (PKI/сертифікати)

## Слайд 25 — Підсумок лекції

# Підсумок лекції 01

**Вивчено:**
- **Тріада CIA** — основа будь-якої системи безпеки; VPN реалізує всі три принципи
- **Хеш-функції та HMAC** — забезпечують цілісність та автентифікацію пакетів
- **AES-256-GCM та ChaCha20-Poly1305** — сучасні AEAD-алгоритми для VPN
- **Site-to-Site VPN** — з'єднання мереж через постійний тунель між шлюзами
- **Remote Access VPN** — підключення окремих користувачів до корпоративної мережі
- **Рівні OSI** — L2/L3/L4/L7 визначають тип тунелювання та overhead

**Ключові поняття:**
`CIA` · `HMAC` · `AES-256-GCM` · `ChaCha20-Poly1305` · `Site-to-Site` · `Remote Access` · `Encapsulation` · `IPsec` · `WireGuard` · `OSI`

**Наступна лекція:**  
Лекція 2 — Асиметричне шифрування, цифрові підписи, PKI та центри сертифікації

## Слайд 26 — Perfect Forward Secrecy (PFS)

**Слайд:** Порівняння без PFS (RSA key transport) та з PFS (ECDHE). TLS 1.3 вимагає PFS обов'язково.

**Контекст:** PFS гарантує, що компрометація довгострокового ключа не розкриє минулих сесій. Ephemeral DH ключ генерується для кожної сесії та знищується після завершення.
