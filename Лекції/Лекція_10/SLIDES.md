## Слайд 3 — Симетрична vs Асиметрична криптографія

# Симетрична vs Асиметрична криптографія

```
СИМЕТРИЧНА                        АСИМЕТРИЧНА
──────────────────────────────    ──────────────────────────────
 ┌──────┐  ключ K  ┌──────┐       ┌──────┐ public ┌──────┐
 │Alice │─────────▶│ Bob  │       │Alice │───────▶│ Bob  │
 └──────┘          └──────┘       └──────┘        └──────┘
  шифрує K          дешиф. K       шифрує          дешиф.
                                   pub(Bob)         priv(Bob)

 ПРОБЛЕМА: як безпечно               РІШЕННЯ: public key
 передати ключ K?                    поширюється відкрито!
```

| Критерій | Симетрична | Асиметрична |
|----------|-----------|------------|
| Ключів | 1 | 2 (пара) |
| Швидкість | Дуже висока | Повільна |
| Розподіл ключів | Складний | Простий |
| Довжина ключа | 128–256 біт | 2048–4096 біт (RSA) |
| Застосування | Масове шифрування | Обмін ключами, підпис |

## Слайд 4 — Принцип асиметричної криптографії

# Принцип асиметричної криптографії

**Пара ключів — фундамент усієї системи:**

```
┌─────────────────────────────────────────────────┐
│           ПАРА КЛЮЧІВ                           │
│                                                 │
│  🔓 Відкритий ключ (Public Key)                 │
│     • Поширюється вільно                        │
│     • Зашифрувати → тільки власник дешифрує     │
│     • Перевірити підпис                         │
│                                                 │
│  🔐 Закритий ключ (Private Key)                 │
│     • Зберігається в таємниці                   │
│     • Дешифрувати → що зашифровано public key   │
│     • Підписати дані                            │
└─────────────────────────────────────────────────┘
```

**Два режими роботи:**

| Операція | Що використовується | Призначення |
|----------|-------------------|-------------|
| Шифрування | Public key одержувача | Конфіденційність |
| Дешифрування | Private key одержувача | Розкриття даних |
| Підписання | Private key підписанта | Автентичність |
| Верифікація | Public key підписанта | Перевірка підпису |

## Слайд 5 — RSA: принцип та особливості

# RSA (Rivest–Shamir–Adleman, 1977)

**Принцип безпеки:** складність **факторизації** великих цілих чисел

```
Генерація ключів (спрощено):
  ① Обираємо два великих простих числа p та q
  ② n = p × q  (відкрита частина: модуль)
  ③ Обчислюємо показники e та d
  ④ Public Key  = (n, e)
  ⑤ Private Key = (n, d)

  Шифрування:  C = M^e mod n
  Дешифрування: M = C^d mod n
```

**Розміри ключів та рекомендований термін використання:**

| Розмір ключа | Стійкість | Рекомендація |
|-------------|-----------|-------------|
| 1024 біт | ~80 біт | ❌ Застарілий |
| 2048 біт | ~112 біт | ✅ Мінімум (до ~2030) |
| 3072 біт | ~128 біт | ✅ Рекомендований |
| 4096 біт | ~140 біт | ✅ Довгострокове |

**Застосування в мережах:**
- TLS/HTTPS: шифрування pre-master secret або підпис у Key Exchange
- Цифровий підпис документів та коду
- S/MIME, PGP — захист електронної пошти

## Слайд 6 — ECC: еліптичні криві

# ECC — Elliptic Curve Cryptography

**Принцип безпеки:** складність задачі **дискретного логарифму** на еліптичній кривій

```
Геометрична інтуїція:
   │         ∙ P
   │        /
   │─────/──────── 
   │    /   ∙ Q = k×P
   │   /
   └──────────────

  Знаючи P і Q, знайти k — обчислювально неможливо
  (задача ECDLP — Elliptic Curve Discrete Log Problem)
```

**Ефективність ECC vs RSA:**

| Рівень стійкості | RSA (біт) | ECC (біт) |
|-----------------|-----------|-----------|
| 112 біт | 2048 | **224** |
| 128 біт | 3072 | **256** |
| 192 біт | 7680 | **384** |
| 256 біт | 15360 | **512** |

**Популярні криві:**
- **P-256 (secp256r1)** — NIST, широко в TLS, HTTPS
- **P-384** — підвищена безпека (NSA Suite B)
- **Curve25519** — висока швидкість, Tor, Signal, OpenSSH

**Переваги ECC:** менший ключ → менше CPU, пам'яті, трафіку → ідеальний для IoT, мобільних пристроїв

## Слайд 7 — Diffie-Hellman: обмін ключами

# Diffie-Hellman (DH)

**Мета:** виробити **спільний секрет** через незахищений канал, не передаючи його безпосередньо

```
  Alice                      Bob
  ──────                     ──────
  Генерує a (private)        Генерує b (private)
  A = g^a mod p              B = g^b mod p
         ─────────A──────────▶
         ◀────────B────────────
  S = B^a mod p              S = A^b mod p
  └──────────────────────────────────┘
            S = g^(ab) mod p  (однакові!)

  Підслуховувач бачить: g, p, A, B  — знайти S неможливо
```

**Різновиди DH:**

| Варіант | Опис | Особливість |
|---------|------|-------------|
| **DH** | Класичний, на цілих числах | Стандарт (RFC 3526) |
| **ECDH** | На еліптичних кривих | Менші ключі |
| **DHE** | Ephemeral DH — тимчасові ключі | **PFS** |
| **ECDHE** | Ephemeral ECDH | **PFS** + ефективність |

**PFS (Perfect Forward Secrecy):** нові ключі для кожної сесії → компрометація довгострокового ключа **не розкриває** минулий трафік

**DH не забезпечує автентифікацію!** → використовується разом із цифровим підписом або сертифікатами

## Слайд 8 — Порівняння алгоритмів: RSA vs ECC vs DH

# Порівняння: RSA / ECC / DH

| Критерій | RSA | ECC | DH / ECDH |
|----------|-----|-----|-----------|
| **Математична основа** | Факторизація | ECDLP | DLP / ECDLP |
| **Розмір ключа** | Великий (2048+) | Малий (256+) | Залежить від варіанту |
| **Швидкість** | Повільніший | Швидший | Швидкий |
| **Шифрування даних** | ✅ | ✅ | ❌ |
| **Цифровий підпис** | ✅ RSA-PSS | ✅ ECDSA | ❌ |
| **Обмін ключами** | ✅ (RSA KeyEx) | ✅ ECDH | ✅ (основна мета) |
| **PFS** | ❌ (без DHE) | ❌ (без ECDHE) | ✅ (DHE/ECDHE) |
| **Квантова стійкість** | ❌ | ❌ | ❌ |
| **Застосування в TLS 1.3** | Підпис | Підпис (ECDSA) | ECDHE обмін ключами |

> **Типова комбінація в TLS 1.3:** ECDHE (обмін ключами) + ECDSA або RSA (автентифікація) + AES-GCM (шифрування даних)

## Слайд 9 — Цифровий підпис: концепція та цілі

# Цифровий підпис

**Цифровий підпис** — криптографічний аналог власноручного підпису, що прив'язує дані до ідентичності підписанта

**Три властивості, що забезпечує:**

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  1. АВТЕНТИЧНІСТЬ — підтвердження особи підписанта   │
│     "Саме Аліса підписала цей документ"              │
│                                                      │
│  2. ЦІЛІСНІСТЬ — виявлення будь-яких змін            │
│     "Документ не змінено після підписання"           │
│                                                      │
│  3. НЕВІДРЕЧЕНІСТЬ — неможливість відмови від підпису│
│     "Аліса не може заявити 'я не підписувала'"       │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**Що підписується насправді?**

> Підписується не саме повідомлення, а його **криптографічний хеш** (дайджест)

- Фіксований розмір незалежно від розміру документа
- Швидше обчислюється
- Будь-яка зміна документа → повністю інший хеш

## Слайд 10 — Механізм цифрового підпису: підписання

# Фаза підписання

```
  ┌─────────────────────────────────────────────────────────┐
  │                    ПІДПИСАНТ (Alice)                    │
  │                                                         │
  │  Документ M         Хеш-функція       Хеш h            │
  │  ┌────────┐    →    (SHA-256)    →    ┌────────┐        │
  │  │"Оплата"│                           │ h(M)   │        │
  │  └────────┘                           └────┬───┘        │
  │                                            │            │
  │  Private Key Alice                    Шифрування       │
  │  ┌──────────┐   ────────────────────▶  RSA/ECDSA        │
  │  │ priv(A)  │                           │              │
  │  └──────────┘                           ▼              │
  │                                    ┌────────────┐      │
  │                                    │  Підпис S  │      │
  │                                    └────────────┘      │
  └─────────────────────────────────────────────────────────┘
  
  Надсилається: Документ M  +  Підпис S  (+  Сертифікат Alice)
```

**Ключовий момент:** підпис `S` створюється **закритим ключем** Alice  
→ лише Alice могла його створити (вона єдиний власник priv(A))

## Слайд 11 — Механізм цифрового підпису: перевірка

# Фаза перевірки

```
  ┌─────────────────────────────────────────────────────────┐
  │                   ПЕРЕВІРЯЛЬНИК (Bob)                   │
  │                                                         │
  │  Отримує: Документ M  +  Підпис S                       │
  │                                                         │
  │  ① Обчислює хеш отриманого документа:                   │
  │     M  ──▶ SHA-256 ──▶  h₁                              │
  │                                                         │
  │  ② Дешифрує підпис відкритим ключем Alice:              │
  │     S  +  pub(A)  ──▶ RSA/ECDSA ──▶  h₂                 │
  │                                                         │
  │  ③ Порівнює:                                             │
  │        h₁ == h₂  ?                                      │
  │        ✅  ТАК → підпис дійсний, дані не змінено        │
  │        ❌  НІ  → підпис недійсний або дані змінено       │
  │                                                         │
  └─────────────────────────────────────────────────────────┘
```

**Звідки Bob знає pub(A)?**  
→ З цифрового сертифіката Alice, виданого довіреним центром сертифікації (CA)

## Слайд 12 — Алгоритми цифрового підпису

# Алгоритми цифрового підпису

| Алгоритм | Основа | Хеш | Типовий розмір | Стандарт |
|----------|--------|-----|---------------|---------|
| **RSA-PKCS#1 v1.5** | RSA | SHA-1/256 | 2048–4096 біт | PKCS#1 (застарілий) |
| **RSA-PSS** | RSA + рандомізація | SHA-256/384 | 2048–4096 біт | PKCS#1 v2.1 ✅ |
| **DSA** | DLP | SHA-1/256 | 1024–3072 біт | FIPS 186 (застарілий) |
| **ECDSA** | ECC | SHA-256/384 | 256–384 біт | FIPS 186, RFC 6979 ✅ |
| **EdDSA** | Ed25519/Ed448 | SHA-512 | 256/448 біт | RFC 8032 ✅ |

**Рекомендовані сьогодні:**
- **RSA-PSS** з ключем ≥ 2048 біт — сумісність з legacy-системами
- **ECDSA P-256** — широке застосування (TLS, HTTPS, підпис коду)
- **EdDSA (Ed25519)** — найновіший, найшвидший, стійкий до помилок реалізації

> TLS 1.3 підтримує лише RSA-PSS, ECDSA та EdDSA (старіші схеми виключені)

## Слайд 13 — Невідреченість: технічний та правовий аспект

# Невідреченість (Non-repudiation)

**Технічний аспект:**

```
  ┌───────────────────────────────────────────────────────┐
  │  Аліса підписала контракт своїм private key           │
  │                                                       │
  │  Пізніше заявляє: "Я не підписувала!"                 │
  │                                                       │
  │  Боб може довести:                                    │
  │  ✓ Підпис перевіряється public key Аліси             │
  │  ✓ Public key прив'язаний до Аліси сертифікатом CA   │
  │  ✓ CA перевірив особу Аліси при видачі сертифіката   │
  │  → Аліса не може відмовитися від підпису             │
  └───────────────────────────────────────────────────────┘
```

**Умови забезпечення невідреченості:**

1. **Захист private key** — лише власник має доступ (HSM, smart card)
2. **Довіра до CA** — CA надійно підтвердив ідентичність власника
3. **Відмітка часу (Timestamp)** — TSA (Time Stamp Authority) фіксує момент підписання
4. **Цілісність** — хеш зафіксований у підписі

**Правовий аспект:**
- EU: Регламент eIDAS визначає три рівні підпису: Simple, Advanced, **Qualified**
- Кваліфікований електронний підпис = юридично еквівалентний власноручному
- Вимагає: кваліфікований CA + захищений пристрій підпису (QSCD)

## Слайд 14 — PKI: інфраструктура відкритих ключів

# PKI — Public Key Infrastructure

**PKI** — система, що вирішує фундаментальну проблему: **як довіряти відкритому ключу?**

```
  Проблема:
  ┌──────┐                              ┌──────┐
  │Alice │  pub_key_alice@internet?     │ Bob  │
  └──────┘  Як знати, що це справжній? └──────┘

  Рішення — PKI:
  ┌──────┐   Надай CSR    ┌──────┐   Видає сертифікат  ┌──────┐
  │Alice │──────────────▶│  CA  │──────────────────────▶│Alice │
  └──────┘               └──────┘                       └──────┘
                            │
                    Підписує pub_key_alice
                    своїм private key CA
                            │
  ┌──────┐   Перевіряє pub key CA (вже довіряє CA)
  │ Bob  │──────────────────────────────────────────▶ Довіряє!
  └──────┘
```

**PKI вирішує задачі:**
- **Прив'язка ключа до ідентичності** (сертифікат)
- **Розподіл довіри** (ієрархія CA)
- **Управління життєвим циклом** (видача, відкликання, поновлення)

## Слайд 15 — Архітектура PKI: компоненти та взаємодія

# Архітектура PKI

```
  ┌─────────────────────────────────────────────────────────┐
  │                      PKI                               │
  │                                                         │
  │  ┌──────────────┐   Реєстрація   ┌──────────────────┐  │
  │  │ Заявник      │──────────────▶│ RA               │  │
  │  │ (End-Entity) │               │ Registration     │  │
  │  └──────────────┘               │ Authority        │  │
  │         ▲                       └────────┬─────────┘  │
  │         │                                │ Верифікація │
  │  Сертифікат                              ▼             │
  │         │                       ┌──────────────────┐  │
  │         └───────────────────────│ CA               │  │
  │                                 │ Certificate      │  │
  │  ┌──────────────┐               │ Authority        │  │
  │  │ VA           │               └────────┬─────────┘  │
  │  │ Validation   │◀────────────────────────┘            │
  │  │ Authority    │    Публікує CRL / OCSP                │
  │  └──────┬───────┘                                      │
  │         │ OCSP Responses        ┌──────────────────┐  │
  │         └──────────────────────▶│ Repository       │  │
  │                                 │ (CRL, Certs)     │  │
  │                                 └──────────────────┘  │
  └─────────────────────────────────────────────────────────┘
```

| Компонент | Функція |
|-----------|---------|
| **CA** | Видає та підписує сертифікати |
| **RA** | Перевіряє особу заявника |
| **VA** | Відповідає на запити OCSP |
| **Repository** | Зберігає сертифікати та CRL |
| **TSA** | Штамп часу для підпису |

## Слайд 16 — Ієрархія довіри

# Ієрархія довіри PKI

```
  ┌─────────────────────────────────┐
  │         Root CA                 │  ← Самопідписаний
  │  (Довірений якір)               │    Зберігається ОФЛАЙН
  │  Термін: 20–25 років            │    Вбудований в ОС/браузер
  └────────────────┬────────────────┘
                   │ Підписує сертифікат
          ┌────────▼────────┐
          │  Intermediate   │  ← Підписаний Root CA
          │  CA (Sub-CA)    │    Зберігається ОНЛАЙН
          │  Термін: 5–10 р │    Видає кінцеві сертифікати
          └───────┬─────────┘
         ┌────────┴────────┐
    ┌────▼────┐       ┌────▼────┐
    │ Server  │       │  User   │  ← End-Entity Certificates
    │  cert   │       │  cert   │    Термін: 1–2 роки
    └─────────┘       └─────────┘
```

**Навіщо Intermediate CA?**
- Root CA залишається **офлайн** → недосяжний для атак
- При компрометації Intermediate CA: Root CA відкликає його сертифікат → видаємо новий Intermediate CA
- Root CA **не потрібно переналаштовувати** на клієнтах

**Довіра:** клієнт довіряє Root CA (вбудований в ОС) → верифікує ланцюжок → довіряє End-Entity

## Слайд 17 — Центр сертифікації (CA): функції

# Центр сертифікації (CA)

**CA — третя довірена сторона (TTP — Trusted Third Party)**

**Основні функції CA:**

| Функція | Опис |
|---------|------|
| **Перевірка особи** | Валідація заявника перед видачею сертифіката |
| **Видача сертифіката** | Підписання відкритого ключа заявника |
| **Управління сертифікатами** | Зберігання, поновлення, відкликання |
| **Публікація CRL** | Список відкликаних сертифікатів |
| **OCSP** | Онлайн-відповіді на запити про статус |

**Типи CA:**
- **Public CA** (комерційні): DigiCert, Sectigo, Let's Encrypt — довіряються усіма ОС/браузерами
- **Private CA** (корпоративні): Microsoft AD CS, HashiCorp Vault, AWS Private CA — для внутрішніх потреб
- **Government CA** — державні CA (наприклад, для eID, e-документів)

**Відповідальність CA:**
- Суворе дотримання **CP (Certificate Policy)** та **CPS (Certification Practice Statement)**
- Регулярний аудит (WebTrust, ETSI EN 319 411)
- Участь у **CA/Browser Forum** — визначає вимоги для публічних CA

## Слайд 18 — X.509: структура цифрового сертифіката

# Стандарт X.509 v3

**X.509** — стандарт ITU-T (RFC 5280), що визначає формат сертифіката відкритого ключа

```
  ┌──────────────────────────────────────────────┐
  │           TBSCertificate                     │
  │  ┌────────────────────────────────────────┐  │
  │  │ Version: v3                            │  │
  │  │ Serial Number: 0x3A4F8B...             │  │
  │  │ Signature Algorithm: SHA256withRSA     │  │
  │  │ Issuer: CN=DigiCert CA, O=DigiCert...  │  │
  │  │ Validity: 2024-01-01 – 2025-01-01      │  │
  │  │ Subject: CN=example.com, O=Corp...     │  │
  │  │ Subject Public Key Info:               │  │
  │  │   Algorithm: RSA / EC                  │  │
  │  │   Public Key: <бітовий рядок>          │  │
  │  │ Extensions: ...                        │  │
  │  └────────────────────────────────────────┘  │
  │                                              │
  │  Signature Algorithm: SHA256withRSA          │
  │  Signature Value: <підпис CA>                │
  └──────────────────────────────────────────────┘
```

> **Підпис CA** охоплює весь блок TBSCertificate → будь-яка зміна будь-якого поля робить підпис недійсним

## Слайд 19 — Поля X.509: детальний розгляд

# Поля сертифіката X.509

| Поле | Опис | Приклад |
|------|------|---------|
| **Version** | Версія стандарту | `v3` (найпоширеніша) |
| **Serial Number** | Унікальний № у межах CA | `3A:4F:8B:…` |
| **Signature Algorithm** | Алгоритм підпису CA | `SHA256withRSA`, `ecdsa-with-SHA256` |
| **Issuer** | Distinguished Name видавця | `CN=DigiCert TLS RSA CA` |
| **Not Before** | Початок терміну дії | `2024-03-01 00:00:00 UTC` |
| **Not After** | Кінець терміну дії | `2025-03-01 23:59:59 UTC` |
| **Subject** | DN власника | `CN=example.com, O=Example Corp` |
| **Public Key Algorithm** | Алгоритм ключа | `RSA 2048`, `EC P-256` |
| **Public Key** | Відкритий ключ | `<4096-bit value>` |
| **Extensions** | Розширення v3 | SAN, KeyUsage, AIA, CDP… |
| **Signature** | Підпис CA | `<RSA/ECDSA signature>` |

**Distinguished Name (DN) — ідентифікаційна інформація:**
- `CN` — Common Name (ім'я домену або особи)
- `O` — Organization
- `OU` — Organizational Unit
- `C` — Country (двохбуквений код ISO 3166)
- `ST` — State/Province
- `L` — Locality (місто)

## Слайд 20 — Розширення X.509 v3

# Розширення X.509 v3

> Розширення — механізм гнучкого розширення сертифіката без порушення базової структури

**Критичні (Critical) розширення** — клієнт **зобов'язаний** їх обробити; якщо не вміє → відхилити сертифікат.  
**Некритичні** — можна проігнорувати.

| Розширення | Critical? | Призначення |
|-----------|-----------|-------------|
| **Basic Constraints** | ✅ Так | `cA: TRUE/FALSE` — чи є суб'єкт CA |
| **Key Usage** | ✅ Так | Дозволені операції: `digitalSignature`, `keyEncipherment`, `keyCertSign` |
| **Extended Key Usage (EKU)** | Ні | `TLS Server Auth`, `TLS Client Auth`, `Code Signing`, `Email Protection` |
| **Subject Alternative Name (SAN)** | ✅ (обов'язковий) | Альтернативні ідентифікатори: `DNS:example.com`, `IP:1.2.3.4`, `email:…` |
| **Subject Key Identifier (SKI)** | Ні | Хеш відкритого ключа суб'єкта |
| **Authority Key Identifier (AKI)** | Ні | Хеш ключа CA-видавця (для побудови ланцюжка) |
| **CRL Distribution Points (CDP)** | Ні | URL для завантаження CRL |
| **Authority Info Access (AIA)** | Ні | URL OCSP-респондера + URL сертифіката видавця |

## Слайд 21 — Перевірка ланцюжка сертифікатів

# Перевірка ланцюжка (Chain Validation)

```
  Клієнт отримує: Сертифікат сервера  +  Intermediate CA сертифікат

  Алгоритм перевірки:
  ┌─────────────────────────────────────────────────────────┐
  │  ① Перевірити підпис: чи підписаний cert сервера       │
  │     ключем Intermediate CA?                            │
  │                                                         │
  │  ② Перевірити підпис: чи підписаний Intermediate CA    │
  │     ключем Root CA?                                     │
  │                                                         │
  │  ③ Чи є Root CA в сховищі довірених (Trust Store)?     │
  │                                                         │
  │  ④ Перевірити терміни дії (Not Before / Not After)      │
  │     для кожного сертифіката в ланцюжку                  │
  │                                                         │
  │  ⑤ Перевірити статус відкликання (CRL або OCSP)        │
  │     для кожного сертифіката в ланцюжку                  │
  │                                                         │
  │  ⑥ Перевірити Key Usage та EKU                         │
  │                                                         │
  │  ⑦ Перевірити Subject / SAN ↔ hostname                 │
  └─────────────────────────────────────────────────────────┘
         Всі перевірки пройдено → ✅ Сертифікат довірений
```

## Слайд 23 — CSR: запит та видача сертифіката

# CSR — Certificate Signing Request

**CSR** — запит на підпис сертифіката, підписаний власним закритим ключем заявника

```
  ┌────────────────────────────────────────────────────────┐
  │                    ПРОЦЕС ВИДАЧІ                       │
  │                                                        │
  │  1. Власник генерує пару ключів:                       │
  │     $ openssl genrsa -out server.key 2048              │
  │                                                        │
  │  2. Формує CSR (вміщує Public Key + Subject + підпис): │
  │     $ openssl req -new -key server.key -out server.csr │
  │     CN=example.com, O=Corp, C=UA                       │
  │                                                        │
  │  3. Надсилає CSR → CA (через веб-інтерфейс або ACME)   │
  │                                                        │
  │  4. CA перевіряє:                                      │
  │     • DV: DNS-запис або HTTP-файл                      │
  │     • OV: документи організації                        │
  │     • EV: посилена перевірка (юридична особа)          │
  │                                                        │
  │  5. CA підписує сертифікат і повертає .crt файл        │
  │     (сертифікат сервера + ланцюжок Intermediate CA)    │
  └────────────────────────────────────────────────────────┘
```

> CSR **не містить** закритого ключа. Закритий ключ **ніколи** не залишає систему власника.

## Слайд 24 — Типи сертифікатів

# Типи сертифікатів

**За рівнем валідації:**

| Тип | Перевіряється | Час видачі | Візуально в браузері |
|-----|--------------|-----------|---------------------|
| **DV** (Domain Validation) | Контроль над доменом | Хвилини–години | 🔒 Замок |
| **OV** (Organization Validation) | Домен + організація | Дні | 🔒 Замок (+ деталі в сертифікаті) |
| **EV** (Extended Validation) | Повна юридична перевірка | Тижні | 🔒 (раніше — зелений рядок) |

**За охопленням доменів:**

| Тип | Охоплення | Приклад |
|-----|-----------|---------|
| **Single Domain** | Один FQDN | `example.com` |
| **Wildcard** | Усі піддомени одного рівня | `*.example.com` → `a.example.com`, `b.example.com` |
| **SAN / Multi-Domain (MDC)** | Перелік різних доменів | `example.com`, `example.net`, `mail.corp.ua` |

**За призначенням (EKU):**
- **TLS Server Auth** — HTTPS-сервери
- **TLS Client Auth** — автентифікація клієнтів (mTLS)
- **Code Signing** — підпис програм та оновлень
- **Email Protection (S/MIME)** — підпис та шифрування e-mail
- **Time Stamping** — TSA (Time Stamp Authority)

## Слайд 25 — CRL: список відкликаних сертифікатів

# CRL — Certificate Revocation List

**CRL** — підписаний CA список серійних номерів відкликаних сертифікатів

```
  ┌──────────────────────────────────────────────────────┐
  │                  CRL (підписаний CA)                 │
  │                                                      │
  │  Issuer: CN=DigiCert CA                              │
  │  This Update: 2024-03-27 10:00:00 UTC                │
  │  Next Update: 2024-03-28 10:00:00 UTC                │
  │  ┌──────────────────────────────────────────────┐    │
  │  │  Serial: 0x1A2B3C  Revoked: 2024-03-20       │    │
  │  │  Reason: keyCompromise                       │    │
  │  │  Serial: 0x4D5E6F  Revoked: 2024-03-25       │    │
  │  │  Reason: cessationOfOperation               │    │
  │  │  …                                           │    │
  │  └──────────────────────────────────────────────┘    │
  │  Signature: <підпис CA>                              │
  └──────────────────────────────────────────────────────┘

  Клієнт:  завантажує весь файл CRL  →  шукає серійний № сертифіката
```

**Причини відкликання (Reason Codes):**
`keyCompromise` | `cACompromise` | `affiliationChanged` | `superseded` | `cessationOfOperation` | `privilegeWithdrawn`

**Недоліки CRL:**
- Клієнт завантажує **весь список** (може бути великим)
- Актуальність обмежена інтервалом публікації
- **CRL Caching** може призводити до використання застарілих даних

## Слайд 26 — OCSP: онлайн-перевірка статусу

# OCSP — Online Certificate Status Protocol

**OCSP (RFC 6960)** — протокол запиту статусу **одного конкретного** сертифіката

```
  ┌──────┐      OCSP Request               ┌──────────────┐
  │Client│  SerialNumber: 0x1A2B3C  ──────▶│ OCSP         │
  │      │  IssuerNameHash: SHA1(…) │       │ Responder    │
  │      │  IssuerKeyHash:  SHA1(…) │       │ (VA)         │
  │      │◀────────────────────────────────│              │
  │      │      OCSP Response              │              │
  │      │  Status: "good"                 └──────────────┘
  │      │  ThisUpdate: 2024-03-27
  │      │  NextUpdate: 2024-03-28
  │      │  ResponderID: CN=OCSP Responder
  │      │  Signature: <підпис>
  └──────┘

  Можливі статуси відповіді:
  ✅ good     — сертифікат дійсний
  ❌ revoked  — сертифікат відкликано (+ час, причина)
  ⚠️ unknown  — невідомий сертифікат
```

**OCSP Stapling (RFC 6066):**  
Сервер **заздалегідь** отримує OCSP-відповідь від VA та "прикріплює" її до TLS Handshake  
→ Клієнт не звертається до OCSP Responder окремо  
→ Менша затримка + збереження приватності

## Слайд 27 — CRL vs OCSP: порівняння

# CRL vs OCSP: порівняння

| Критерій | CRL | OCSP | OCSP Stapling |
|----------|-----|------|---------------|
| **Механізм** | Завантаження списку | Запит одного сертифіката | Відповідь від сервера |
| **Розмір трафіку** | Великий (весь CRL) | Малий (1 запит/відповідь) | Мінімальний (у TLS HS) |
| **Актуальність** | До наступного CRL | Майже реального часу | За кешем (кілька годин) |
| **Навантаження на CA** | Низьке | Може бути високим | Знижує навантаження |
| **Конфіденційність** | ✅ (офлайн) | ❌ (CA бачить запити) | ✅ (прямого запиту немає) |
| **Підтримка** | Усі клієнти | Сучасні клієнти | TLS 1.2+ (з розширенням) |
| **Проблема** | Затримка відкликання | Hard-fail vs Soft-fail | Сервер може не підтримувати |

**Soft-fail vs Hard-fail:**
- **Soft-fail** (за замовчуванням): якщо OCSP недоступний → сертифікат прийнятий  
  → ризик: атака відключення OCSP
- **Hard-fail**: якщо OCSP недоступний → з'єднання відхилено  
  → вищий рівень безпеки, але менша відмовостійкість

## Слайд 28 — Застосування PKI

# Застосування PKI

| Сценарій | Протокол/Стандарт | Роль PKI |

| **IoT / Device Auth** | EST (RFC 7030) | Автоматична видача сертифікатів пристроям |
