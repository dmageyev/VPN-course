## Конспект лекції

### 1. Симетрична та асиметрична криптографія

Симетрична криптографія використовує **один спільний секретний ключ** для шифрування і дешифрування. Основна проблема — безпечна передача ключа між сторонами.

Асиметрична криптографія вирішує цю проблему шляхом використання **пари математично пов'язаних ключів**: відкритого (public key) та закритого (private key). Відкритий ключ поширюється вільно; закритий зберігається виключно у власника.

Асиметричні алгоритми є обчислювально значно важчими, тому на практиці їх використовують **не для масового шифрування даних**, а для:
- узгодження сеансових ключів (DH, ECDH);
- шифрування невеликих блоків даних (наприклад, симетричних ключів);
- цифрового підпису.

### 2. RSA

**RSA (Rivest–Shamir–Adleman, 1977)** — найпоширеніший асиметричний алгоритм. Безпека базується на складності факторизації великих цілих чисел. Типові розміри ключів: 2048, 3072, 4096 біт.

Застосування: TLS-рукостискання (шифрування pre-master secret), цифровий підпис, шифрування e-mail (S/MIME, PGP).

**Обмеження**: великий розмір ключа, відносно повільний порівняно з ECC.

### 3. ECC (Elliptic Curve Cryptography)

**ECC** базується на математичних властивостях еліптичних кривих. Головна перевага: **коротший ключ при еквівалентній стійкості**.

| RSA    | ECC   |
|--------|-------|
| 2048 б | 224 б |
| 3072 б | 256 б |
| 7680 б | 384 б |

Популярні криві: P-256 (secp256r1), P-384, Curve25519. Застосування: TLS 1.3, SSH, HTTPS, мобільні та IoT пристрої.

### 4. Diffie-Hellman (DH та ECDH)

**Diffie-Hellman (1976)** — перший опублікований протокол обміну ключами. Дозволяє двом сторонам виробити **спільний секрет через незахищений канал**, не передаючи його безпосередньо. Самостійно DH не забезпечує автентифікацію сторін.

**ECDH** — варіант DH на еліптичних кривих: значно коротші ключі та вища продуктивність.

**DHE/ECDHE** (Ephemeral): тимчасові ключі для кожної сесії → **Perfect Forward Secrecy (PFS)**: компрометація довгострокового ключа не розкриває записаний трафік минулих сесій.

### 5. Цифровий підпис

**Цифровий підпис** — криптографічний механізм, що забезпечує:
- **Автентичність** — підтвердження особи підписанта;
- **Цілісність** — виявлення будь-яких змін у документі після підписання;
- **Невідреченість** — підписант не може відмовитися від факту підписання.

**Механізм підписання:**
1. Обчислити хеш повідомлення (SHA-256, SHA-384 тощо).
2. Зашифрувати хеш **закритим ключем** підписанта → отримаємо підпис.
3. Відправити повідомлення разом із підписом.

**Механізм перевірки:**
1. Обчислити хеш отриманого повідомлення.
2. Розшифрувати підпис **відкритим ключем** підписанта → отримаємо очікуваний хеш.
3. Порівняти хеші. Збіг → підпис дійсний.

### 6. PKI — Інфраструктура відкритих ключів

**PKI (Public Key Infrastructure)** — сукупність апаратних засобів, програмного забезпечення, людей, політик і процедур для управління цифровими сертифікатами та ключами.

Основні компоненти:
- **CA (Certificate Authority)** — видає, підписує та відкликає сертифікати.
- **RA (Registration Authority)** — перевіряє особу заявника перед видачею сертифіката.
- **VA (Validation Authority)** — відповідає на запити про статус сертифікатів (OCSP).
- **Сховище (Repository/CRL)** — зберігає сертифікати та списки відкликання.
- **Суб'єкт (End-Entity)** — власник сертифіката (сервер, користувач, пристрій).

**Ієрархія довіри:**
```
Root CA  (самопідписаний, офлайн)
  └── Intermediate CA  (онлайн, видає кінцеві сертифікати)
        └── End-Entity Certificate  (сервер, користувач тощо)
```

Root CA зберігається офлайн для максимального захисту. Intermediate CA — операційний вузол ієрархії.

### 7. X.509 — Стандарт цифрового сертифіката

**X.509** — стандарт ITU-T, що визначає формат сертифіката відкритого ключа. Актуальна версія — **X.509 v3**.

Ключові поля:
| Поле | Опис |
|------|------|
| Version | Версія стандарту (v3) |
| Serial Number | Унікальний номер у межах CA |
| Signature Algorithm | Алгоритм підпису (SHA256withRSA тощо) |
| Issuer | DN центру сертифікації |
| Validity (Not Before / Not After) | Термін дії |
| Subject | DN власника сертифіката |
| Subject Public Key Info | Алгоритм + відкритий ключ |
| Extensions (v3) | Додаткові атрибути |
| Signature | Підпис CA |

Важливі розширення v3:
- **Basic Constraints** — чи є суб'єкт CA.
- **Key Usage** — дозволені криптографічні операції.
- **Extended Key Usage (EKU)** — TLS Server Auth, Code Signing тощо.
- **Subject Alternative Name (SAN)** — альтернативні імена (DNS, IP, e-mail).
- **CRL Distribution Points** — URL для завантаження CRL.
- **Authority Information Access (AIA)** — URL OCSP-респондера та Issuer.

### 8. Життєвий цикл сертифіката

**Видача (CSR — Certificate Signing Request):**
1. Власник генерує пару ключів.
2. Формує CSR (містить відкритий ключ та ідентифікаційні дані, підписаний закритим ключем).
3. Надсилає CSR до CA/RA.
4. CA перевіряє особу заявника (Domain Validation, Organization Validation тощо).
5. CA підписує сертифікат своїм закритим ключем і повертає власнику.

**Відкликання (Revocation):**
- **CRL (Certificate Revocation List)** — список відкликаних серійних номерів, підписаний CA. Публікується за розкладом (наприклад, щодня). Клієнт завантажує весь список.
- **OCSP (Online Certificate Status Protocol)** — клієнт надсилає запит із серійним номером, сервер (VA) повертає відповідь: `good`, `revoked`, `unknown`. Актуальніший і ефективніший за CRL.
- **OCSP Stapling** — сервер заздалегідь кешує підписану OCSP-відповідь і передає її клієнту під час TLS-рукостискання; знижує затримку.

### 9. Застосування PKI

| Сценарій | Опис |
|----------|------|
| **TLS/HTTPS** | Автентифікація сервера, узгодження сеансового ключа |
| **VPN (IKEv2/IPsec)** | Автентифікація вузлів замість PSK |
| **S/MIME** | Шифрування та підпис електронної пошти |
| **Підпис коду** | Верифікація програмного забезпечення та оновлень |
| **Smart Card / eID** | Автентифікація користувачів |
| **Document Signing** | Юридично значущий електронний підпис |
