Підпротоколи TLS

  • Record Protocol — базовий транспорт; фрагментація, стиснення, шифрування
  • Handshake Protocol — узгодження параметрів, аутентифікація
  • Change Cipher Spec — сигнал переходу до нових ключів
  • Alert Protocol — повідомлення про помилки та закриття

TLS розташований між TCP і застосунком — захищає будь-який TCP-трафік без змін на нижніх рівнях.

Фаза 1 — Negotiation

ClientHello + ServerHello: вибір версії TLS, cipher suite, обмін random-значеннями.

Фаза 2 — Authentication

Сервер надсилає сертифікат X.509. Клієнт перевіряє ланцюжок до довіреного CA.

Фаза 3 — Key Exchange

Pre-Master Secret → Master Secret → 4 ключі (enc + MAC для кожного напряму).

Фаза 4 — Finished

Верифікація цілісності Handshake. 2 RTT до початку передачі даних.

Ключові покращення TLS 1.3

  • 1-RTT замість 2-RTT — вдвічі швидше
  • 0-RTT (Early Data) — відновлення без затримки
  • Сертифікат і розширення зашифровані
  • Обов'язковий Forward Secrecy (ECDHE)
  • Вилучено: RSA KeyEx, DH статичний, RC4, 3DES, MD5

0-RTT застереження

Early Data вразливе до replay-атак. Не підходить для не-ідемпотентних операцій (POST, платежі).

TLS 1.3 Cipher Suites

Лише 5 дозволених: AES-128-GCM-SHA256, AES-256-GCM-SHA384, ChaCha20-Poly1305-SHA256 та ін. Обмін ключами — завжди ECDHE.

Механізми обміну ключами

RSA Key Exchange (застарілий)

Клієнт шифрує Pre-Master Secret відкритим ключем сервера. Немає Forward Secrecy — компрометація приватного ключа розкриває всі минулі сесії.

Diffie-Hellman (DH)

Сторони обмінюються ga mod p та gb mod p → спільний секрет gab mod p. Статичний DH також не забезпечує PFS.

ECDHE — рекомендований

Ephemeral Elliptic Curve DH — тимчасові ключі для кожної сесії. Забезпечує Perfect Forward Secrecy: компрометація довгострокових ключів не розкриває минулі сесії.

  • Крива P-256 (secp256r1) — стандартна
  • X25519 — рекомендована в TLS 1.3
  • Менший розмір ключів ніж RSA

Master Secret

Pre-Master Secret + random клієнта + random сервера → PRF → Master Secret (48 байт) → 4 симетричні ключі (enc + MAC × 2 напрями).

Ключові особливості OpenVPN

  • Працює у User-space (не в ядрі)
  • Використовує OpenSSL або mbed TLS
  • Підтримує TUN (L3) та TAP (L2)
  • Транспорт: UDP 1194 (default) або TCP 443
  • Кросплатформений: Linux, Windows, macOS, iOS, Android
  • Open Source (GNU GPL v2)

Два незалежних канали

  • Control Channel — TLS, управління, обмін ключами
  • Data Channel — зашифровані дані користувача

Control Channel vs Data Channel

🔑 Control Channel

  • Базується на TLS
  • Взаємна аутентифікація (PKI, сертифікати X.509)
  • Обмін та оновлення ключів Data Channel
  • Передача конфігурації: IP, маршрути, DNS
  • Heartbeat / keepalive пакети
  • Захищено tls-auth або tls-crypt (HMAC)

📦 Data Channel

  • Передача зашифрованих даних користувача
  • Алгоритми: AES-256-GCM, ChaCha20-Poly1305
  • Автоматичне оновлення ключів (rekeying) кожні 3600 с
  • HMAC аутентифікація пакетів (якщо не AEAD)
  • Compression — опціонально (знижує безпеку!)

Ключі Data Channel генеруються через Control Channel і регулярно оновлюються — забезпечується Forward Secrecy навіть для потоку даних.

Фази встановлення з'єднання OpenVPN

🔹 Фаза 1: Транспортне з'єднання

Клієнт встановлює UDP (1194) або TCP (443) з'єднання з сервером. Лише транспортний рівень — даних ще немає.

🔹 Фаза 2: TLS Handshake (Control Channel)

Взаємна аутентифікація через X.509. Обмін ключами ECDHE. Якщо налаштовано tls-crypt — шифрування Control Channel від початку.

🔹 Фаза 3: Узгодження параметрів

Через Control Channel: ключі Data Channel, IP-адреси, маршрути (push), DNS, компресія.

🔹 Фаза 4: Налаштування інтерфейсу

OpenVPN створює tun0 / tap0. Призначає IP (push від сервера). Додає маршрути в таблицю маршрутизації ОС.

🔹 Фаза 5: Передача даних

Трафік: tun0 → OpenVPN шифрує (Data Channel) → UDP/TCP → сервер розшифровує → інжектує у tun0 → корпоративна мережа.

🔹 Фаза 6: Rekeying

Кожні 3600 с Control Channel оновлює ключі Data Channel без переривання трафіку.

OpenVPN: UDP проти TCP

🚀 UDP (порт 1194) — рекомендований

  • Нижча затримка, вища пропускна здатність
  • Без підтвердження доставки — менші накладні витрати
  • VPN сам керує надійністю поверх UDP
  • Уникає проблеми TCP-over-TCP meltdown
  • Ідеально для відео/голосу/ігор у тунелі

🛡️ TCP (порт 443) — обхід обмежень

  • Імітує HTTPS-трафік → обхід DPI-фільтрів
  • Гарантована доставка пакетів
  • Підходить при блокуванні UDP провайдером
  • Проблема: TCP всередині TCP при втратах пакетів

TCP-over-TCP Meltdown

Якщо зовнішній TCP-пакет втрачається → обидва TCP-стеки (зовнішній і внутрішній) починають ретрансмісію з exponential backoff → лавиноподібне падіння продуктивності. При 5% втрат TCP-режим може бути у 10× повільнішим за UDP.

Аутентифікація в OpenVPN

PKI (сертифікати X.509)

  • Кожен клієнт — унікальний сертифікат
  • Сервер перевіряє ланцюжок до Root CA
  • Відкликання: CRL або OCSP
  • Генерація: easy-rsa або openssl

Username / Password

Додаткова аутентифікація поверх PKI. Перевіряється через PAM, RADIUS або LDAP плагін.

MFA / OTP — багатофакторна

  • Фактор 1: сертифікат X.509 (щось є)
  • Фактор 2: TOTP-код (Google Authenticator, Authy)
  • Фактор 3: пароль або апаратний токен (YubiKey)
  • OTP передається у полі пароля через PAM-плагін
  • TOTP: RFC 6238; 6-значний код, оновлюється кожні 30 с

tls-auth / tls-crypt

  • tls-auth: HMAC-підпис Control пакетів — захист від DoS та сканування
  • tls-crypt: шифрування + HMAC Control Channel — прихованість від DPI

AEAD: сучасне шифрування даних

AEAD (Authenticated Encryption with Associated Data) — одночасно шифрує та аутентифікує дані. Замінює окремий HMAC + CBC.

AES-256-GCM

  • AES у режимі Galois/Counter Mode
  • 128-бітний Authentication Tag вбудований
  • Апаратне прискорення (AES-NI на x86)
  • Рекомендований для серверів з AES-NI
  • TLS 1.3 та OpenVPN 2.4+

ChaCha20-Poly1305

  • ChaCha20 (stream cipher) + Poly1305 (MAC)
  • Швидший на пристроях без AES-NI (мобільні, IoT)
  • Рівна безпека з AES-256-GCM
  • Автор: Д. Бернштейн (DJB)
  • TLS 1.3 та OpenVPN 2.5+

tls-auth та tls-crypt: детальне порівняння

tls-auth (HMAC firewall)

Симетричний PSK-ключ. HMAC-підпис кожного Control-пакету. Пакет без правильного HMAC відкидається до початку TLS Handshake.

  • Захист від DoS-атак на TLS стек
  • Захист від port scanning
  • Сервер: tls-auth ta.key 0
  • Клієнт: tls-auth ta.key 1
  • Не шифрує Control Channel

tls-crypt (шифрування + HMAC)

Розширення tls-auth: HMAC + шифрування всього Control Channel. Весь TLS Handshake прихований від стороннього спостерігача.

  • Усі переваги tls-auth плюс:
  • Конфіденційність Handshake (від DPI)
  • Захист від offline-аналізу трафіку
  • OpenVPN 2.4+ — рекомендований
  • Єдиний ключ для сервера і клієнта
openvpn --genkey tls-crypt tc.key

Вразливості та захисні заходи OpenVPN

Відомі вразливості

  • SWEET32 — атака на Blowfish/BF-CBC (64-бітний блок)
  • VORACLE — compression oracle при увімкненому LZO/LZ4
  • Слабкий tls-version-min → downgrade до TLS 1.0
  • Компрометація CA → підроблені клієнтські сертифікати
  • Відсутність tls-auth → DoS на TLS стек

Рекомендовані налаштування

  • cipher AES-256-GCM — замість BF-CBC
  • tls-version-min 1.2 — заборонити старі версії
  • comp-lzo no — вимкнути стиснення
  • tls-crypt tc.key — захист Control Channel
  • Регулярне оновлення CRL / OCSP Stapling
  • verify-client-cert require — обов. сертифікат клієнта
