User-space архітектура

OpenVPN працює у просторі користувача (не в ядрі). Перевага: простота розробки, переносимість. Недолік: context switching → додаткові затримки.

Control Channel

TLS/SSL — безпечний канал для керування сесією, обміну ключами та параметрами підключення.

Data Channel

Шифрований канал передачі користувацького трафіку (AES-GCM або ChaCha20-Poly1305).

Формат пакету OpenVPN

Зовнішній транспорт:
┌──────────────────────────────────┐
│  IP Header (IPv4/IPv6)           │
├──────────────────────────────────┤
│  UDP / TCP Header                │
├───────┬──────────────────────────┤
│OpCode │  Key ID (3 bits)         │  ← 1 байт
│5 bits │                          │
├───────┴──────────────────────────┤
│  Peer ID (3 байти, опціон.)      │
├──────────────────────────────────┤
│  Packet ID / Sequence Number     │  ← replay-захист
├──────────────────────────────────┤
│  HMAC / AEAD Authentication Tag  │
├──────────────────────────────────┤
│  IV (Initialization Vector)      │
├──────────────────────────────────┤
│  Encrypted Payload               │
│  (TLS record або IP-пакет)       │
└──────────────────────────────────┘

Типи Op Code (5 біт):

КодТип
0x01CONTROL_HARD_RESET_CLIENT
0x02CONTROL_HARD_RESET_SERVER
0x04CONTROL_V1 (керуючий)

Overhead (UDP-режим): IP(20)+UDP(8)+OpHdr(4)+PacketID(4)+AEAD Tag(16)+IV(12) ≈ ~65 байт

Фази встановлення з'єднання OpenVPN

Клієнт                              Сервер
  │                                    │
  │── ФАЗА 1: HARD_RESET ────────────► │
  │◄── HARD_RESET_SERVER ────────────  │
  │── ACK ──────────────────────────► │
  │                                    │
  │═══ ФАЗА 2: TLS Handshake ═════════ │
  │    (ClientHello → Finished)        │
  │                                    │
  │═══ ФАЗА 3: Key Negotiation ═══════ │
  │    (cipher, ключі Data Channel)    │
  │                                    │
  │═══ ФАЗА 4: Data Transfer ═════════ │
  │    (зашифровані IP-пакети)         │

Таймери за замовчуванням

  • tls-timeout 2 — повторна передача TLS-пакетів (сек)
  • reneg-sec 3600 — переузгодження ключів (1 год)
  • keepalive 10 60 — ping / disconnect timeout

Чому 5 фаз?

Фази 1–3 забезпечують безпечну ініціалізацію. Фаза 5 (Renegotiation) гарантує Forward Secrecy — навіть якщо ключ скомпрометований, попередні сесії залишаються захищеними.

Фаза 1 — Ініціалізація транспорту

OpenVPN Reliability Layer (над UDP):

Клієнт                              Сервер
  │                                    │
  │── HARD_RESET_CLIENT ─────────────► │
  │   (Session ID клієнта, 8 байт)     │
  │                                    │
  │◄── HARD_RESET_SERVER ────────────  │
  │    (Session ID сервера, 8 байт)    │
  │                                    │
  │── ACK ──────────────────────────► │
  │   (підтвердження HARD_RESET_SRV)  │

Компоненти Reliability Layer

  • Session ID — 8 байт, унікальний ID сесії
  • Packet ID — порядковий номер, replay-захист
  • ACK — підтвердження доставки
  • Sliding Window — до 8 пакетів
OpenVPN реалізує власний надійний транспорт поверх UDP лише для Control Channel. Data Channel не потребує гарантованої доставки на рівні VPN.

PUSH_REPLY від сервера

  • ifconfig — IP-адреса клієнта у VPN
  • route — маршрути до корпоративних мереж

PULL_REQUEST від клієнта

Клієнт запитує конфігурацію, якщо у налаштуваннях є директива --pull. Сервер відповідає PUSH_REPLY.

Розрахунок overhead (UDP)

IP(20) + UDP(8)
+ OpenVPN Hdr(1-4)
+ PacketID(4)
+ AEAD Tag(16)
+ IV(12)
= ~65 байт overhead

При MTU 1500 байт → ефективна MTU ≈ 1434 байти

Зворотній шлях (сервер → клієнт)

Сервер отримує зашифрований UDP-пакет, розшифровує, витягує IP-пакет та вкидає його у TUN-інтерфейс серверної мережі.

OpenVPN: UDP проти TCP

UDP (порт 1194) — рекомендований

  • Менші затримки, вища пропускна здатність
  • Немає проблеми «TCP over TCP meltdown»
  • Краще для стрімінгу, VoIP, ігор
  • OpenVPN реалізує свій Reliability Layer

TCP (порт 443) — для обходу фільтрів

  • Проходить крізь корпоративні файрволи (TCP 443)
  • Маскується під HTTPS-трафік
  • Менша продуктивність через подвійний TCP-стек

Проблема «TCP over TCP meltdown»

Внутрішній TCP (додаток)
↓ всередині
Зовнішній TCP (OpenVPN)
  • Обидва TCP незалежно керують retransmission
  • При втраті пакетів: подвійний backoff
  • Результат: експоненційне зростання затримок

«Що ти знаєш?» → LDAP, RADIUS, PAM

Багатофакторна аутентифікація (MFA)

TOTP-аутентифікація в OpenVPN:

Клієнт              OpenVPN + RADIUS/PAM
  │                         │
  │── TLS Handshake ────────► │ ← Фактор 1
  │   (X.509 cert)            │   (сертифікат)
  │                           │
  │── AUTH: user + pass+OTP ─► │ ← Фактор 2+3
  │   alice / p@ss123456       │   (пароль + TOTP)
  │                           │
  │              ┌────────────┤
  │              │ RADIUS/PAM │
  │              │ verify OTP │
  │              └────────────┤
  │◄── AUTH_SUCCEEDED ───────  │

Стандарти OTP

  • TOTP (RFC 6238) — Time-based OTP (Google Authenticator)
  • HOTP (RFC 4226) — HMAC-based OTP (лічильникові токени)
  • U2F/FIDO2 — апаратні ключі (YubiKey)

Переваги MFA

  • Захист при компрометації сертифікату або пристрою
  • Відповідність: GDPR, PCI DSS, ISO 27001, NIST SP 800-63B

Захист Control Channel: tls-auth та tls-crypt

Без захисту — вразливість

Будь-хто → UDP 1194 → OpenVPN
                  ↑ TLS Handshake flood
                    → CPU exhaustion DoS

--tls-auth (HMAC firewall)

Пакети без HMAC ──X──► відкидаються
                    до TLS Handshake

Захист від DoS, але Handshake видно у DPI

--tls-crypt (рекомендовано)

│Op Hdr  │ AEAD-зашифрований TLS Rec │
└────────┴────────────────────────────┘
↑ Handshake прихований від DPI!

Причини нижчої продуктивності OpenVPN

  • User-space: context switch ~1–5 мкс на пакет
  • TUN/TAP read/write syscall на кожен пакет
  • Одна нитка (single-threaded без патчів)

Оптимізації OpenVPN

  • cipher AES-256-GCM + AES-NI
  • --fast-io (тільки UDP)
  • --mssfix / --fragment для MTU

Мінімальна безпечна конфігурація OpenVPN:

tls-version-min 1.2       # заборонити TLS < 1.2
cipher AES-256-GCM        # AEAD-режим
auth SHA256               # HMAC для Data Channel
tls-crypt /etc/openvpn/ta.key   # захист Control Channel
verify-client-cert require      # обов'язкові сертифікати
remote-cert-tls server         # перевірка Extended KU
crl-verify /etc/openvpn/crl.pem # відкликані сертифікати

TLS 1.3 детальний аналіз

Що видалено в TLS 1.3
  • Статичний RSA key exchange (немає PFS)
  • RC4, DES, 3DES, MD5, SHA-1
  • CBC cipher suites
  • Компресія (CRIME атака)
Обов'язкові cipher suites
  • TLS_AES_128_GCM_SHA256
  • TLS_AES_256_GCM_SHA384
  • TLS_CHACHA20_POLY1305_SHA256
  • Всі з ECDHE + AEAD: нема слабких варіантів

SSL Inspection (SSL/TLS Interception)

SSL Inspection — техніка MITM для аналізу зашифрованого трафіку. Корпоративний проксі або NGFW дешифрує, інспектує та повторно шифрує трафік.

Конфіденційність
  • Корпоративний CA встановлюється на всі ПК
  • IT може бачити весь HTTPS трафік
  • Порушує e2e шифрування, HPKP
Безпека
  • Виявлення malware у зашифрованих завантаженнях
  • Запобігання exfiltration через HTTPS
  • DLP (Data Loss Prevention)

Кластеризація VPN-шлюзів

Load Balancing підходи
  • DNS Round Robin (простий, без health check)
  • L4 Load Balancer (HAProxy, AWS NLB)
  • Anycast IP (BGP-based, для великих мереж)
  • GSLB (Global Server Load Balancing)
Stateful HA
  • Синхронізація сесій між вузлами
  • Failover без розриву VPN-з'єднань
  • OpenVPN за L4 LB + sticky sessions
  • IPsec + VRRP для gateway HA

SSL VPN Web Portal (Clientless)

Clientless SSL VPN — доступ до внутрішніх ресурсів через веб-браузер без встановлення VPN-клієнта.

Переваги
  • Не потрібен клієнт на пристрої
  • Доступ з будь-якого браузера
  • Ідеально для партнерів та підрядників
  • Контроль доступу на рівні застосунку
Обмеження
  • Лише HTTP/HTTPS застосунки
  • Немає доступу до мережевих ресурсів
  • HTML5 RDP/SSH через браузер (laggy)
  • Перевірка stану пристрою неможлива

Підсумок: SSL/TLS VPN

Коли обирати SSL VPN
  • Потрібен обхід рестриктивних firewall (TCP 443)
  • Клієнтська база різнорідна (OS, пристрої)
  • Потрібен Clientless доступ для партнерів
  • Вже є інфраструктура PKI та сертифікати
Best Practices
  • TLS 1.3 або мінімум TLS 1.2
  • tls-crypt (не лише tls-auth)
  • MFA обов'язково
  • DCO для підвищення throughput
🔐

Підсумок лекції

  • Еволюція SSL/TLS: SSL 2.0 → TLS 1.3 (1-RTT, PFS)
  • TLS Record: 5-байтний заголовок (тип + версія + довжина)
  • TLS 1.2 Handshake: 4 фази, 2 RTT, деривація ключів через PRF
  • TLS 1.3: 1 RTT, ECDHE обов'язково, 5 cipher suites
  • OpenVPN: User-space, TUN (L3) / TAP (L2), OpenSSL
  • 5 фаз: Init → TLS → Key Neg → Data → Renegotiation
  • UDP > TCP: немає TCP over TCP meltdown
  • MFA: cert + password + TOTP = максимальний захист
