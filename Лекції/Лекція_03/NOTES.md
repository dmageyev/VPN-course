# Конспект — Лекція 03 — Мережева архітектура та механізми тунелювання

### 1. Underlay та Overlay мережі

Розуміння різниці між Underlay та Overlay є фундаментальним для проектування будь-якої VPN-інфраструктури. Underlay мережа — це фізична або реальна мережева інфраструктура: фізичні кабелі, маршрутизатори провайдерів, оптичні магістралі та всі фізичні вузли, що забезпечують передачу пакетів через Інтернет. Власник Underlay — це Internet Service Provider (ISP), і кінцевий користувач не має контролю над маршрутизацією в цій мережі.

Overlay мережа — це логічна (віртуальна) мережа, побудована поверх Underlay. Вона існує лише як програмна абстракція: вузли в Overlay мережі спілкуються один з одним за власними правилами, навіть якщо фізично їх пакети проходять через десятки маршрутизаторів ISP. VPN є класичним прикладом Overlay мережі. Інші приклади: MPLS (Multiprotocol Label Switching), VXLAN (Virtual eXtensible LAN), GRE-тунелі, SD-WAN.

Чому Overlay корисний? По-перше, він дозволяє організаціям побудувати власну логічну топологію мережі, незалежну від фізичного розташування вузлів та топології Underlay. По-друге, Overlay надає шифрування та конфіденційність — трафік у тунелі не читається маршрутизаторами ISP. По-третє, Overlay дозволяє використовувати приватні IP-адреси (RFC 1918) через публічний Інтернет.

Взаємодія Underlay та Overlay: коли комп'ютер A надсилає пакет комп'ютеру B через VPN-тунель, Overlay бачить це як пряме з'єднання A до B. Але реально пакет обгортається у зовнішній IP-заголовок і подорожує через Underlay від VPN-шлюзу A до VPN-шлюзу B, проходячи через багато проміжних маршрутизаторів. На кожному hop Underlay-маршрутизатори бачать лише зовнішній заголовок і не знають нічого про внутрішній пакет або його зміст.

### 2. Механізм інкапсуляції та накладні витрати

Інкапсуляція (encapsulation) — ключовий механізм тунелювання: оригінальний мережевий пакет (або кадр) повністю вкладається у новий пакет як корисне навантаження (payload). Зовнішній пакет містить свої заголовки і є носієм (carrier), а внутрішній — пасажиром (passenger). Принцип схожий на матрьошку: кожен рівень обгортання додає власний заголовок.

Детальний приклад IPsec Tunnel Mode:
1. Оригінальний пакет: IP src=10.0.1.5 dst=10.0.2.10, TCP src=54321 dst=80, HTTP дані
2. IPsec ESP шифрує внутрішній IP+TCP+HTTP, додає ESP заголовок (SPI, Sequence Number) та трейлер (Padding, Next Header, ICV)
3. Новий зовнішній пакет: IP src=203.0.113.1 dst=198.51.100.1, ESP Header, зашифровані дані, ESP Trailer+ICV
4. Зовнішній пакет передається через Інтернет; маршрутизатори бачать лише зовнішній IP-заголовок
5. На приймальній стороні: IPsec перевіряє ICV, дешифрує ESP payload, отримує оригінальний пакет

Накладні витрати (Overhead) — це додаткові байти, що з'являються внаслідок інкапсуляції. Типові значення: GRE додає 24 байти (4 байти GRE заголовок плюс 20 байтів IP заголовок), IPsec ESP з AES-256-GCM додає 50-70 байтів залежно від вирівнювання, L2TP/IPsec при подвійній інкапсуляції дає 80-100 байтів overhead, OpenVPN UDP додає 28-40 байтів, WireGuard має фіксований заголовок 32 байти. Overhead безпосередньо зменшує ефективну пропускну здатність тунелю і збільшує відносний розмір малих пакетів більше, ніж великих.

### 3. Віртуальні мережеві інтерфейси TUN та TAP

Для роботи VPN в операційній системі використовуються спеціальні віртуальні мережеві інтерфейси: TUN і TAP. Вони є програмними псевдопристроями, які виглядають як мережеві карти, але замість фізичного середовища передачі взаємодіють із простором користувача через файловий дескриптор.

TUN (Network TUNnel) — віртуальний мережевий інтерфейс рівня 3 (мережевий рівень OSI). Він оперує IP-пакетами. Коли застосунок пише IP-пакет у файловий дескриптор TUN, операційна система приймає цей пакет як вхідний мережевий трафік на цьому інтерфейсі. Аналогічно, пакети, призначені для маршруту через TUN, читаються з файлового дескриптора VPN-демоном для подальшого шифрування та відправки.

Використання TUN: OpenVPN у режимі «routed» (tun mode), WireGuard, більшість IPsec реалізацій для Remote Access VPN. Клієнт отримує IP-адресу з підмережі тунелю (наприклад, 10.8.0.2/24), і трафік маршрутизується через цей інтерфейс.

TAP (Network TAP) — віртуальний мережевий інтерфейс рівня 2 (канальний рівень OSI). Він оперує Ethernet-кадрами, включаючи MAC-адреси. TAP емулює фізичну мережеву карту на рівні Ethernet. Це дозволяє: передавати не-IP протоколи (IPX, NetBEUI) через тунель, створювати мости між двома сегментами LAN через VPN, організувати Layer-2 VPN, де клієнт отримує IP-адресу через DHCP сервера віддаленої мережі.

Порівняння: TUN працює на рівні L3, передає лише IP протоколи, broadcast не передається, має вищу продуктивність. TAP працює на рівні L2, передає Ethernet кадри будь-яких протоколів, broadcast передається (що збільшує трафік), підходить для bridge конфігурацій.

Пристрій TUN/TAP створюється через системний виклик open на /dev/net/tun з наступним ioctl() для іменування та конфігурації. VPN-демон утримує дескриптор відкритим. ОС бачить tun0 або tap0 як звичайний мережевий інтерфейс і дозволяє налаштовувати йому IP-адреси та маршрути за допомогою стандартних утиліт.

### 4. Проблема MTU та фрагментація пакетів

MTU (Maximum Transmission Unit) — максимальний розмір пакета, який може бути переданий через певне мережеве з'єднання без фрагментації. Стандартний MTU для Ethernet — 1500 байтів. Для PPPoE (часто використовується у DSL) — 1492 байти. Для деяких тунельних інтерфейсів — ще менше.

Проблема виникає через те, що інкапсуляція збільшує розмір пакета. Якщо внутрішній пакет має розмір 1500 байт (максимальний Ethernet MTU), то після додавання VPN-заголовків зовнішній пакет стане, наприклад, 1548 байтів для GRE/IPsec. Цей пакет не вміщається в стандартний MTU Ethernet і повинен бути або фрагментований, або відкинутий.

Фрагментація IP: IPv4 дозволяє фрагментацію на будь-якому маршрутизаторі по шляху. Якщо встановлено біт «Don't Fragment» (DF=1) у IP-заголовку, пакет відкидається і надсилається ICMP повідомлення «Fragmentation Needed». Фрагментація знижує продуктивність: кожен фрагмент містить власний IP-заголовок, отримувач повинен зібрати фрагменти, і втрата одного фрагмента призводить до перепередачі всього оригінального пакета.

Path MTU Discovery (PMTUD, RFC 1191): TCP автоматично намагається визначити мінімальний MTU по маршруту. Надсилає пакети з DF=1 та максимальним розміром; якщо отримує ICMP «Fragmentation Needed» — зменшує MSS. Проблема: деякі firewall блокують ICMP, що призводить до «PMTUD Black Holes» — пакети відправляються але ніколи не досягають призначення.

TCP MSS (Maximum Segment Size): MSS — це максимальний розмір TCP payload (без заголовків TCP та IP). Для Ethernet: MSS = MTU(1500) мінус IP header(20) мінус TCP header(20) = 1460 байтів. TCP MSS Clamping — механізм примусового зменшення MSS в TCP SYN пакетах при проходженні через VPN-шлюз. VPN-шлюз перехоплює TCP SYN та SYN-ACK і змінює значення MSS у TCP options на безпечне значення (наприклад, 1360 для тунелів з великим overhead).

Оптимальне налаштування MTU для різних VPN: WireGuard — рекомендований MTU 1420 (враховує WireGuard overhead 60 байтів плюс IP 20 плюс UDP 8), OpenVPN UDP — MTU близько 1432, IPsec ESP з AES-GCM — MTU приблизно 1400-1420 в залежності від конфігурації.

### 5. Split-Tunnel vs Full-Tunnel

Split-Tunneling — режим маршрутизації, при якому лише частина трафіку направляється через VPN-тунель, а решта іде напряму до Інтернету, оминаючи VPN. Як це працює: VPN-сервер при встановленні з'єднання надсилає клієнту лише конкретні маршрути для корпоративних підмереж (наприклад, 192.168.0.0/16 та 10.0.0.0/8 через VPN; все решта напряму). Клієнт додає ці маршрути у свою таблицю маршрутизації без зміни маршруту за замовчуванням.

Переваги Split-Tunnel: вища продуктивність, корпоративний трафік захищений але YouTube або Netflix не навантажує VPN-сервер, менша затримка для інтернет-ресурсів, знижене навантаження на VPN-сервер та канал, краще сприймається кінцевими користувачами.

Недоліки Split-Tunnel: інтернет-трафік не захищений VPN (актуально для публічних Wi-Fi), DNS-витоки, DNS-запити для не-VPN трафіку йдуть через ISP DNS, розкриваючи активність, складніше центральне управління та аудит трафіку, неможливо застосувати корпоративні политики безпеки до всього трафіку.

Full-Tunnel — режим, при якому весь трафік клієнта направляється через VPN. Клієнт отримує маршрут за замовчуванням 0.0.0.0/0 через VPN-інтерфейс.

Переваги Full-Tunnel: повний захист всього трафіку (критично у недовірених мережах), централізована фільтрація та аудит, захист від DNS-витоків, відповідність вимогам compliance. Недоліки: VPN-сервер стає пляшковим горлом для всього інтернет-трафіку, вища latency для некорпоративних ресурсів, більше навантаження на сервер та канал.

Inverse Split-Tunnel: окремий режим, де через VPN іде весь трафік крім конкретного переліку. Наприклад, весь трафік через VPN, але трафік до Office 365 та Google Workspace — напряму.

### 6. Маршрутизація у VPN та протоколи

Для того щоб пакети потрапляли у VPN-тунель, у таблиці маршрутизації клієнта та сервера повинні бути відповідні записи. Конфігурація маршрутизації залежить від топології та режиму тунелювання.

Статична маршрутизація: адміністратор вручну налаштовує маршрути на обох кінцях тунелю. Просто для малих мереж, але не масштабується. Приклад команд: ip route add 10.0.2.0/24 via tun0, ip route add 192.168.100.0/24 via tun0.

Динамічна маршрутизація через VPN: запуск протоколів маршрутизації (OSPF, BGP) всередині VPN-тунелю. Маршрутизатори автоматично обмінюються таблицями маршрутизації. Особливість: протоколи L2TP та GRE дозволяють передавати multicast пакети OSPF та EIGRP, тоді як чистий IPsec потребує GRE-обгортки для multicast, оскільки IPsec не підтримує multicast у Tunnel Mode.

Проблема рекурсивного маршрутизування: якщо маршрут до VPN-ендпойнту проходить через сам VPN-інтерфейс, виникає петля маршрутизації. Рішення: явно вказувати маршрут до VPN-сервера через фізичний інтерфейс (default gateway) з вищим пріоритетом (нижчою метрикою), щоб статичний маршрут до VPN-сервера перекривав маршрут за замовчуванням.

DMVPN (Dynamic Multipoint VPN): технологія Cisco, що поєднує mGRE (multipoint GRE), NHRP (Next Hop Resolution Protocol) та IPsec. Дозволяє автоматично встановлювати spoke-to-spoke VPN-тунелі без участі hub-маршрутизатора. NHRP виконує роль адресної книги: hub зберігає зовнішні IP-адреси всіх spoke, і spoke може запитати адресу іншого spoke для прямого тунелю.

### 7. DNS-витоки та їх запобігання

DNS-витік (DNS leak) — ситуація, коли DNS-запити користувача, попри активний VPN, відправляються через ISP DNS-сервери, а не через DNS-сервер VPN. Це розкриває список відвіданих доменів провайдеру або зловмиснику в публічній мережі.

Причини DNS-витоків: VPN-клієнт налаштований лише для маршрутизації IP-трафіку, але не підміняє DNS-налаштування системи. Windows Async DNS — ОС паралельно надсилає DNS-запити через всі мережеві інтерфейси і використовує першу відповідь. IPv6 DNS: VPN захищає IPv4, але IPv6 DNS іде поза тунелем. WebRTC leak: браузери можуть розкривати локальний IP через WebRTC навіть при активному VPN.

Методи запобігання: VPN-клієнт повинен встановлювати DNS-сервер провайдера VPN як системний DNS при підключенні. Kill Switch: блокувати весь трафік якщо VPN-з'єднання перервано. Використання DNS over HTTPS або DNS over TLS через VPN. Вимкнення IPv6 або повне тунелювання IPv6 трафіку. Використання firewall правил для блокування DNS (UDP/TCP порт 53) через не-VPN інтерфейси.

### 8. Топології VPN Overlay

Point-to-Point: найпростіша топологія — тунель між двома вузлами. Підходить для Site-to-Site між двома офісами. N вузлів потребують N помножити на (N-1) і поділити на 2 тунелів для повного mesh.

Hub-and-Spoke: один центральний вузол (Hub) з'єднаний з усіма периферійними (Spoke). Весь трафік між Spoke проходить через Hub. Переваги: просто управляється, Hub може застосовувати централізовані politiky. Недоліки: Hub є точкою відмови (Single Point of Failure), трафік між Spoke має подвійну затримку.

Full Mesh: кожен вузол з'єднаний з кожним напряму. Оптимальна продуктивність та відмовостійкість. Складно масштабується: 100 вузлів = 4950 тунелів. Підходить для невеликих мереж (до 10-15 вузлів).

Partial Mesh: компроміс між Hub-and-Spoke та Full Mesh. Прямі тунелі лише між вузлами, що активно обмінюються даними. Решта через Hub. DMVPN дозволяє автоматично створювати spoke-to-spoke тунелі на вимогу і відключати їх при відсутності трафіку.

### 9. Безпекові аспекти мережевої архітектури VPN

Принцип мінімальних привілеїв у маршрутизації: VPN-клієнт повинен мати доступ лише до необхідних ресурсів, а не до всієї корпоративної мережі. Split-tunnel з явними дозволами краще, ніж Full-tunnel з необмеженим доступом.

VPN та IDS/IPS: зашифрований трафік не може бути проінспектований системами виявлення вторгнень (IDS). Корпоративне рішення: SSL Inspection (MITM) на VPN-шлюзі або корпоративних проксі-серверах. Це дозволяє аналізувати трафік, але потребує установки корпоративного кореневого CA на пристрої.

Ізоляція мережевих сегментів: VPN-клієнти не повинні автоматично отримувати доступ до всіх VLAN корпоративної мережі. Після підключення через VPN слід застосовувати додаткові ACL (Access Control Lists) або VLAN-сегментацію для обмеження бокового переміщення.

### 10. Практичні аспекти налаштування

При розгортанні VPN-рішення важливо врахувати кілька практичних аспектів.

Keepalive механізми: VPN-тунель може завмирати при відсутності трафіку, якщо stateful firewall або NAT відкидає state. Keepalive пакети (DPD — Dead Peer Detection у IKEv2, ping через тунель) підтримують тунель активним і виявляють розриви.

Redundancy та failover: критичні VPN-з'єднання повинні мати резервні канали. IKEv2 з MOBIKE підтримує failover між IP-адресами. OpenVPN може автоматично перемикатись між адресами серверів у конфігурації.

Логування та моніторинг: всі VPN-з'єднання повинні логуватись: час підключення та відключення, IP-адреса клієнта, тривалість сесії, переданий обсяг. Логи відправляються у SIEM-систему для кореляції та виявлення аномалій.

Автоматичне підключення (Always-On VPN): режим, при якому VPN-клієнт автоматично підключається при запуску системи та блокує весь мережевий доступ до встановлення VPN. Запобігає незахищеним сесіям. Конфігурується через MDM (Mobile Device Management) для корпоративних пристроїв.

### 11. Продуктивність та оптимізація VPN

Продуктивність VPN залежить від кількох ключових факторів. Перший — шифрування. Апаратне прискорення AES (інструкція AES-NI на процесорах Intel/AMD) суттєво підвищує throughput: без AES-NI типовий процесор обробляє 200-500 Мбіт/с IPsec, з AES-NI — 2-10 Гбіт/с. ChaCha20 показує кращу продуктивність на пристроях без AES-NI (більшість ARM-процесорів мобільних пристроїв та IoT).

Другий фактор — розмір черги та затримка обробки. У OpenVPN (user-space) кожен пакет проходить через мережевий стек двічі: з ядра у user-space для шифрування, та назад у ядро для відправки. WireGuard (kernel-space) уникає цих переключень контексту, що дає нижчу latency і вищий throughput.

Третій фактор — паралелізм. Multi-threaded або multi-process VPN може масштабуватись на багатоядерних процесорах. WireGuard нативно підтримує multi-core обробку. OpenVPN 2.5+ підтримує DCO (Data Channel Offload) — kernel-space обробку data channel.

Четвертий фактор — мережева топологія та RTT. Висока затримка до VPN-сервера збільшує час встановлення тунелю та час відповіді на запити. Обирайте VPN-сервер у найближчому географічному регіоні до більшості клієнтів.

### 12. Налагодження мережевих проблем VPN

При виникненні проблем із VPN-тунелем існує стандартний алгоритм діагностики.

Крок 1: перевірити базову IP-зв'язність. Чи пінгується зовнішній IP VPN-сервера? Чи доступний UDP/TCP порт сервера (nc -u server_ip port або telnet для TCP)?

Крок 2: перевірити встановлення тунелю. Чи показує статус VPN-клієнта підключено? Чи з'явився новий мережевий інтерфейс (tun0, wg0)? Чи отримав клієнт внутрішню IP-адресу від сервера?

Крок 3: перевірити маршрутизацію. Команда ip route show показує поточні маршрути. Чи є маршрут через VPN-інтерфейс? Чи немає конфлікту з маршрутом до VPN-сервера?

Крок 4: перевірити MTU. Відправити великий пакет: ping -M do -s 1400 destination_ip. Якщо маленькі пакети проходять, а великі ні — проблема MTU.

Крок 5: перевірити DNS. Чи резолвляться доменні імена? Команда nslookup або dig вкажуть, який DNS-сервер використовується.

Крок 6: перевірити firewall правила. Чи дозволяє локальний firewall пакети через VPN-інтерфейс? Чи не блокує ISP UDP-трафік на нестандартних портах?

Wireshark та tcpdump дозволяють захоплювати пакети на інтерфейсах для глибокої діагностики: tcpdump -i tun0 -n показує розшифрований трафік всередині тунелю, tcpdump -i eth0 -n port 500 or port 4500 показує IKE-обмін.
