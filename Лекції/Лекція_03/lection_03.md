## Слайд 3

## Underlay vs Overlay мережі

### 🌐 Underlay (Фізична мережа)

Реальна фізична або логічна інфраструктура Інтернету. Маршрутизатори, провайдери (ISP), фізичні канали, протоколи BGP/OSPF. Не контролюється VPN-адміністратором.

### 🔒 Overlay (Віртуальна мережа)

Логічна мережа, що «накладається» поверх Underlay. VPN-тунель — класичний приклад Overlay. Керується адміністратором VPN незалежно від фізичної топології.

      Хост A
10.0.0.1
      ⟶
      Underlay
Інтернет
      ⟶
      Хост B
10.0.0.2

      VPN Overlay тунель між Хост A ↔ Хост B (зашифрований)

## Слайд 4

## Навіщо потрібна Overlay архітектура?

### Переваги Overlay

- Незалежність від фізичної топології мережі

- Єдиний адресний простір для розподілених команд

- Шифрування поверх будь-якого Underlay

- Гнучка зміна маршрутів без зміни фізики

- Підтримка multi-vendor середовищ

### Недоліки / Виклики

- Додаткові накладні витрати (overhead)

- Збільшення затримки (latency)

- Складніше відлагодження мережевих проблем

- MTU-проблеми через подвійну інкапсуляцію

## Слайд 5

## Механізм інкапсуляції

Інкапсуляція — це обгортання оригінального пакета в новий пакет із додаванням нових заголовків. Оригінальний пакет стає «корисним навантаженням» (payload) для зовнішнього пакета.

        Outer
IP Hdr
        VPN
Hdr
        Inner
IP Hdr
        TCP/UDP
Hdr
        Payload
(дані)

      ← зовнішній заголовок ─────────────── оригінальний пакет ────────────────────→

### Приклад: IPsec Tunnel Mode

Весь IP-пакет (заголовок + дані) шифрується та обгортається новим IP-заголовком. Зовнішній заголовок містить адреси VPN-шлюзів.

### Приклад: WireGuard / OpenVPN

Оригінальний IP-пакет зашифровується та вкладається в UDP-датаграму. Overhead: ~40–60 байт на пакет залежно від протоколу.

---

## Слайд 6

## Накладні витрати (Overhead)

Кожний доданий заголовок «їсть» корисну пропускну здатність. Реальна ефективна швидкість завжди менша за фізичну.

        | Протокол | Додаткові байти (типово) | Ефективна MTU при Ethernet 1500 б |

        | WireGuard (IPv4/UDP) | ~60 байт | ~1440 байт |
        | OpenVPN (UDP режим) | ~40–55 байт | ~1445–1460 байт |
        | IPsec (ESP Tunnel) | ~50–73 байт | ~1427–1450 байт |
        | L2TP/IPsec | ~70–80 байт | ~1420–1430 байт |
        | GRE + IPsec | ~80–94 байт | ~1406–1420 байт |

* Точні значення залежать від версії IP (v4/v6), режиму шифрування та опцій аутентифікації.

## Слайд 7

## Віртуальні мережеві інтерфейси

Для роботи VPN-клієнту необхідний «вхідний/вихідний» інтерфейс в ОС. Замість фізичної карти використовуються віртуальні адаптери, реалізовані через драйвери ядра.

### TUN (Network Tunnel)

- Працює на Рівні 3 (IP)

- Передає IP-пакети напряму

- Найефективніший для маршрутизації

- Назва інтерфейсу: tun0, wg0

- Використовується: WireGuard, OpenVPN (routed)

### TAP (Ethernet Tunnel)

- Емулює Ethernet-адаптер (Рівень 2)

- Передає Ethernet-фрейми (з MAC-адресами)

- Підтримує non-IP трафік (IPX, AppleTalk)

- Назва інтерфейсу: tap0

- Використовується: OpenVPN (bridged), QEMU

## Слайд 8

## Як працює TUN/TAP в ОС

          Застосунок (VPN-клієнт)
          ↕
          Файл /dev/tun або /dev/tap
          ↕
          Мережевий стек ядра Linux
          ↕
          Фізичний інтерфейс (eth0)
          ↕
          Мережа (Інтернет)

### Принцип роботи

VPN-демон читає пакети з /dev/tun0 → шифрує → відправляє через реальний eth0 в Інтернет. Зворотньо: приймає з eth0 → розшифровує → вкидає у /dev/tun0 для стека ядра.

### Ключова відмінність

TUN: повертає IP-пакети. TAP: повертає Ethernet-фрейми (містять MAC-адресу). TAP займає більше ресурсів але підтримує бриджинг та широкомовлення (broadcast).

## Слайд 9

## Проблема MTU (Maximum Transmission Unit)

MTU — максимальний розмір кадру/пакета, який може передатися без фрагментації. Стандартне значення для Ethernet: 1500 байт.

### Проблема в тунелях

Після інкапсуляції пакет збільшується на розмір заголовків. Якщо новий розмір перевищує MTU каналу — пакет або фрагментується (повільніше) або відкидається (якщо встановлено DF bit).

          Оригінальний пакет: 1500 байт
          ↓ +60 байт VPN overhead
          Після інкапсуляції: 1560 байт ❌
          ↓ Перевищує MTU 1500!
          Фрагментація або Drop

## Слайд 10

## Вирішення проблем MTU

### TCP MSS Clamping

Примусово зменшує значення поля MSS (Maximum Segment Size) у SYN-пакетах TCP. MSS = MTU_тунелю – 40 байт (IP + TCP заголовки).

iptables -t mangle -A FORWARD \
  -p tcp --tcp-flags SYN,RST SYN \
  -j TCPMSS --clamp-mss-to-pmtu

### PMTUD (Path MTU Discovery)

          Механізм автоматичного виявлення мінімального MTU на шляху. Хост надсилає пакети з DF=1 (Don't Fragment) і зменшує розмір при отриманні ICMP «Fragmentation Needed».

### Ручне налаштування MTU

Встановлення MTU інтерфейсу тунелю на 1420–1450 байт замість 1500 вирішує більшість проблем без MSS Clamping.

          ip link set tun0 mtu 1420

## Слайд 11

## MSS vs MTU — ключова різниця

        | Параметр | Де діє | Типове значення | Хто задає |

        | MTU (Max Transmission Unit) | Канальний рівень (L2) | 1500 байт (Ethernet) | Адміністратор / обладнання |
        | MSS (Max Segment Size) | Транспортний рівень (TCP) | 1460 байт (MTU-40) | Узгоджується в TCP SYN/SYN-ACK |
        | MTU тунелю (VPN) | Мережевий рівень (L3) | 1420–1450 байт | VPN-адміністратор |
        | MSS у тунелі | Транспортний рівень (TCP) | 1360–1410 байт | MSS Clamping / PMTUD |

Правило: MSS = MTU_інтерфейсу – 20 (IP) – 20 (TCP) = MTU – 40

## Слайд 12

## Фрагментація пакетів

Якщо пакет перевищує MTU і не встановлено DF bit, він фрагментується на кілька менших частин.

### Наслідки фрагментації

- Збільшення навантаження на маршрутизатори

- Додаткова затримка при збірці фрагментів

- Проблеми з брандмауерами (відкидають фрагменти)

- Зниження продуктивності на 10–30%

### Сценарій: DF bit = 1

Якщо хост встановив DF bit (Don't Fragment) і пакет перевищує MTU — маршрутизатор відкидає пакет і відправляє ICMP Type 3 Code 4 («Fragmentation Needed»).

Проблема: якщо ICMP заблоковано → «Black Hole» — з'єднання «зависає», бо хост не дізнається, що треба зменшити MTU.

## Слайд 13

## Маршрутизація у VPN

Коли VPN-тунель встановлено, виникає питання: який трафік направляти через тунель? Є дві основні стратегії.

### 🔀 Split-Tunnel

Через VPN-тунель направляється лише визначений трафік (корпоративні підмережі). Решта трафіку (YouTube, соцмережі) іде прямо через ISP.

### 🛡️ Full-Tunnel

Весь трафік клієнта направляється через VPN-сервер. Маршрут за замовчуванням (0.0.0.0/0) переписується на VPN-інтерфейс.

---

## Split-Tunnel: деталі

          Хост (клієнт)

              ↙ VPN
              10.0.0.0/8
192.168.0.0/16
              VPN-сервер
(корп. ресурси)

              ↘ Напряму
              0.0.0.0/0
(решта)
              ISP → Інтернет

### Переваги

- Менше навантаження на VPN-сервер

- Вища швидкість для інтернет-трафіку

- Менша затримка для непрофільних сервісів

### Ризики

- Інтернет-трафік не захищений VPN

- DNS-витоки (DNS leaks)

- Складніше аудитувати дії користувача

## Слайд 15

## Full-Tunnel: деталі

          Хост (клієнт)
          ↓ Весь трафік
          tun0 (VPN інтерфейс)
маршрут: 0.0.0.0/0
          ↓ Зашифрований
          VPN-сервер
          ↓ Розшифрований
          Інтернет

### Переваги

- Весь трафік захищений та зашифрований

- Приховування реального IP від сайтів

- Захист від перехоплення у публічних Wi-Fi

- Централізована фільтрація та аудит

### Недоліки

- Більше навантаження на VPN-сервер

- Вища затримка для всього трафіку

- Зниження швидкості інтернету

## Слайд 16

## Full-Tunnel vs Split-Tunnel: Порівняння

        | Критерій | Full-Tunnel | Split-Tunnel |

        | Захист трафіку | Весь трафік | Тільки вибраний |
        | Навантаження на сервер | Високе | Низьке |
        | Швидкість інтернету | Нижча | Вища |
        | Конфіденційність | Максимальна | Часткова |
        | DNS-витоки | Мінімальні | Можливі |
        | Сценарій застосування | Публічні Wi-Fi, журналісти | Корпоративний доступ |

## Слайд 18

## DNS-витоки (DNS Leaks)

У Split-Tunnel режимі DNS-запити можуть іти поза тунелем, розкриваючи ISP інформацію про відвідувані сайти, навіть якщо сам трафік зашифрований.

### Сценарій витоку

- Хост підключається до VPN (Split-Tunnel)

- DNS налаштований на ISP-сервер (8.8.8.8)

- DNS-запити йдуть напряму через ISP

- ISP бачить усі запити ↔ витік!

### Виправлення

- Налаштувати DNS на VPN-сервер (наприклад, 10.0.0.1)

- Використовувати DNS = ... у конфігурації клієнта

- Перевірити витоки: dnsleaktest.com

- У повному тунелі проблема вирішується автоматично

## Слайд 20

## Топології VPN Overlay

### Hub-and-Spoke (зірка)

Усі клієнти з'єднані з центральним VPN-сервером. Трафік між клієнтами проходить через хаб. Проста конфігурація, але хаб — єдина точка відмови.

### Full-Mesh (повна сітка)

Прямі тунелі між кожною парою вузлів. Мінімальна затримка, немає єдиної точки відмови. Складніша конфігурація: O(n²) з'єднань.

### Site-to-Site VPN

Два офіси (мережі) з'єднані постійним тунелем між VPN-шлюзами. Усі пристрої всередині офісу автоматично отримують доступ через тунель без налаштувань на хостах.

### Remote Access VPN

Окремий користувач підключається до корпоративної мережі з будь-якого місця. Потребує VPN-клієнта на пристрої. Типово: OpenVPN, WireGuard, AnyConnect.

## Слайд 22

## Безпекові аспекти архітектури

### Вразливості

- IPv6 Leak: VPN не охоплює IPv6 трафік

- WebRTC Leak: браузер розкриває реальний IP

- Timing attacks через аналіз трафіку

- Split-horizon DNS атаки

### Захисні заходи

- Блокувати IPv6 або тунелювати разом з IPv4

- Вимикати WebRTC у браузерах

- Перевіряти DNS-витоки після підключення

- Використовувати Kill Switch при обриві VPN

### Kill Switch

Механізм блокування всього трафіку при обриві VPN-з'єднання. Реалізується через iptables правила, що дозволяють трафік лише через VPN-інтерфейс.

## Слайд 26 — DMVPN та динамічні топології

**Слайд:** DMVPN = mGRE + NHRP + IPsec. Три фази DMVPN. Порівняльна таблиця топологій: Hub-and-Spoke, Full Mesh, DMVPN Phase 2.

**Контекст:** DMVPN дозволяє масштабувати VPN-мережі без потреби у конфігурації кожного тунелю окремо.

---

## Слайд 27 — NAT та проблеми VPN

**Слайд:** AH несумісний з NAT. ESP + NAT-T (UDP 4500). IKE автовиявлення NAT через NAT_DETECTION payloads.

**Контекст:** NAT traversal є обов'язковою вимогою для сучасних VPN — більшість клієнтів знаходяться за NAT.

---

## Слайд 28 — DNS-витоки та Kill Switch

**Слайд:** DNS-витік: запити поза тунелем розкривають відвідані сайти. Причини, наслідки та методи захисту.

**Контекст:** DNS-витоки є частою прихованою вразливістю навіть при активному VPN-з'єднанні.
